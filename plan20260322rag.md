# 2026-03-22 NLP2SQL RAG Design Draft

## Goal

Build a Vanna-like NLP2SQL assistant in DeerFlow that can learn from example SQL, business documentation, terminology, uploaded files, and historical SQL, then use RAG to improve SQL generation and business-semantic understanding.

The storage target is PostgreSQL, with semantic retrieval implemented through `pgvector`, so the system avoids introducing an extra vector middleware.

## Confirmed Decisions

### 1. Product Entry

- Use the existing standalone `nlp2sql agent` chat experience as the main entry.
- The user enters through the frontend agent page, which behaves as an independent SQL-focused chat page.
- The same backend retrieval and generation pipeline should support both chat usage and training-management operations.

### 2. Frontend and Backend Scope

- Phase 1 includes both frontend and backend.
- Backend must provide:
  - training data ingestion
  - chunking and embedding
  - vector retrieval
  - prompt context assembly for SQL generation
- Frontend must provide:
  - training data management UI
  - file upload and import UI
  - retrieval visibility inside the SQL assistant experience

### 3. Knowledge Sources in Scope

Phase 1 knowledge sources include:

- example `question -> SQL`
- business documentation / metric definitions / caliber notes
- database schema / field comments
- terminology glossary
- common filter value examples
- join relationship descriptions
- uploaded files: Markdown, PDF, Word, TXT
- historical SQL import

This is the agreed "extended" scope rather than a minimal MVP.

### 4. Isolation Boundary

- Knowledge is strictly isolated by `data_source_id`.
- Each data source maintains its own examples, docs, terminology, imported files, and historical SQL.
- Retrieval for one data source must not use training data from another data source.

### 5. Embedding / Indexing Mode

Use a hybrid indexing model:

- small manual entries: synchronous indexing
- large files and historical SQL imports: asynchronous indexing

This keeps the common path responsive while still supporting large knowledge ingestion safely.

### 6. PostgreSQL Deployment Model

The system should support either:

- a dedicated PostgreSQL database for `nlp2sql knowledge`, or
- a shared PostgreSQL database with isolated schema

Recommended default:

- design for a dedicated knowledge database first
- keep the code DSN-driven so it can later run inside a shared PostgreSQL deployment without rework

## Options Considered

### Option A. Conservative Add-On

Keep the current DeerFlow `nlp2sql` data source registry, schema cache, validation, and execution flow. Add a new knowledge subsystem for training data storage, chunking, embeddings, pgvector retrieval, and prompt assembly.

Pros:

- smallest delivery risk
- easiest to integrate into current codebase
- minimal disruption to existing SQL execution path

Cons:

- storage is not fully unified yet because existing data source config remains outside PostgreSQL

### Option B. Fully Unified NLP2SQL Storage

Move training data, chunks, indexing tasks, schema cache, and even data source config into PostgreSQL under a dedicated schema.

Pros:

- strongest long-term consistency
- simplest operational model after migration
- better foundation for auditing, permissions, stats, and versioning

Cons:

- larger first-phase refactor
- higher migration complexity

### Option C. Agent-First Rebuild

Rebuild the `nlp2sql agent` as a more opinionated SQL copilot product with redesigned backend boundaries.

Pros:

- strongest product coherence

Cons:

- too broad for phase 1
- likely to turn into a refactor instead of an incremental delivery

## Recommended Direction

Adopt a middle path:

- implement the first phase operationally like Option A
- design the storage model with Option B in mind

Concretely:

- put `training data`, `chunks`, `embeddings`, `retrieval`, and `index task state` in PostgreSQL + `pgvector`
- keep the current data source registry for now
- optionally migrate schema cache later or in a compatible second step
- enhance the existing frontend entry points instead of creating a new product surface

This gives fast delivery now without blocking a later full PostgreSQL convergence.

## Approved High-Level Architecture

### 1. Knowledge Store

A dedicated backend subsystem that stores:

- raw training items
- processed chunks
- vector embeddings
- indexing task state

Storage is PostgreSQL with `pgvector`, and all records are filtered by `data_source_id`.

### 2. Ingestion Pipeline

Handles:

- manual example SQL creation
- glossary / terminology entries
- documentation ingestion
- file upload ingestion
- historical SQL import

Responsibilities:

- normalize source data
- split content into retrieval chunks
- generate embeddings
- persist source and chunk records
- run sync or async indexing depending on payload size

### 3. Retrieval Orchestrator

For each user question in the `nlp2sql agent`:

- identify the active data source
- retrieve relevant schema context
- retrieve semantically similar examples, docs, terminology, and SQL history
- rerank and trim by prompt budget
- build the final SQL-generation context

### 4. SQL Generation Runtime

Continue to reuse the current DeerFlow `nlp2sql` execution path:

- schema service
- SQL validation
- SQL execution
- adapter factory

RAG is inserted before SQL generation, not as a replacement for the existing safety and execution flow.

### 5. Frontend SQL Copilot Experience

Extend the existing `nlp2sql agent` flow with two visible capability areas:

- chat-side retrieval visibility
- management-side training and indexing controls

The intent is not to create a second separate SQL tool, but to upgrade the current SQL agent into a trainable SQL copilot.

## Current Non-Goals

The following are intentionally not fixed yet in this draft:

- detailed PostgreSQL table schema
- chunking rules by source type
- embedding model selection
- reranking algorithm
- exact frontend page layout
- exact API contract for ingestion and retrieval inspection
- migration path for current local schema cache and data source registry

These should be defined in the next design sections.

## Embedding Model Rotation Strategy

Changing the embedding model must not require destructive rewrites of historical knowledge records. The system should treat embeddings as a rebuildable projection, not as the source of truth.

### Source of Truth

The source of truth remains:

- `knowledge_items.content`
- `knowledge_chunks.chunk_text`
- related metadata and chunk boundaries

Embeddings are derived artifacts and can be regenerated at any time.

### Required Design Rule

Do not store a single embedding directly as the only vector state for a chunk.

Instead, separate:

- the chunk record
- the embedding record generated from a specific embedding model / profile

### Additional Tables

To support model changes safely, add two more tables.

#### `embedding_profiles`

Represents one embedding configuration.

Suggested fields:

- `id`
- `name`
- `provider`
- `model`
- `dimensions`
- `distance_metric`
- `is_active`
- `config jsonb`
- `created_at`
- `archived_at`

Examples:

- `openai-text-embedding-3-large-3072`
- `bge-m3-1024`
- `gte-qwen2-1536`

#### `chunk_embeddings`

Stores one vector per `chunk_id + embedding_profile_id`.

Suggested fields:

- `id`
- `chunk_id`
- `data_source_id`
- `embedding_profile_id`
- `embedding vector(n)`
- `embedding_hash`
- `created_at`

Unique constraint:

- `(chunk_id, embedding_profile_id)`

This allows the same historical chunk to have vectors from multiple model versions during migration.

### Retrieval Rule

At query time:

- resolve the active embedding profile
- generate the query embedding with that profile
- retrieve only from `chunk_embeddings` for that profile

This avoids mixing incompatible vector spaces.

### Re-Embedding Workflow

When the embedding model changes:

1. Create a new `embedding_profile`
2. Mark it as `inactive` initially
3. Launch a background re-embedding job for all chunks of one data source or all data sources
4. Write new vectors into `chunk_embeddings`
5. Track progress in `index_jobs`
6. After completion and optional validation, switch the new profile to `active`
7. Keep the old profile for rollback until explicitly cleaned up

This makes model migration an additive process instead of an in-place rewrite.

### Operational Modes

Support both:

- full rebuild for all chunks under one profile
- incremental rebuild for one `data_source_id`

This is important because some users may want to test a new model on one data source before global rollout.

### Frontend / Admin Requirements

The management UI should expose:

- current active embedding profile
- available profiles
- create / register new profile
- trigger re-embedding for one data source
- trigger full re-embedding
- watch job progress
- switch active profile after rebuild

### Failure Handling

If re-embedding fails:

- old profile remains active
- partially built new vectors stay isolated under the new profile
- the job can be resumed or restarted safely

No historical source data should be lost, and no retrieval traffic should be moved to the new profile before activation.

### Recommendation

For phase 1, implement model rotation from the start.

This is not optional plumbing. If omitted now, the first embedding model change will force either:

- a long blocking rewrite of the vector table, or
- schema-breaking retrofits later

Treat embedding versioning as a first-class requirement in the initial storage design.

## Retrieval and Reranking Strategy

For each user question:

1. resolve the bound `data_source_id`
2. construct a normalized retrieval query
3. retrieve schema context
4. retrieve keyword matches from `search_tsv`
5. retrieve semantic matches from `chunk_embeddings` under the active embedding profile
6. rerank by source type and relevance
7. trim by prompt budget
8. assemble final SQL-generation context

### Retrieval Buckets

Do not mix all sources into one undifferentiated ranking pool. Retrieve and score by bucket:

- example SQL
- glossary / metric definitions
- join hints
- filter value hints
- documentation
- historical SQL
- schema facts

### Preferred Priority

Recommended priority order:

1. example SQL
2. glossary / metric definitions
3. join hints and filter hints
4. schema facts
5. documentation
6. historical SQL

### Prompt Budgeting

Suggested first-pass limits:

- example SQL: `2-4`
- glossary / metric definitions: `3-6`
- join hints / filter hints: `2-4`
- documentation: `2-3`
- historical SQL: `1-3`
- schema facts: selected summaries only

## Re-Embedding Workflow

When the embedding model changes:

1. create a new `embedding_profile`
2. keep it `inactive`
3. create a background `index_job`
4. rebuild vectors into `chunk_embeddings` for that profile
5. validate completion and basic retrieval quality
6. switch the new profile to `active`
7. keep the old profile for rollback until manual cleanup

Properties:

- no destructive overwrite of historical vectors
- supports one-data-source gray rollout
- old profile remains active until the new one is ready

## Failure Handling

The system must fail in a controlled way. Knowledge retrieval should improve SQL generation quality, but failure in the knowledge layer must not silently corrupt execution behavior.

### Ingestion Failures

If file parsing or historical SQL import fails:

- mark the corresponding `knowledge_file` or `index_job` as failed
- preserve the original uploaded file reference if available
- surface the failure in the management UI
- allow retry without requiring users to recreate the whole source manually

If chunking or embedding generation fails for one item:

- keep the `knowledge_item`
- do not mark it as searchable
- record the exact failure on the related indexing job

### Retrieval Failures

If the retrieval layer is unavailable for one request:

- do not block the whole `nlp2sql agent`
- fall back to:
  - active data source binding
  - existing schema search
  - existing SQL generation / validation path
- clearly mark in logs and traces that retrieval fallback occurred

The agent may degrade in quality, but it must not fail open in a way that bypasses SQL safety checks.

### Embedding Rotation Failures

If re-embedding fails partway:

- the old active profile remains active
- the new profile remains inactive
- partially built vectors remain isolated under the new profile
- the rebuild job can be resumed or restarted

Do not switch active embedding profiles until completion and validation pass.

### Data Isolation Failures

Every repository and query path must explicitly include `data_source_id`.

This should be treated as a hard rule, not a convention:

- no cross-data-source fallback
- no global retrieval pool in phase 1
- no query path that infers scope indirectly if it can be passed explicitly

## Observability

Add lightweight but explicit observability around ingestion, retrieval, and re-embedding.

### Structured Logging

At minimum, log:

- `data_source_id`
- `item_id`
- `job_id`
- `embedding_profile_id`
- operation type
- duration
- result status
- error summary

### Retrieval Diagnostics

For each retrieval request, capture:

- active embedding profile
- retrieval buckets queried
- candidate counts by bucket
- final selected context counts by bucket
- retrieval latency

This is necessary because most quality issues in NLP2SQL RAG come from bad recall, not from SQL execution.

### Indexing Diagnostics

Track:

- queue time
- parse duration
- chunk count
- embedding duration
- success / failure counts

### Admin Visibility

The UI should expose enough operational state to debug common problems without reading raw logs:

- job status
- parse failures
- active embedding profile
- last schema refresh time
- last successful indexing time per data source

## Testing Strategy

Testing should be split by risk area.

### 1. Storage / Repository Tests

Cover:

- create / update / delete knowledge items
- chunk persistence
- embedding profile activation rules
- chunk embedding uniqueness
- `data_source_id` isolation

### 2. Ingestion Pipeline Tests

Cover:

- manual example SQL indexing
- documentation chunking
- file ingestion state transitions
- historical SQL import
- sync vs async indexing decision rules

### 3. Retrieval Tests

Cover:

- bucketed retrieval behavior
- active profile filtering
- hybrid keyword + semantic retrieval
- prompt budget trimming
- no cross-data-source recall

### 4. Runtime Integration Tests

Cover:

- retrieval context injection into the SQL assistant flow
- fallback when retrieval storage is unavailable
- existing validation and execution still work unchanged

### 5. Frontend Tests

Cover:

- settings page CRUD flows
- file upload and job polling
- embedding profile management interactions
- retrieval preview rendering
- chat-side retrieval evidence display

### 6. Regression Focus

Regression risk is highest in:

- active data source binding
- SQL validation path
- retrieval leakage across data sources
- embedding profile switching

These should receive explicit targeted tests, not only broad happy-path coverage.

## Implementation Order

Implement in thin vertical slices so the system becomes usable early and risk stays localized.

### Phase 1. Storage Foundation

- add PostgreSQL knowledge config
- add schema bootstrap SQL
- implement repositories for:
  - knowledge items
  - chunks
  - embedding profiles
  - chunk embeddings
  - index jobs

### Phase 2. Manual Knowledge CRUD

- backend APIs for manual `knowledge_items`
- frontend `Knowledge` tab
- synchronous indexing for small text inputs

This gives a usable first path before file parsing or batch jobs.

### Phase 3. Retrieval Service

- implement active profile lookup
- implement keyword + semantic retrieval
- implement bucketing and prompt trimming
- add retrieval preview API

### Phase 4. Agent Integration

- inject retrieval context into the `nlp2sql agent`
- add retrieval evidence display in chat
- preserve current SQL validation and execution flow

### Phase 5. File and Historical SQL Ingestion

- file upload APIs and parsing pipeline
- historical SQL import APIs
- async indexing jobs
- UI for files, import, and job tracking

### Phase 6. Embedding Rotation

- embedding profile management APIs
- rebuild job flow
- activate / rollback controls in frontend

### Phase 7. Optional Schema Snapshot Migration

- persist schema snapshots in PostgreSQL
- optionally phase out the current in-memory-only schema cache path

## Implementation Guardrails

- do not migrate existing data source registry to PostgreSQL in the same phase unless clearly necessary
- do not replace current SQL safety validation with retrieval logic
- do not introduce an external vector middleware
- do not allow mixed embedding profiles in one retrieval query
- do not allow retrieval without explicit `data_source_id`
