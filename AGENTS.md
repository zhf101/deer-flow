# AGENTS.md

本文件是本仓库的 AI Agent 项目操作系统。进入项目后先理解，再修改；先复用，再重构；先验证，再交付；先查阅历史经验，再做决策。

## 阅读顺序

执行任务前按需阅读以下文件：

1. `CODE_MAP.md`：快速定位模块、入口、测试位置。
2. `ARCHITECTURE.md`：理解系统边界、依赖方向、核心数据流。
3. `PRODUCT.md`：涉及业务逻辑、造数流程、用户体验时阅读。
4. `API.md`：涉及接口、前后端契约、OpenAPI 时阅读。
5. `RETRO.md`：每次开始任务必须阅读，避免重复踩坑。
6. `DECISION_DRIFT.md`：涉及架构调整、目录迁移、重构时阅读。
7. `TEST_STRATEGY.md`：改完代码前用来确定验证范围。
8. `IMPACT_MAP.md`：做影响分析和回归测试建议时阅读。

已有长文档继续作为扩展材料：

- `docs/PROJECT_SKELETON_GUIDE_ZH.md`

## 通用修改原则

- 修改前先搜索类似实现、公共组件、工具函数和既有测试。
- 优先局部修复，避免无关重构、风格统一改造和大规模目录迁移。
- 需要重构时必须先说明调用方、被调用方、配置、数据库、API 和测试影响。
- 处于开发阶段，代码方案追求简洁完善；不需要兼容旧方案时可以直接做破坏性重构。
- 表结构变更可以清空并重建旧开发数据，不需要迁移旧数据。
- 字段名变更时前端、后端、测试、文档要一起改，完全废弃旧字段和旧 API。
- 每次改完代码由开发者在合适时间自行提交 git commit，Agent 不主动提交。
- 代码实现中的注释统一写中文，不写英文注释。

## datagen 强制规则

- datagen 后端数据模型注释写在 Pydantic 层。
- Pydantic 类用 docstring 说明整体用途。
- Pydantic 字段用 `Field(description=...)` 说明前端/后端契约和运行时含义。
- datagen 相关后端 API 只使用 GET / POST。
- 查询类接口使用 GET，参数放 path 或 query。
- 修改、删除、执行类接口使用 POST + JSON body。
- datagen 不再新增 PUT / DELETE 接口。

## 修改前影响分析

每次改代码前至少识别：

- 入口：用户、前端页面、API 或 Agent 节点从哪里进入。
- 调用方：哪些模块直接调用被改对象。
- 被调用方：被改对象依赖哪些服务、仓储、模型、配置。
- 数据影响：数据库表、JSON 存储、线程文件、配置文件是否受影响。
- API 影响：请求字段、响应字段、HTTP 方法、状态码、OpenAPI 是否变化。
- 测试影响：需要补充或更新哪些单测、接口测试、前端测试、回归测试。

## 测试要求

代码修改后必须给出测试建议，至少包含：

- 功能测试：验证正常流程。
- 边界测试：验证空值、最小值、最大值、特殊字符、重复输入。
- 异常测试：验证错误参数、依赖失败、权限不足、执行失败。
- 回归测试：验证 `IMPACT_MAP.md` 中受影响模块。

## 发现新经验

如果出现以下情况，必须新增 `RETRO.md` 条目：

- 重复 Bug。
- 错误决策。
- 调试耗时超过 30 分钟。
- 已知问题再次出现。
- 测试遗漏导致返工。

同类问题累计出现 3 次及以上时，要从 `RETRO.md` 升级到本文件的强制规则。

## 交付输出

任务结束时输出：

1. 修改内容。
2. 影响范围。
3. 已执行验证。
4. 测试建议。
5. 是否新增 RETRO。
6. 是否需要升级规则。
