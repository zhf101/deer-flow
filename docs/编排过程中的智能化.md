 一、 编排过程中的 AI 能力注入：从“辅助工具”到“智能助手”

  你目前已经实现了“贴入 JSON 自动生成结构”、“MyBatis 语法解析”等能力，这属于启发式逻辑。要接入 Deer-Flow 的 AI
  能力，可以从以下三个维度切分：

   1. 语义化自动映射 (AI-Mapping)：
       * 现状：目前是通过字段名完全匹配。
       * AI 结合：当用户点击“自动关联变量”时，调用 Deer-Flow 的 Agent。AI 可以识别 idNumber 和 id_card
         是同一个东西，即使名称不同也能实现语义级的精准关联。
   2. 业务判定逻辑生成 (AI-Logic Generator)：
       * 现状：用户需要手动选 EQ、NEQ 等。
       * AI 结合：用户只需在响应配置区写一句话：“帮我配置：只要 code 不是 0000 或者 msg 包含‘失败’，就判定为业务异常。” AI
         自动将其转化为对应的 allOf/anyOf JSON 规则。
   3. SQL 与报文的智能生成 (AI-Content Creator)：
       * 在配置 SQL 节点时，用户只需描述“查询用户最近一笔有效的信用卡账单”，AI 结合已有的数据源 Schema 自动生成标准 SQL 或 MyBatis
         模板。

  二、 如何让 AI 知道有哪些造数场景？ (元数据感知)

  AI 并不直接读取你的数据库表，它通过 MCP (Model Context Protocol) 协议或 Tool (工具) 描述来感知。

   1. 场景即工具 (Scene as a Tool)：
       * 你需要为 GDP 开发一个专用的 MCP Server。
       * 这个 Server 会动态地将 df_scene 表中状态为 PUBLISHED 的场景注册为 AI 可用的“工具”。
       * 关键点：你在表单里填写的 sceneName 和 sceneRemark（备注）就是 AI 的“指令说明”。AI
         会根据这些描述来判断什么时候该调用哪个场景。
   2. 动态文档注入：
       * 利用 Deer-Flow 的记忆系统（MemoryMiddleware），定期将所有已发布的造数场景摘要同步到 AI
         的“业务知识库”中。当用户说“我想测一下下单流程”时，AI 能够通过检索 (RAG) 锁定 create_order 这个场景。

  三、 如何组合编排完成复杂任务？ (AI 规划器)

  这是最核心的能力：当用户提出一个复杂需求（如：“帮我在生产环境模拟一个高风险用户，并给他下 10 笔不同品类的订单”）时，AI 如何操作？

   1. 二级编排架构：
       * 一级编排（你现在的 GDP）：负责确定性的、高性能的底层原子动作（如：登录、下单、查库）。
       * 二级编排（AI Planner）：AI 充当“指挥官”。它发现任务需要分三步：
           1. 调用 create_high_risk_user 场景。
           2. 从输出中拿到 userId。
           3. 循环 10 次调用 place_order 场景，并动态变换 productId。
   2. 参数自动“接龙”：
       * 因为你的 GDP 场景已经定义了标准的 inputSchema 和 resultMapping，AI 具备了“强类型”感。
       * AI 会自动解析第一个场景的输出，将其填充到第二个场景的输入中。这就不再需要你手动在 UI 上连线，而是 AI
         在运行时动态生成的逻辑连线。

  一、 HAR 解析与智能过滤的闭环

  首先，解析 HAR 文件绝不能仅仅是简单的 URL 提取，必须经过 AI 代理的“脱水处理”：

   1. 静态脱水：自动剔除 .js, .css, 图片、以及前端埋点 (/track, /log) 等非业务请求。
   2. 上下文关联推导 (AI-Correlation)：
       * AI 分析连续的请求：发现接口 A 返回了 {"orderId": "1001"}，接口 B 的请求体里带着 {"id": "1001"}。
       * 智能动作：AI 会自动建议在 GDP 场景中生成一条数据映射连线（提取 A 的 orderId 作为 B 的输入），而不是将 1001 写死。
   3. 人类确认边界：将提炼出的核心链路以图形化展示，让用户剔除不必要的中间查询步骤，最终一键转换为 GDP 的 SceneDefinition 结构。

  二、 核心痛点：如何记录与管理字典/枚举等业务知识？

  在录制的报文中，常常充满 {"orderType": 2, "status": "01"} 这种黑盒数据。如果直接回放，换个环境就可能报错。为了让 AI
  读懂这些数据并能在编排时灵活运用，我们需要建立业务知识沉淀机制。

  以下是四种递进的解决方案：

  方案 1：利用 Deer-Flow 的知识图谱（Memory 系统）
  正如 @docs/造数系统设计.md 中提到的，Deer-Flow 拥有记忆能力。
   * 做法：当 AI 在解析 HAR 或用户手动修正时，如果用户指出 2 代表“分期订单”，我们就调用 MCP Tool（如
     mcp_memory_create_entities）将这个映射写入知识图谱。
   * 效果：图谱中会形成节点：Entity: 订单类型字典 -> Observation: 2=分期订单, 1=全款订单。
   * 应用：下次 AI 解析到类似接口，或者用户用自然语言说“帮我造一个分期订单”时，AI 会自动去 Memory 里查到对应的值是 2，并在 GDP
     场景入参中填入正确的值。

  方案 2：动态字典查询节点 (Dynamic Resolution)
  有些字典数据是随环境变化的（比如在联调环境和压测环境，某个配置项的 ID 不同）。
   * 做法：在 AI 分析 HAR 时，如果发现某个参数高度疑似动态字典表的主键（比如一个不规则的长 UUID）。
   * 智能动作：AI 会在 GDP 画布的最前面，自动插入一个前置的“字典查询接口 (HTTP 节点)”或“字典查库 (SQL 节点)”。
   * 然后将真实业务接口中的参数，配置为对这个前置节点查询结果的变量引用。这使得造数场景具备了环境自适应能力。

  方案 3：强化 GDP 的参数 Schema（类型元数据沉淀）
  你目前的 GDP 输入参数已经支持了 optionsSource（选项来源）和详细的 validation。
   * 做法：当通过 HAR 生成场景时，AI 发现请求参数 status 的值为 "01"。AI 可以通过上下文猜测，并在 GDP 的 InputSchemaPanel
     中，将该字段类型设置为 enum，并引导用户补全 "01":"待审核", "02":"已审核" 的描述。
   * 沉淀位置：这些信息直接沉淀在你设计的 df_scene_version.input_schema_json 中。这个场景本身就变成了一个附带字典解释的微型知识库。

  方案 4：引入全局字典中心 (Global Dictionary Registry)
  如果这种枚举极其庞大且跨场景通用（比如全公司的国家代码、行业分类）：
   * 做法：在 GDP 后端建立一张 df_business_dict 表。
   * 在前端参数配置时，optionsSource 字段可以选择绑定到全局字典。
   * AI 的角色：AI 在解析 HAR 时，如果发现报文里有 countryCode: "CN"，它去对比全局字典，发现匹配，就会自动在 GDP
     场景中将该参数打上全局字典的 Tag。

 单纯靠 HAR 文件（黑盒）只能看到表象，而引入源代码分析（白盒）则能为你提供“上帝视角”。结合 Deer-Flow 的 AI
  能力，这种方式不仅更精确，而且能解决你提到的跨系统调用、异步逻辑、隐藏依赖等核心难题。

  以下是结合源码分析后，造数系统能力的质变：

  1. 填补“看不见”的空隙：跨服务调用追踪 (East-West Traffic)
   * 黑盒局限：HAR 文件只能记录用户浏览器发出的请求。如果后台 API A 收到请求后，通过 Feign 或 Dubbo 偷偷调用了服务 B 和服务 C，HAR
     是完全不知道的。
   * 白盒增强：AI 拿到源码后，可以从 API A 的 Controller 层开始，一路向下追踪：
       * 识别逻辑：Controller -> Service -> FeignClient。
       * AI 发现：AI 会告诉你：“虽然你只贴了一个 A 请求，但源码显示它依赖了服务 B 的查询。为了保证造数成功，我建议在 GDP
         场景中增加一个针对服务 B 的前置 mock 或真实调用节点。”

  2. 数据库副作用的“精准定位”
   * 现状：很多造数任务最终是为了在数据库里落下一条特定状态的数据。
   * 结合源码：AI 扫描 Service 层的 @Transactional 方法或 MyBatis 的 Mapper.xml。
   * 自动推导 SQL：AI 会自动识别出：“这个下单操作最终操作了 t_order 和 t_stock
     两张表。如果您希望跳过复杂的接口调用直接造数，我可以根据源码里的 SQL 逻辑，直接为您生成 GDP 的 SQL 节点。”

  3. 解决登录态与复杂的会话共享
   * 问题：用户从系统 A 跳到系统 B，Token 是如何传递的？Session 怎么同步的？
   * 白盒分析：AI 扫描代码中的 Interceptor（拦截器）或 SecurityConfig。
   * 自动配置认证：AI 能识别出系统 B 是如何解析来自 A 的加密 Cookie 或 JWT 的。它会直接在 GDP 的 AUTH_HTTP
     步骤中配置好正确的解密/转换逻辑，而不需要用户去猜测。

  4. 落地流程：AI 驱动的“源码到编排”路径

  如果将你的 GDP 系统与 Deer-Flow 的 AI 源码分析能力结合，工作流将变成这样：

   1. 确定入口：用户在聊天框输入：“我想梳理‘创建贷款合同’的流程，这是它的入口 URL：/api/contract/create。”
   2. AI 深入源码 (Deep Dive)：
       * AI 调用 codebase_investigator 寻找对应的 Java/Python 后台处理类。
       * AI 递归分析调用栈：发现它先校验了用户资格（查库），再调用了风控接口（HTTP），最后写入合同表（SQL）。
   3. 智能蓝图生成：
       * AI 自动在你的 SceneEditor 中绘图：
           * 节点 1 (SQL)：根据源码逻辑查用户表，验证状态。
           * 节点 2 (HTTP)：调用风控系统，入参结构自动根据 Feign 定义生成。
           * 节点 3 (HTTP)：正式的合同创建接口。
   4. 业务逻辑翻译：
       * AI 会把代码里的 if (status == 1) 翻译成你刚才要求的“业务成功规则”：当 status 等于 1 时判定为成功。

  5. 解决“大量字典/枚举”的终极方案：静态代码分析

  针对你之前担心的字典存储问题，结合源码后有了完美的解法：

   * 枚举类提取：AI 直接扫描项目的 Enums 目录或 constant 包。
   * 自动映射：当 AI 在代码里看到 order_status 字段关联了 OrderStatusEnum 时，它会自动把枚举里的中文注释提取出来，填入你 GDP 系统的
     inputSchema 的 label 字段。
   * 知识库化：代码里的 // 1-待支付, 2-已支付 这种注释，会被 AI 自动作为“业务知识”存储到 Deer-Flow 的 Memory
     中。这样以后任何人问“订单状态 2 是什么意思”，AI 都能根据它看过的源码给出准确答案。
