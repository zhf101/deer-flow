# 造数编排系统实施计划

## 1. 目标和范围

本计划用于落地造数编排系统的配置侧能力，先完成“可建模、可持久化、可编辑、可保存、可发布”的闭环。

本期实施范围：

- 定义造数编排数据模型。
- 设计数据库表结构。
- 实现后端配置保存、查询、校验、发布接口。
- 实现前端可视化编排页面。
- 支持 HTTP、AUTH_HTTP、SQL、ASSERT、TRANSFORM 步骤的配置保存。
- 支持复杂 JSON 请求体、响应成功/失败规则、错误映射等配置保存。

本期不实现：

- `/v1/scene/run` 场景执行引擎。
- HTTP 实际调用。
- SQL 实际执行。
- Excel 批量执行。
- 运行日志和运行状态机。
- 凭证真实解密和使用。

执行引擎后续设计时必须复用本期保存的数据模型，避免重新设计配置结构。

## 2. 总体实施顺序

按以下顺序推进：

1. 数据模型：先定义后端 Pydantic 模型和前端 TypeScript 类型。
2. 数据表结构：设计并创建场景、版本、环境、数据源、SQL 模板等配置表。
3. 后端保存能力：实现配置 CRUD、结构校验、发布版本。
4. 前端编排页面：实现列表、基础信息、参数配置、步骤编排、JSON 配置编辑、保存和发布。
5. 联调验收：用一个完整场景验证从前端配置到后端落库的闭环。

## 5. 数据模型设计

### 5.1 场景定义模型

```json
{
  "sceneCode": "create_test_order",
  "sceneName": "创建测试订单",
  "sceneRemark": "创建订单、更新状态并返回订单号",
  "sceneType": "credit_card",
  "environmentField": "env",
  "inputSchema": [],
  "steps": [],
  "resultMapping": {},
  "batchConfig": {},
  "status": "DRAFT"
}
```

关键规则：

- `sceneCode` 全局唯一。
- `environmentField` V1 固定为 `env`。
- `steps` 按数组顺序执行，`dependsOn` 不影响顺序。
- `resultMapping` 是唯一最终返回定义，不引入 `END` 步骤。

### 5.2 输入参数模型

```json
{
  "name": "customerId",
  "label": "客户编号",
  "type": "string",
  "required": true,
  "defaultValue": null,
  "optionsSource": null,
  "validation": {
    "minLength": 1,
    "maxLength": 64,
    "pattern": null
  },
  "batchEnabled": true
}
```

字段类型：

```text
string
number
boolean
date
enum
object
array
```

### 5.3 步骤公共模型

```json
{
  "stepId": "createOrder",
  "stepName": "创建订单",
  "type": "HTTP",
  "enabled": true,
  "dependsOn": [],
  "description": "",
  "position": {
    "x": 100,
    "y": 100
  }
}
```

公共规则：

- `stepId` 在场景内唯一。
- `dependsOn` 只用于校验和可视化。
- `position` 只服务前端画布。

### 5.4 HTTP 步骤模型

```json
{
  "stepId": "createOrder",
  "type": "HTTP",
  "method": "POST",
  "url": "${env.services.orderService.baseUrl}/api/orders",
  "requestMapping": {
    "headers": {
      "Authorization": "Bearer ${auth.login.token}",
      "Content-Type": "application/json"
    },
    "query": {},
    "body": {
      "customer": {
        "customerId": "${input.customerId}"
      },
      "order": {
        "amount": "${input.orderAmount}"
      }
    }
  },
  "responseHandling": {
    "expectedContentType": "JSON",
    "statusCode": {
      "success": [200]
    },
    "businessSuccess": {
      "allOf": [
        {
          "path": "$.body.success",
          "op": "EQ",
          "value": true
        }
      ]
    },
    "businessFailure": {
      "anyOf": [
        {
          "path": "$.body.errorCode",
          "op": "EXISTS"
        }
      ]
    }
  },
  "errorMapping": {
    "messageTemplate": "创建订单失败：${error.bizCode} - ${error.bizMessage}",
    "fields": {
      "bizCode": "$.body.code",
      "bizMessage": "$.body.message"
    },
    "fallbackMessage": "创建订单失败，请查看执行日志",
    "exposeRawResponse": false
  },
  "outputMapping": {
    "orderNo": "$.body.data.orderNo"
  },
  "retryPolicy": {
    "enabled": true,
    "maxAttempts": 3,
    "intervalMs": 1000,
    "retryOn": ["NETWORK_TIMEOUT", "CONNECTION_RESET"]
  }
}
```

### 5.5 AUTH_HTTP 步骤模型

`AUTH_HTTP` 复用 HTTP 配置，并增加：

```json
{
  "authMapping": {
    "token": "$.body.data.token",
    "tokenType": "Bearer"
  }
}
```

保存规则：

- 认证结果只写配置模型，不在本期执行。
- 编排配置中不能保存明文密码或 token。

### 5.6 SQL 步骤模型

```json
{
  "stepId": "updateOrderStatus",
  "type": "SQL",
  "datasource": "${env.datasources.tradeDb}",
  "sqlTemplateCode": "update_order_status",
  "operation": "UPDATE",
  "paramMapping": {
    "orderNo": "${steps.createOrder.outputs.orderNo}",
    "status": "PAID"
  },
  "assertions": [
    {
      "expression": "${steps.updateOrderStatus.result.affectedRows} == 1"
    }
  ],
  "outputMapping": {
    "affectedRows": "$.affectedRows"
  }
}
```

### 5.7 ASSERT 步骤模型

```json
{
  "stepId": "assertOrderCreated",
  "type": "ASSERT",
  "assertions": [
    {
      "expression": "${steps.createOrder.outputs.orderNo} NOT_EMPTY"
    }
  ]
}
```

### 5.8 TRANSFORM 步骤模型

```json
{
  "stepId": "buildOrderRemark",
  "type": "TRANSFORM",
  "assignments": {
    "vars.orderRemark": "AUTO_${steps.createOrder.outputs.orderNo}_${system.now}"
  }
}
```

### 5.9 SQL 模板模型

```json
{
  "templateCode": "update_order_status",
  "templateName": "更新订单状态",
  "operation": "UPDATE",
  "datasourceType": "MYSQL",
  "sqlText": "update t_order set status = :status where order_no = :orderNo",
  "parameters": [
    {
      "name": "orderNo",
      "type": "string",
      "required": true
    }
  ],
  "safety": {
    "requireWhere": true,
    "maxAffectedRows": 1
  },
  "status": "ENABLED"
}
```

## 8. 后端校验规则

保存草稿时做轻量校验：

- `sceneCode` 非空、格式合法、唯一。
- `sceneName` 非空。
- `inputSchema` 字段名唯一。
- `steps.stepId` 唯一。
- `steps.type` 是允许值。
- JSON 字段可序列化。

发布前做完整结构校验：

- `env` 入参存在且必填。
- 每个步骤引用的变量路径语法合法。
- 引用的历史步骤必须存在且出现在当前步骤之前。
- `dependsOn` 中步骤必须存在且出现在当前步骤之前。
- `outputMapping` 字段名唯一。
- HTTP 步骤必须配置 method、url、responseHandling。
- HTTP `businessSuccess.allOf` 至少有一条规则。
- SQL 步骤引用的 `sqlTemplateCode` 必须存在且启用。
- SQL `paramMapping` 必须覆盖模板必填参数。
- `resultMapping` 引用的变量必须可解析到历史步骤或输入。

本期不校验：

- HTTP URL 是否真实可访问。
- SQL 是否真实能执行。
- JSONPath 是否一定能从真实响应中取到值。
- 业务规则运行时结果。

## 9. 前端页面设计

页面入口：

```text
/workspace/data-factory
```

### 9.1 页面布局

建议采用三栏结构：

```text
左侧：场景列表 / 新建场景 / 分类筛选
中间：编排画布 / 步骤顺序
右侧：基础信息、参数、步骤配置、返回结果、批量配置
```

前端只做编排配置，不做执行调试。

### 9.2 场景列表

能力：

- 查询场景列表。
- 按 `sceneType`、`status`、关键词筛选。
- 新建场景。
- 进入编辑。
- 查看当前版本和状态。

### 9.3 基础信息面板

字段：

- `sceneCode`
- `sceneName`
- `sceneRemark`
- `sceneType`
- `status`

规则：

- 已发布版本不可直接编辑。
- 编辑已发布场景时，前端提示创建新草稿。

### 9.4 输入参数面板

能力：

- 增删改 `inputSchema` 字段。
- 设置字段类型、是否必填、默认值、选项来源、校验规则、是否参与批量。
- `env` 字段默认存在，不允许删除。
- 支持 `object`、`array` 类型入参，为复杂 JSON 请求体透传做准备。

### 9.5 编排画布

使用 `@xyflow/react`。

能力：

- 添加 HTTP、AUTH_HTTP、SQL、ASSERT、TRANSFORM 节点。
- 节点按 `steps` 数组顺序保存。
- 连线只表达 `dependsOn` 和可视化依赖，不决定执行顺序。
- 支持调整节点位置并保存到步骤 `position`。
- 点击节点后在右侧显示配置面板。

### 9.6 HTTP 配置面板

能力：

- 配置 method、url、headers、query。
- 用 JSON 编辑器配置复杂 `requestMapping.body`。
- 配置 `responseHandling.statusCode.success`。
- 配置 `businessSuccess.allOf`。
- 配置 `businessFailure.anyOf`。
- 配置 `errorMapping.fields`、`messageTemplate`、`fallbackMessage`。
- 配置 `outputMapping`。
- 配置 `retryPolicy`。

建议使用 CodeMirror 编辑 JSON，并提供格式化和 JSON 语法校验。

### 9.7 SQL 配置面板

能力：

- 选择 SQL 模板。
- 展示模板参数。
- 配置 `paramMapping`。
- 配置 `assertions`。
- 配置 `outputMapping`。

### 9.8 TRANSFORM 和 ASSERT 面板

能力：

- TRANSFORM 配置 `assignments`。
- ASSERT 配置断言表达式列表。
- 提供上下文变量选择辅助。

### 9.9 返回结果和批量配置

返回结果：

- 配置 `resultMapping`。
- 支持引用 `input`、`steps`、`vars`、`system`。

批量配置：

- 配置是否启用批量。
- 配置默认失败策略。
- 配置并发模式。
- 默认 `maxConcurrency=1`。

## 10. 实施任务拆分

### Task 1：定义后端 Pydantic 数据模型
