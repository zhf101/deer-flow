# 09 记忆系统

对应字幕：`09-记忆系统_哔哩哔哩_bilibili_BV1SVdHBEEhZ_字幕.srt`

## 本章目标

这一集讲 DeerFlow 如何把有用信息跨会话保存下来。源码入口：

- `backend/packages/harness/deerflow/agents/middlewares/memory_middleware.py`
- `backend/packages/harness/deerflow/agents/memory/queue.py`
- `backend/packages/harness/deerflow/agents/memory/storage.py`
- `backend/packages/harness/deerflow/agents/memory/updater.py`
- `backend/packages/harness/deerflow/config/memory_config.py`
- `backend/app/gateway/routers/memory.py`

## 记忆不等于聊天记录

DeerFlow 的记忆不是把所有消息永久拼进上下文。它存储的是跨会话有价值的摘要和事实：

```json
{
  "version": "1.0",
  "lastUpdated": "...",
  "user": {
    "workContext": {"summary": "", "updatedAt": ""},
    "personalContext": {"summary": "", "updatedAt": ""},
    "topOfMind": {"summary": "", "updatedAt": ""}
  },
  "history": {
    "recentMonths": {"summary": "", "updatedAt": ""},
    "earlierContext": {"summary": "", "updatedAt": ""},
    "longTermBackground": {"summary": "", "updatedAt": ""}
  },
  "facts": []
}
```

自然语言摘要适合注入给 LLM 阅读；结构化 facts 适合去重、排序、过滤和置信度控制。

## `MemoryMiddleware`

文件：`memory_middleware.py`

它在 `after_agent()` 中工作：

1. 读取 memory 配置，未启用则跳过。
2. 解析 `thread_id`。
3. 从 state 读取 `messages`。
4. 过滤出适合记忆提取的用户输入和最终助手回复。
5. 检测 correction / reinforcement 信号。
6. 捕获当前 `user_id`。
7. 调用 `get_memory_queue().add(...)` 加入后台队列。

它不直接调用 LLM 更新记忆，这是刻意的性能设计。

## 为什么异步和防抖

同步记忆更新会让用户当前请求变慢，而且连续消息会重复触发 LLM 提取。

`MemoryUpdateQueue` 用 `threading.Timer` 做防抖：

- 新消息到来时取消旧 timer。
- 把同一 `(thread_id, user_id, agent_name)` 的待处理项合并。
- 等 `debounce_seconds` 后批量处理。

默认防抖时间来自 `memory.debounce_seconds`。字幕里说“30 秒”的设计，对应默认配置思路：记忆是跨会话优化，不需要在当前对话内实时生效。

## 存储隔离

`FileMemoryStorage` 支持多层路径：

- 无用户无 Agent：legacy 全局 `memory.json`。
- 有用户无 Agent：`users/{user_id}/memory.json`。
- 有用户有 Agent：`users/{user_id}/agents/{agent_name}/memory.json`。
- 无用户有 Agent：legacy `agents/{agent_name}/memory.json`。

存储读写带 mtime 缓存，保存时先写临时文件再 `replace()`，减少写坏文件的风险。

## LLM 更新流程

`MemoryUpdater.update_memory()` 会：

1. 加载当前 memory。
2. 把对话格式化进 `MEMORY_UPDATE_PROMPT`。
3. 调用配置的模型或默认模型。
4. 从模型输出里解析第一个合法 JSON 对象。
5. 规范化 `newFacts`、`factsToRemove`。
6. 应用更新。
7. 移除文件上传事件类记忆，避免未来会话找不存在的上传文件。
8. 保存结果。

facts 写入时会检查：

- `confidence >= fact_confidence_threshold`
- 内容去重
- `max_facts` 上限
- 超限时按 confidence 保留高价值事实

## correction 和 reinforcement

源码会检测用户纠错或正向确认，并把提示加进记忆更新 prompt：

- 纠错时更重视“正确做法”，可记录为 `correction` 高置信事实。
- 正向确认时更重视用户偏好或行为模式。

这让记忆系统不是简单摘要，而能学习用户对 Agent 输出的反馈。

## 与摘要中间件的关系

`_create_summarization_middleware()` 在启用记忆时会添加 `memory_flush_hook`。这表示在旧消息被摘要压缩前，记忆系统有机会先处理即将丢失的详细上下文。

这是长时序 Agent 的关键：摘要负责短期上下文压缩，记忆负责跨会话长期保留。

## 常见误区

不要把上传文件路径长期写进记忆。源码专门用 `_strip_upload_mentions_from_memory()` 去掉上传事件，因为 `/mnt/user-data/uploads` 是线程级临时上下文，未来会话不一定存在。

也不要把所有对话原文写入记忆。那会把记忆变成另一个无限增长的上下文窗口，失去结构化和压缩价值。

## 本章结论

DeerFlow 的记忆系统是异步、按用户/Agent 隔离、带置信度和防抖的长期上下文提取器。它保存的是可复用事实和摘要，不是完整聊天日志。
