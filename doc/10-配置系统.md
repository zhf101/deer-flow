# 10 配置系统

对应字幕：`10-配置系统_哔哩哔哩_bilibili_BV1SVdHBEEhZ_字幕.srt`

## 本章目标

这一集讲 `config.yaml`、`extensions_config.json`、热重载和环境变量解析。源码入口：

- `backend/packages/harness/deerflow/config/app_config.py`
- `backend/packages/harness/deerflow/config/extensions_config.py`
- `backend/packages/harness/deerflow/config/paths.py`
- `config.example.yaml`
- `extensions_config.example.json`

## 两个配置文件

DeerFlow 把配置拆成两类。

### `config.yaml`

核心运行时配置：

- 模型 `models`
- 工具 `tools`
- 工具组 `tool_groups`
- 沙箱 `sandbox`
- 技能路径 `skills`
- 记忆 `memory`
- 摘要 `summarization`
- 子 Agent `subagents`
- checkpointer、database、stream bridge、run events 等

这是部署级配置，通常由运维或开发者手工修改。

### `extensions_config.json`

扩展状态配置：

- MCP servers
- skills enabled/disabled 状态
- MCP interceptors 等额外扩展字段

这是运行时可被 Gateway API 修改的配置。前端启用/禁用 MCP 或技能时，写的是这个文件。

## 为什么不合并

分开有三点实际价值：

- 安全边界：`config.yaml` 可能包含模型端点和密钥引用，不应该被前端管理接口随意写。
- 修改频率：核心配置少改，扩展配置经常通过 UI/API 改。
- 格式匹配：YAML 适合人工维护和注释，JSON 适合 API round-trip 和程序写入。

## `AppConfig.from_file()`

加载流程：

1. `resolve_config_path()` 找配置路径。
2. 读取 YAML。
3. `_check_config_version()` 比较 `config_version`。
4. `resolve_env_variables()` 递归解析 `$ENV_VAR`。
5. `_apply_database_defaults()` 补数据库默认值。
6. 单独加载 `ExtensionsConfig.from_file()`。
7. Pydantic `model_validate()`。
8. `_apply_singleton_configs()` 同步部分 legacy singleton 配置。

`resolve_config_path()` 的优先级：

1. 显式参数。
2. `DEER_FLOW_CONFIG_PATH`。
3. caller project root。
4. legacy backend/repo root。

## 热重载

`get_app_config()` 使用模块级缓存和文件 mtime：

- 如果配置路径或 mtime 没变，返回缓存。
- 如果 mtime 变化，重新加载。
- 如果存在 runtime `ContextVar` override，优先返回 override。

Gateway 的 `deps.get_config()` 每个请求都走 `get_app_config()`，所以请求级配置可以热重载。

但不是所有东西都热重载。`langgraph_runtime()` 启动时创建的 checkpointer、store、event store、stream bridge 等持有连接和句柄，绑定启动时 snapshot。改这些配置通常需要重启。

## 环境变量解析

`config.yaml` 中字符串如果以 `$` 开头，会用环境变量替换。找不到会报错。

`extensions_config.json` 中环境变量找不到时会写成空字符串，避免把 `$VAR` 原样传给 MCP 服务器。

这个差异是合理的：核心配置缺少密钥应该 fail fast；扩展配置可能包含未启用服务或可选 secret。

## 配置版本

`_check_config_version()` 会读取当前 `config.yaml` 和 `config.example.yaml` 的 `config_version`。如果用户配置版本低于示例版本，记录 warning，提示运行配置升级。

这解决的是长期演进问题：配置字段不断增加，旧用户的本地配置不会自动包含新字段。

## 路径配置

`Paths` 是运行时数据目录的统一入口。重要环境变量：

- `DEER_FLOW_HOME`：指定应用数据目录。
- `DEER_FLOW_HOST_BASE_DIR`：容器内 Docker-outside-of-Docker 场景下，指定宿主机可见路径。

`Paths` 同时提供容器虚拟路径和宿主路径，避免 Windows/Linux 路径风格混乱。

## 本章结论

DeerFlow 配置系统的核心是“部署核心配置”和“运行时扩展配置”分离。`config.yaml` 管模型、工具、沙箱、存储等基础行为；`extensions_config.json` 管 MCP 和技能开关。热重载基于 mtime，够简单可靠，也避免文件系统监听在容器和跨平台环境里的复杂性。
