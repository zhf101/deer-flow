# 06 工具系统

## 本章目标

字幕这一集讲工具从哪里来、为什么用反射加载、工具组如何限制能力、内置工具为什么特殊。源码入口：

- `backend/packages/harness/deerflow/tools/tools.py`
- `backend/packages/harness/deerflow/tools/builtins/`
- `backend/packages/harness/deerflow/sandbox/tools.py`
- `backend/packages/harness/deerflow/mcp/`
- `backend/packages/harness/deerflow/skills/tool_policy.py`

## LLM 怎么使用工具

工具调用不是模型真的执行命令，而是：

1. 后端把工具名称、描述、参数 schema 绑定给模型。
2. 模型在响应中声明要调用哪个工具以及参数。
3. DeerFlow 的 ToolNode / middleware 执行真实工具。
4. 执行结果作为 `ToolMessage` 回到 `messages`。
5. 模型基于工具结果继续推理或给最终回答。

所以工具描述和参数 schema 是模型决策的一部分，不是普通注释。`sandbox/tools.py` 里每个工具的 docstring 都强调参数顺序和路径要求，就是为了降低模型误用概率。

## 工具来源

`get_available_tools()` 是统一入口，文件：

`backend/packages/harness/deerflow/tools/tools.py`

它收集几类工具。

### 1. 配置工具

来自 `config.yaml` 的 `tools:` 配置。每个工具通过 `use` 字段反射加载：

```yaml
tools:
  - use: deerflow.sandbox.tools:read_file_tool
```

源码通过 `resolve_variable(cfg.use, BaseTool)` 动态导入。这让部署方不用改源码就能增删工具。

### 2. 内置工具

`BUILTIN_TOOLS` 默认包含：

- `present_file_tool`
- `ask_clarification_tool`

这些工具有框架级语义，不只是普通动作：

- `present_files` 会把产出物呈现给前端。
- `ask_clarification` 会被 `ClarificationMiddleware` 拦截并中断运行。

如果模型支持视觉，还会按模型配置加入 `view_image_tool`。

### 3. 子 Agent 工具

当 `subagent_enabled=True` 时加入 `task_tool`。这个工具不是始终可见，因为不是所有会话都需要委派子任务。

### 4. MCP 工具

如果 `include_mcp=True`，会从 MCP 缓存读取工具。启用 `tool_search` 时，MCP 工具会先注册到延迟工具 registry，模型通过 `tool_search` 按需提升工具，而不是一次性看到所有 schema。

### 5. ACP 工具

如果配置了 `acp_agents`，会加入 `invoke_acp_agent`，让 DeerFlow 调用外部 ACP Agent。

## 工具组

`get_available_tools(groups=...)` 会先按 `ToolConfig.group` 过滤配置工具。自定义 Agent 可以配置自己的 `tool_groups`，从而只暴露某类工具。

工具组解决的是权限和上下文污染问题：

- 权限：让某个 Agent 不具备危险工具。
- 质量：工具越多，模型选择错误概率越高。
- 可维护：按能力域组织工具，而不是在代码里写死。

## Host bash 安全开关

`get_available_tools()` 会调用 `is_host_bash_allowed(config)`。当 LocalSandboxProvider 且未显式允许主机 bash 时，会过滤掉 host bash 工具。

这点很重要：本地沙箱不是容器隔离。直接暴露 `bash` 相当于给模型主机 shell 权限，所以 DeerFlow 默认关闭，推荐容器沙箱承载命令执行。

## 工具去重

源码最后按 `tool.name` 去重：

```python
seen_names: set[str] = set()
for t in all_tools:
    if t.name not in seen_names:
        unique_tools.append(t)
```

如果配置工具、内置工具或 MCP 工具重名，后出现的会被跳过并记录 warning。这样避免模型收到歧义 schema，也避免运行时路由不知道该调用哪一个。

## Sandbox 文件工具

文件：`backend/packages/harness/deerflow/sandbox/tools.py`

常用工具包括：

- `bash`
- `ls`
- `glob`
- `grep`
- `read_file`
- `write_file`
- `str_replace`

这些工具都接收注入的 `Runtime`，从 `runtime.state` 和 `runtime.context` 找到 `thread_data`、`sandbox`、`thread_id`。LocalSandbox 模式下还会做虚拟路径校验和路径翻译。

`read_file`、`ls`、`grep`、`bash` 都有输出截断逻辑，避免工具结果把上下文塞爆。`write_file` 和 `str_replace` 使用文件操作锁，避免同一沙箱同一路径的读改写竞争。

## Skills 如何影响工具

文件：`backend/packages/harness/deerflow/skills/tool_policy.py`

如果加载的技能声明了 `allowed-tools`，DeerFlow 会取这些白名单的并集，并过滤 Agent 可见工具。没有任何技能声明 `allowed-tools` 时，保留传统的 allow-all 行为。

这意味着技能系统不仅注入方法论，也能限制工具边界。

## 本章结论

DeerFlow 的工具系统不是简单的“函数列表”。它是一个由配置、内置语义、MCP、ACP、子 Agent、技能白名单、工具组、沙箱安全共同决定的能力集合。读源码时抓住 `get_available_tools()`，就能看清工具如何从配置进入模型上下文。

## 结合当前源码的补充分析

`get_available_tools()` 的设计精髓是“模型可见能力的运行时装配”。它不是把所有 Python 函数都丢给模型，而是在每次 Agent 构造时，根据模型、Agent、技能、工具组、MCP、subagent 开关、安全配置计算出一组具体工具。

源码里工具进入模型上下文的路径大致是：

```text
config.tools 反射加载
  -> host bash 安全过滤
  -> 内置工具追加
  -> subagent 工具按运行时开关追加
  -> vision 工具按模型能力追加
  -> MCP 工具从缓存追加或 deferred 注册
  -> ACP 工具按配置追加
  -> 按 tool.name 去重
  -> 技能 allowed-tools 二次过滤
  -> create_agent(... tools=...)
```

### 配置工具：反射加载但以工具自身名称为准

`ToolConfig` 只有三个核心字段：

- `name`
- `group`
- `use`

`use` 通过 `resolve_variable(cfg.use, BaseTool)` 反射加载真实工具对象。这里有一个重要细节：`get_available_tools()` 会检查 `cfg.name` 和 `loaded.name` 是否一致，但最终绑定给模型的是工具对象自己的 `.name`。如果两者不一致，只记录 warning，不用配置名覆盖工具名。这样做是对的，因为 LangChain ToolNode 最终按工具对象名称路由。

这说明工具名字是运行时契约，不是 UI 标签。工具重命名会影响模型调用、ToolNode 路由、middleware 判断和前端事件归因。

### 工具组：粗粒度能力边界

`get_available_tools(groups=...)` 先按 `ToolConfig.group` 过滤配置工具。自定义 Agent 的 `tool_groups` 会传进这里，所以工具组是 Lead Agent 最早的一层能力边界。

但工具组只作用于配置工具，内置工具、MCP、ACP 和 subagent 工具是后续追加的。因此如果希望某个 Agent 连 `task`、`present_files`、`ask_clarification` 这类内置工具都不可见，不能只靠 `tool_groups`，还需要技能白名单、subagent config 的 disallowed tools，或者额外 middleware/工厂逻辑。

### host bash 过滤：工具系统内置安全兜底

`is_host_bash_allowed(config)` 会在工具列表构造阶段过滤 host bash。判断 `_is_host_bash_tool()` 时不仅看 `use == "deerflow.sandbox.tools:bash_tool"`，也看 `group == "bash"`。

这不是纯权限配置，而是运行时环境判断。LocalSandboxProvider 下直接暴露 bash 等于暴露主机 shell，所以 DeerFlow 默认不把 host bash 工具交给模型。这里体现了工具系统的一个原则：危险能力应该在工具进入模型上下文之前就被剔除，而不是等模型调用后再拒绝。

### 内置工具：函数本身只是契约入口

内置工具和普通工具不同，它们通常必须配合 state 或 middleware 才完整：

- `present_files`：校验文件必须在 `/mnt/user-data/outputs` 下，返回 `Command(update={"artifacts": ...})`，依赖 `ThreadState.artifacts` reducer。
- `ask_clarification`：工具函数只是占位，真正的中断由 `ClarificationMiddleware` 拦截 tool call 后返回 `Command(goto=END)`。
- `view_image`：读取图片、校验虚拟路径、写入 `viewed_images`，再由 `ViewImageMiddleware` 在下一次模型调用前注入图片内容。
- `task`：启动子 Agent，继承父 Agent 的 thread/sandbox/tool_groups/skills/model 上下文，并把结果回传给父 Agent。
- `setup_agent` / `update_agent`：只在 bootstrap 或自定义 Agent 分支额外加入，不是默认 Lead Agent 的普通工具。

所以内置工具不是普通业务函数。它们经常是“工具函数 + state reducer + middleware + 前端事件”的组合能力。

### 技能 allowed-tools：不是每个技能都自动放权

`skills/tool_policy.py` 的逻辑很克制：

- 如果没有任何加载的 skill 声明 `allowed-tools`，返回 `None`，保留历史 allow-all 行为。
- 只要有一个 skill 显式声明 `allowed-tools`，就进入白名单模式。
- 多个 skill 的白名单取并集。
- 声明 `allowed-tools: []` 的 skill 不贡献任何工具。

这意味着技能不仅是 Prompt 方法论，也可以成为工具权限边界。但它不是强安全边界，因为前提是“当前 Agent 加载了哪些 skill”本身由工厂决定。真正高风险工具仍要靠工具组、guardrail、sandbox audit 和运行时策略共同限制。

### deferred tool：解决 MCP schema 膨胀

`tool_search` 机制解决的是工具过多导致的上下文污染。MCP 工具很多时，如果一次性把所有 schema 绑定给模型，会出现两个问题：

- prompt token 被工具 schema 消耗。
- 模型在大量工具里误选概率上升。

开启 `tool_search` 后，MCP 工具会进入 `DeferredToolRegistry`。模型只在系统 Prompt 的 `<available-deferred-tools>` 里看到工具名，完整 schema 暂时不绑定。模型需要先调用 `tool_search`，返回匹配工具的 OpenAI function schema；registry 随后 `promote()` 这些工具，`DeferredToolFilterMiddleware` 后续就不再过滤它们。

这里有两个细节很重要：

1. `DeferredToolRegistry` 用 `ContextVar` 保存，所以每次 async 图运行有独立 deferred registry，避免并发请求互相污染。
2. `get_available_tools()` 重入时会复用已有 registry，保留已经 promote 的工具。注释里明确提到这是为了避免 `task_tool` 创建子 Agent 时重建 registry，把父 Agent 已经提升的工具重新隐藏。

因此 deferred tool 不是静态过滤，而是一种“本次运行内逐步扩展工具可见性”的状态机。

### MCP 工具：不只是加载，还管理会话

MCP 工具通过 `mcp/cache.py` 缓存，并根据 `extensions_config.json` 的 mtime 判断是否失效。`get_cached_mcp_tools()` 支持懒初始化，这样 FastAPI 和 LangGraph Studio 都能工作。

`mcp/tools.py` 还有一层会话管理：

- stdio MCP 工具会被包装成持久 session tool，按 `(server_name, thread_id)` 复用 session。
- HTTP/SSE MCP 工具不池化，避免 anyio TaskGroup 在跨 task 清理时出错。
- OAuth 和自定义 interceptor 会在工具调用前介入。
- MCP 返回的 `CallToolResult` 会转换成 LangChain content/artifact 格式，错误会转为 `ToolException`。

这说明工具系统不仅管理“模型能调用什么”，也管理“工具运行时和外部服务的连接生命周期”。

### subagent task 工具：继承父 Agent 的能力边界

`task_tool` 不是简单异步函数。它会从父 runtime 中读取：

- `sandbox`
- `thread_data`
- `thread_id`
- `metadata.model_name`
- `metadata.tool_groups`
- `metadata.available_skills`
- `trace_id`
- runtime 注入的 `app_config`

然后它做几件关键事情：

- 校验 `subagent_type` 是否存在，bash subagent 还要检查 host bash 是否允许。
- 子 Agent 继承父 Agent 的 `tool_groups`。
- 子 Agent 禁止继续暴露 `task`，避免递归嵌套。
- 父 Agent 的 `available_skills` 会和子 Agent 配置的 `skills` 做交集，避免子 Agent 越权获得父 Agent 不可见技能。
- 后台执行子 Agent，父工具调用每 5 秒轮询结果并发 SSE 事件。
- token usage 会缓存到 `tool_call_id`，再由 `TokenUsageMiddleware` 归因回父 Agent 的 AIMessage。
- 父 run 被取消时，会请求子 Agent 协作取消，并尽量等待终态以回收 usage 和后台任务。

这部分的设计精髓是：子 Agent 是工具系统的一部分，但不是普通工具。它必须继承父上下文，同时又要阻止递归、越权和无界并发。

## GDP 当前工具系统对照

GDP 当前工具目录在 `backend/app/gdp/agent/tools/`，包括：

- `task_tools.py`
- `scene_tools.py`
- `scene_design_tools.py`
- `source_config_tools.py`
- `infra_config_tools.py`

这些文件已经按业务域拆分了能力，但它们和 Lead Agent 工具系统不是同一种形态。

### GDP 工具更多是业务 service 封装

当前 GDP 工具函数主要是对业务服务的薄封装：

- `search_scene_contracts()` 调 `AgentCatalogService.search_scene_contracts()`。
- `bind_scene_inputs()` 根据场景入参、用户输入和变量栈做绑定。
- `run_datagen_scene_for_task()` 执行场景，并写 `TaskStep/TaskEvent/visibleVariables`。
- `search_source_contracts()` 搜索 HTTP/SQL Source 能力契约。
- `publish_scene_from_source()` 生成、保存、发布场景，并记录任务步骤和事件。
- `upsert_http_source_from_agent()` / `upsert_sql_source_from_agent()` 保存 Source，缺基础配置时返回 `INFRA_CONFIG` 缺口。
- `resolve_infra_basis()` / `upsert_*_from_agent()` 处理系统、环境、端点、数据源。
- `get_datagen_task_state()` 返回 TaskRun、计划、变量栈、最近事件摘要。

这些能力的权威输出不是 `ToolMessage`，而是 TaskRun、TaskStep、TaskEvent、SceneRun、Source/Infra 配置表。

### build_*_tools 目前没有接入 GDP 主图

GDP 文件里已经有 `build_scene_tools()`、`build_source_config_tools()`、`build_infra_config_tools()`、`build_task_tools()`，它们会返回 `StructuredTool`。

但从当前代码搜索看，GDP 主图节点没有通过统一 `get_gdp_tools()` 把这些工具交给 LLM ToolNode，而是直接导入底层函数调用。例如：

- `scene_fulfillment` 直接调用 `search_scene_contracts()`、`bind_scene_inputs()`、`run_datagen_scene_for_task()`。
- `scene_design` 直接调用 `search_source_contracts()`、`publish_scene_from_source()`。
- `source_config` 直接调用 `upsert_http_source_from_agent()`、`upsert_sql_source_from_agent()`。
- `infra_config` 直接调用 `resolve_infra_basis()` 和 upsert 函数。

所以现阶段 GDP 的“工具系统”更像业务能力函数库，而不是 Lead Agent 那种模型可见工具市场。这不是坏事，因为当前 GDP 图是确定性业务编排，节点直接调用 service 更容易控制副作用、恢复和审计。

### GDP 工具的风险比通用工具更业务化

Lead Agent 工具风险通常是文件、bash、外部 MCP、过大输出、澄清中断等。GDP 工具风险更偏业务：

- 场景执行可能写业务数据。
- Source 保存会改变造数能力目录。
- Infra 保存会改变系统、环境、端点和数据源配置。
- SQL Source 可能涉及危险 SQL、敏感字段、生产环境数据源。
- 场景发布会让后续任务复用一个自动生成能力。
- 变量栈可能携带敏感数据，不能直接注入 Prompt。

因此 GDP 不能简单套 Lead 的工具组和 guardrail。GDP 需要的是业务动作级 policy，而不是通用 tool_name policy。

## GDP 应该怎样借鉴 Lead 工具系统

如果后续 GDP 引入类似 Lead Agent 的 middleware 和更多 LLM 节点，建议建立一套物理隔离的 GDP 工具系统，而不是直接复用 `deerflow.tools.get_available_tools()`。

### 1. 区分内部函数和模型可见工具

GDP 应该明确两类能力：

- 内部能力函数：节点确定性调用，例如搜索场景、绑定入参、保存配置、记录事件。
- 模型可见工具：LLM 可以选择调用，必须有 schema、描述、权限、幂等、审计、输出预算和审批策略。

现在的 `build_*_tools()` 可以作为模型可见工具的雏形，但不应把所有函数无差别暴露给模型。特别是 `run_datagen_scene_for_task()`、`publish_scene_from_source()`、`upsert_*_from_agent()` 这类有副作用的工具，必须包上 GDP 专用 guardrail、approval 和 idempotency。

### 2. 做 GDP 专用 `get_gdp_tools()`

建议后续新增类似入口：

```python
def get_gdp_tools(
    *,
    phase: DatagenTaskPhase,
    task_run: DatagenTaskRunResponse,
    services: GDPAgentServices,
    policy: GDPToolPolicy,
) -> list[StructuredTool]:
    ...
```

它不应该返回全量工具，而应该按阶段暴露：

- `SCENE_FULFILLMENT`：搜索场景、读取场景契约、绑定入参、申请执行确认。
- `SCENE_EXECUTING`：只暴露已确认场景执行工具。
- `SCENE_DESIGN`：搜索 Source、生成场景草稿、申请发布确认。
- `SOURCE_CONFIG`：解析/保存/测试 HTTP/SQL Source。
- `INFRA_CONFIG`：解析/保存基础配置。
- `PROGRESS_REFLECTION`：读取任务状态、反思结果、提出下一步。

这相当于 GDP 版 tool_groups，但粒度应该是业务阶段、任务状态和审批状态，而不只是配置里的 group 字符串。

### 3. 给每个 GDP 工具声明生命周期元数据

Lead 工具靠 `name/description/schema` 和 middleware 组合管理。GDP 工具建议显式增加业务元数据，例如：

```python
class GDPToolSpec(BaseModel):
    name: str
    phase: list[DatagenTaskPhase]
    sideEffectLevel: Literal["NONE", "CONFIG_WRITE", "BUSINESS_WRITE"]
    requiresApproval: bool
    idempotencyKeyFields: list[str]
    outputTarget: Literal["STATE", "TASK_STEP", "TASK_EVENT", "VARIABLE_STACK", "STORAGE_REF"]
    sensitiveOutput: bool = False
```

这些元数据可以服务于：

- 工具进入模型前过滤。
- 工具调用前审批。
- 工具调用后审计。
- 异常恢复和幂等判断。
- 输出是否允许进入 Prompt。
- 前端展示工具执行进度。

这比只靠工具描述让模型自觉遵守更可靠。

### 4. GDP 的 deferred tool 应该是“能力目录检索”，不是暴露所有场景为工具

Lead 的 deferred tool 是为 MCP 工具 schema 过多设计的。GDP 也会遇到类似问题：场景、Source、SQL、基础配置数量可能很多。

但 GDP 不应该把每个场景都变成一个独立 LLM tool。更合理的是保留当前思路：

- 用 `search_scene_contracts()` 搜索候选场景。
- 用 `get_scene_contract()` 读取具体契约。
- 用 `bind_scene_inputs()` 绑定参数。
- 通过受控执行工具运行场景。

也就是说，GDP 版 deferred 不是“把 1000 个场景 schema 都藏起来再提升”，而是“能力目录检索 + 契约读取 + 受控执行”。这更适合业务能力目录，也更容易做权限和审批。

### 5. 工具输出必须先业务落库，再给模型摘要

Lead 的 `ToolOutputBudgetMiddleware` 会把大工具输出写到 outputs 文件，再给模型预览。GDP 应该采用同一思想，但输出落点不同：

- 场景执行结果：完整结果进 SceneRun/TaskStep output，变量摘要进 visibleVariables。
- SQL 查询结果：大结果进 storageRef 或专用结果表，Prompt 只拿 schema、preview、row count。
- HTTP 响应：完整响应可落 TaskStep output，敏感字段脱敏后再进模型。
- Source/Infra 保存结果：配置主键、编码、环境、版本进入 TaskEvent，完整配置按权限展示。

GDP 的核心原则应该是：模型上下文只拿“下一步决策需要的最小摘要”，业务事实仍以数据库为准。

### 6. GDP 工具必须和中断/审批打通

Lead 用 `ask_clarification` + `ClarificationMiddleware` 把澄清变成图中断。GDP 已经有 `human_confirm` 和 `interrupt(payload)`，后续工具系统应该统一走 GDP 的中断协议。

例如高风险工具不应该直接执行：

- `run_write_scene_for_task`
- `publish_scene_from_source`
- `upsert_datasource_from_agent`
- `execute_sql_source_test` 指向生产数据源

它们应该先返回或抛出一个结构化 `GDPToolApprovalRequired`，由 `GDPInterruptMiddleware` 写入 `pendingInterrupts`，前端展示确认，恢复后再从同一幂等步骤继续执行。

### 7. GDP 子任务工具要和 Lead task 物理隔离

GDP 后续可以借鉴 `task_tool`，但不要直接把 Lead 的 `task` 当成 GDP 业务子任务工具。

建议 GDP 定义自己的子任务入口，例如：

- `delegate_gdp_scene_design_task`
- `delegate_gdp_source_config_task`
- `delegate_gdp_data_verification_task`
- `run_gdp_subtask`

这些子任务结果必须写回 `TaskStep/TaskEvent`，并携带：

- parent `taskRunId`
- parent goal
- subtask goal
- input variables
- output variables
- approval state
- recovery checkpoint
- token/cost/耗时统计

Lead 的 `task` 更适合通用探索、代码分析、研究任务。GDP 的子任务要服务造数目标，必须受 TaskRun 生命周期约束。

### 8. GDP 工具权限要融合 memory 和业务偏好

你前面提到 GDP memory 适合记录用户常用环境、系统别名、业务域偏好、字段语义映射、常用造数目标表达、审批偏好、常用 SQL 等。这些信息也应该影响工具系统，但不能直接替代工具 policy。

合理做法是：

- memory 帮助 `get_gdp_tools()` 排序或默认参数填充。
- memory 可以影响候选场景/Source 的搜索 query 和 rerank。
- memory 可以提示常用 env/sys/datasource。
- 但最终是否允许执行，仍由 task phase、approval、env policy、sideEffectLevel 决定。

也就是说，memory 是工具选择的上下文，不是工具授权的依据。

## 对 GDP 的结论

Lead Agent 的工具系统适合通用 Agent：动态工具配置、MCP/ACP 扩展、技能白名单、沙箱安全、deferred schema、subagent 委派都围绕模型可见工具集合构建。

GDP 当前更适合业务状态机：工具函数主要服务节点确定性编排，副作用直接落 TaskRun/TaskStep/TaskEvent 和业务配置表。这个方向是合理的，但后续一旦引入 LLM-heavy 节点、子 Agent 或开放更多工具给模型，就需要补一层 GDP 专用工具系统。

建议 GDP 学 Lead 的这些设计：

- 工具入口集中化，不让节点各自拼工具。
- 工具按运行时上下文动态过滤。
- 工具 schema 和描述视为模型契约。
- 工具权限不只靠 Prompt，要有运行时策略。
- 大输出不直接塞模型上下文。
- 子任务工具继承父上下文但不能越权。
- 工具调用结果必须可审计、可恢复、可归因。

但 GDP 不应该照搬 Lead 的工具集合。GDP 工具的核心不是“让模型能做更多事”，而是“让模型在造数任务生命周期内只能做当前阶段允许、可审计、可恢复、可审批的业务动作”。
