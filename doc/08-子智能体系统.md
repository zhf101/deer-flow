# 08 子智能体系统

对应字幕：`08-子智能体系统_哔哩哔哩_bilibili_BV1SVdHBEEhZ_字幕.srt`

## 本章目标

这一集讲 Lead Agent 如何通过 `task` 工具委派子任务。源码入口：

- `backend/packages/harness/deerflow/tools/builtins/task_tool.py`
- `backend/packages/harness/deerflow/subagents/config.py`
- `backend/packages/harness/deerflow/subagents/registry.py`
- `backend/packages/harness/deerflow/subagents/executor.py`
- `backend/packages/harness/deerflow/subagents/builtins/`
- `backend/packages/harness/deerflow/agents/middlewares/subagent_limit_middleware.py`

## 为什么需要子 Agent

单个 Agent 串行处理长任务有三个限制：

- 上下文窗口有限，多个大文件或多轮工具结果会快速膨胀。
- 串行执行慢，独立任务不能并行。
- 工具列表过大时，模型工具选择质量会下降。

子 Agent 的目标是隔离上下文、并发处理、按任务收窄工具。

## `task` 工具参数

源码里的工具签名：

```python
async def task_tool(
    runtime: Runtime,
    description: str,
    prompt: str,
    subagent_type: str,
    tool_call_id: Annotated[str, InjectedToolCallId],
) -> str:
```

注意 `description` 和 `prompt` 的区别：

- `description`：短描述，用于日志、前端展示和 SSE task 事件。
- `prompt`：给子 Agent 的完整任务说明，必须自包含。
- `subagent_type`：选择 `general-purpose`、`bash` 或自定义子 Agent。

子 Agent 不自动看到父 Agent 的完整对话历史，所以 `prompt` 不能写成“继续处理上面的内容”，而要包含必要背景、输入、约束和期望输出格式。

## 内置子 Agent

`general-purpose`：

- 用于复杂、多步骤、需要探索和执行的任务。
- 默认继承父 Agent 工具，但禁用 `task`、`ask_clarification`、`present_files`。
- 默认 `model="inherit"`。
- 默认 `max_turns=100`。

`bash`：

- 命令执行专用子 Agent。
- 只有 host bash 允许或容器沙箱可用时才暴露。

注册入口在 `subagents/builtins/__init__.py` 和 `registry.py`。

## 配置解析

`SubagentConfig` 字段包括：

- `name`
- `description`
- `system_prompt`
- `tools`
- `disallowed_tools`
- `skills`
- `model`
- `max_turns`
- `timeout_seconds`

`get_subagent_config()` 的解析顺序：

1. 内置子 Agent。
2. `config.yaml` 的 `custom_agents`。
3. `config.yaml` 的按 Agent 覆盖项，如 timeout、max_turns、model、skills。

内置 Agent 可以被全局默认配置覆盖，自定义 Agent 保留自己的默认值，除非显式配置覆盖。

## 执行流程

`task_tool()` 的主要步骤：

1. 读取当前 runtime 中的 `sandbox_state`、`thread_data`、`thread_id`、父模型和 metadata。
2. 根据 `subagent_type` 获取配置。
3. 如果父 Agent 有 skills allowlist，将父子白名单取交集。
4. 调用 `get_available_tools(..., subagent_enabled=False)`，防止递归创建孙 Agent。
5. 构造 `SubagentExecutor`。
6. 调用 `execute_async(prompt, task_id=tool_call_id)` 后台执行。
7. 每 5 秒轮询结果。
8. 通过 `get_stream_writer()` 发送 `task_started`、`task_running`、`task_completed` 等事件。
9. 最终把子 Agent 结果作为工具返回给 Lead Agent。

## `SubagentExecutor`

执行器做几件重要的事：

- 使用 `create_agent()` 为子 Agent 创建独立 Agent 图。
- 复用与主 Agent 类似的运行时中间件。
- 根据子 Agent 配置过滤工具。
- 加载子 Agent 自己的技能，并合并进 SystemMessage。
- 把父线程的 `sandbox` 和 `thread_data` 透传给子 Agent。
- 通过后台线程池和持久 isolated event loop 执行异步流。
- 采集子 Agent AI 消息和 token usage。

## 并发和超时

`SubagentLimitMiddleware` 控制 Lead Agent 单轮最多并发多少个子 Agent。`task_tool` 自身还有轮询超时保护：

- 后台执行 timeout 来自 `SubagentConfig.timeout_seconds`。
- 轮询间隔 5 秒。
- 轮询上限是执行超时加 60 秒 buffer。
- 超时或取消时会设置 `cancel_event`，让子 Agent 在下一次流式迭代边界协作退出。

## 常见误区

子 Agent 不是新进程里的万能隔离。它隔离的是上下文、工具可见性和执行生命周期；它仍可能共享同一线程沙箱和 thread data，这样才能读取父任务的上传文件和工作目录。

另外，子 Agent 默认不能再调用 `task`。这不是功能缺失，而是为了避免无限递归、成本爆炸和状态难以追踪。

## 本章结论

子智能体系统把长任务拆成可隔离、可并发、可限制的执行单元。Lead Agent 负责决定“要委派什么”，子 Agent 负责在收窄上下文里完成任务，结果再回到 Lead Agent 汇总。

## 结合当前源码的补充分析

上面的流程可以再往源码细节展开。DeerFlow 的 Lead Agent 子智能体协作不是简单的“模型再开一个模型”，而是围绕 `task` 工具、子 Agent 注册表、执行器、中间件链、流事件和 token 统计形成了一套受控委派机制。

### 子 Agent 能力如何暴露给 Lead Agent

Lead Agent 是否能调用子 Agent，由运行时配置 `subagent_enabled` 控制。

- `backend/packages/harness/deerflow/agents/lead_agent/agent.py` 在创建 Lead Agent 时读取 `subagent_enabled` 和 `max_concurrent_subagents`，并把它们同时传给工具加载和系统提示词。
- `backend/packages/harness/deerflow/tools/tools.py` 只有在 `subagent_enabled=True` 时才把 `SUBAGENT_TOOLS = [task_tool]` 加入 Lead Agent 工具列表。
- `backend/packages/harness/deerflow/agents/lead_agent/agent.py` 还会在 `subagent_enabled=True` 时追加 `SubagentLimitMiddleware`，用于硬限制单轮模型响应里并发 `task` tool call 的数量。

这个设计有两个重点：

1. 子 Agent 是显式能力开关，不是所有 Lead Agent 默认都可委派。
2. 并发限制不是只靠 prompt 约束，而是在模型输出工具调用后由 middleware 再做一次运行时裁剪。

另外，`backend/packages/harness/deerflow/tools/tools.py` 还会在配置了 `acp_agents` 时追加 `invoke_acp_agent` 工具。它和内置 `task` 子 Agent 属于两类协作：

- `task`：DeerFlow 进程内子 Agent，复用 DeerFlow 的工具、sandbox、thread state 和中间件。
- `invoke_acp_agent`：外部 ACP Agent，作为独立进程启动，工作目录隔离在 ACP workspace，通过 ACP 协议返回最终文本。

### `task` 工具的真实职责

`backend/packages/harness/deerflow/tools/builtins/task_tool.py` 是 Lead Agent 委派内置子 Agent 的入口。它表面上是一个 LangChain tool，但实际承担了完整的子任务生命周期编排。

`task_tool()` 参数包括：

- `description`：3-5 个词的短描述，用于日志和 stream 事件。
- `prompt`：给子 Agent 的完整任务说明，必须自包含。
- `subagent_type`：选择 `general-purpose`、`bash` 或配置里的自定义子 Agent。
- `tool_call_id`：由 LangChain 注入，源码里直接作为后台子任务 `task_id`，便于把父模型 tool call 和子任务执行关联起来。

执行流程里有几个关键点：

1. 先通过 `get_available_subagent_names()` 和 `get_subagent_config()` 解析子 Agent 配置；未知类型会返回错误并列出可用类型。
2. 如果是 `bash` 子 Agent，会再检查 host bash 或容器沙箱是否允许，避免在不安全配置下暴露命令执行能力。
3. 从父 Agent runtime 读取 `sandbox`、`thread_data`、`thread_id`、父模型名、trace_id、tool_groups、available_skills。
4. 如果父 Agent 有 skills 白名单，子 Agent 的 skills 白名单会和父白名单取交集，避免子 Agent 越权加载父 Agent 不允许的技能。
5. 重新调用 `get_available_tools(..., subagent_enabled=False)` 构造子 Agent 工具列表，明确禁止子 Agent 再看到 `task` 工具，防止递归创建孙 Agent。
6. 创建 `SubagentExecutor` 后调用 `execute_async(prompt, task_id=tool_call_id)`，子 Agent 在后台执行。
7. `task_tool()` 自己每 5 秒轮询后台结果，并通过 `get_stream_writer()` 发出 `task_started`、`task_running`、`task_completed`、`task_failed`、`task_cancelled`、`task_timed_out` 等事件。
8. 最终把子 Agent 的终态结果压成一段工具返回文本，例如 `Task Succeeded. Result: ...`，再交给 Lead Agent 汇总。

这里最值得 GDP 借鉴的是第 6-8 步：Lead LLM 不需要自己反复调用“查子任务状态”工具，后端工具层完成等待、流事件和终态收敛。这能减少模型循环、降低上下文膨胀，也让前端能看到子任务进度。

### `SubagentExecutor` 的隔离边界

`backend/packages/harness/deerflow/subagents/executor.py` 是子 Agent 执行器。它的设计不是创建一个完全隔离的新业务系统，而是在“隔离上下文”和“共享必要运行资源”之间做了平衡。

隔离的部分：

- 子 Agent 使用独立的 `create_agent()` 调用，拥有自己的消息历史。
- 子 Agent 初始消息只包含自身 `system_prompt`、加载到的 skill 内容，以及父 Agent 传入的 `prompt`。
- 工具列表会按子 Agent 的 `tools` allowlist 和 `disallowed_tools` denylist 过滤。
- 子 Agent 默认禁用 `task`、`ask_clarification`、`present_files`，避免递归、避免把澄清问题抛给子 Agent、避免子 Agent 直接展示文件。
- 子 Agent 的 `recursion_limit` 来自 `max_turns`，避免无限循环。
- 子 Agent 有自己的 timeout、取消事件、token collector 和 AI message 捕获列表。

共享的部分：

- 父 Agent 的 `sandbox` 会透传给子 Agent，让子 Agent 继续访问同一个沙箱环境。
- 父 Agent 的 `thread_data` 会透传给子 Agent，让子 Agent 知道 workspace、uploads、outputs 路径。
- 父线程的 `thread_id` 会放入子 Agent `RunnableConfig.configurable` 和 `context`，便于 sandbox 和 tracing 继续按同一线程归属。
- 父模型名可以被子 Agent 继承；如果子 Agent 配置了专用模型，则使用子 Agent 模型。

这说明 DeerFlow 的“子 Agent 隔离”不是进程级完全隔离，而是上下文、工具面、执行生命周期的隔离。它仍然共享父线程的工作空间和运行时身份，原因是子 Agent 要能处理同一个用户任务里的文件、产物和沙箱资源。

### 子 Agent 的中间件链

子 Agent 不是裸 LLM。`SubagentExecutor._create_agent()` 会调用 `build_subagent_runtime_middlewares()`，给子 Agent 配一套运行时中间件：

- `ToolOutputBudgetMiddleware`：控制工具输出预算，防止工具结果撑爆上下文。
- `ThreadDataMiddleware`：恢复或初始化线程工作目录信息。
- `SandboxMiddleware`：恢复或初始化 sandbox 状态。
- `DanglingToolCallMiddleware`：修补模型消息里缺失的 ToolMessage，避免工具调用链断裂。
- `LLMErrorHandlingMiddleware`：处理 LLM 调用异常。
- Guardrail middleware：如果配置启用，则对模型输入输出做安全控制。
- `SandboxAuditMiddleware`：审计 bash 等沙箱操作。
- `ToolErrorHandlingMiddleware`：把工具异常转换成可继续推理的 ToolMessage。
- `ViewImageMiddleware`：仅在当前模型支持视觉时追加。
- `SafetyFinishReasonMiddleware`：处理 provider 因安全原因截断 tool calls 的情况。

Lead Agent 自己还有更完整的 lead-only middleware，例如动态上下文、消息压缩、Todo、标题、记忆、工具检索、并发子 Agent 限制、循环检测、澄清拦截等。子 Agent 没有完整复制 Lead 链，而是复用“运行时安全和工具调用稳定性”这一部分，避免把 Lead 的记忆、标题、澄清、Todo 等会污染父会话语义的能力带入子任务。

这个取舍很重要：子 Agent 应该像一个可控的执行单元，而不是另一个完整对话入口。

### 并发限制为什么放在 middleware

`backend/packages/harness/deerflow/agents/middlewares/subagent_limit_middleware.py` 在 `after_model` 阶段读取最后一条 AIMessage 的 `tool_calls`，只保留前 `max_concurrent` 个 `task` 调用，多余的直接裁剪。限制值会被夹在 2 到 4 之间，默认是 3。

这样设计有几个好处：

- prompt 可以告诉模型不要开太多子 Agent，但模型不一定遵守；middleware 是确定性兜底。
- 裁剪发生在 ToolNode 执行前，超额子任务根本不会启动，避免成本和资源失控。
- 非 `task` 工具调用会保留，只裁剪超额 `task` 调用。
- 通过 `clone_ai_message_with_tool_calls()` 同步修正 provider 原始 tool call 元数据，避免 LangChain 状态里出现“可见 tool_calls 和原始 tool_calls 不一致”的问题。

GDP 后续如果支持多个子任务并行，也需要类似的硬限制，不能只依赖“请最多启动 N 个子任务”的提示词。

### 配置系统的分层

子 Agent 配置由三层叠加：

1. 内置子 Agent：`general-purpose` 和 `bash`。
2. `config.yaml` 的 `subagents.custom_agents` 自定义子 Agent。
3. `config.yaml` 的 `subagents.agents` 按 Agent 覆盖 timeout、max_turns、model、skills。

源码里还有一个细节：全局 `timeout_seconds` 和 `max_turns` 默认只覆盖内置子 Agent；自定义子 Agent 保留自己的默认值，除非显式按 Agent 覆盖。这能避免用户定义的专用 Agent 被全局默认值意外改坏。

当前源码内置默认值以代码为准：

- `general-purpose`：`model="inherit"`，`max_turns=100`，默认 timeout 来自 `SubagentConfig` 或全局配置。
- `bash`：`model="inherit"`，`max_turns=60`，工具白名单为 `bash`、`ls`、`read_file`、`write_file`、`str_replace`。

### 外部 ACP Agent 的协作边界

`backend/packages/harness/deerflow/tools/builtins/invoke_acp_agent_tool.py` 提供另一类协作：调用外部 ACP-compatible agent。

它和内置 `task` 子 Agent 的关键区别是：

- ACP Agent 作为独立进程启动，不复用 DeerFlow 的 `create_agent()`。
- 每个 DeerFlow thread 会有独立 ACP workspace：`threads/{thread_id}/acp-workspace/`。
- 工具描述明确要求 prompt 必须自包含，不能把 `/mnt/user-data` 路径直接交给 ACP Agent。
- ACP Agent 可以接入 DeerFlow 启用的 MCP servers。
- 权限请求默认拒绝，除非对应 ACP Agent 配置了 `auto_approve_permissions=true`。
- 最终只收集 ACP session update 的文本输出，作为工具结果返回给 Lead Agent。

这类能力适合调用 Codex、Claude Code 或其他外部专用 Agent，但它的状态、产物、权限模型和内置子 Agent 不一样。GDP 如果借鉴 ACP，更应该把它包装成“异步业务子任务”，而不是直接把外部 Agent 文本塞回 `messages`。

## GDP Agent 应该如何借鉴

GDP Agent 当前是一个业务状态机，不是 Lead Agent 那种通用对话型 `create_agent()`。

当前 GDP 图在 `backend/app/gdp/agent/graph.py` 中用 `StateGraph(GDPState)` 明确编排：

- `intake`
- `scene_fulfillment`
- `scene_design`
- `human_confirm`
- `source_config`
- `infra_config`
- `scene_execute`
- `progress_reflection`

当前 GDP 的强项是业务状态清晰：TaskRun 保存任务状态、阶段、目标、变量栈和中断信息；TaskStep 保存业务步骤；TaskEvent 保存审计事件；`human_confirm` 使用 LangGraph `interrupt()` 做用户恢复。这个方向是对的，不能因为要借鉴 Lead 子 Agent 就退回“所有东西都塞进聊天消息”的做法。

### 不能直接照搬 Lead `task` 的原因

Lead 的 `task` 工具适合通用聊天 Agent，但对 GDP 来说不能原样作为主机制：

1. Lead `task` 的后台任务表是进程内 `_background_tasks`，适合一次工具调用内等待完成，不是持久化业务子任务系统。GDP 的长任务、长等待、进程重启恢复需要数据库权威状态。
2. Lead `task` 最终返回字符串，GDP 更需要结构化结果，例如候选场景、Source 草稿、SQL 解析结果、基础配置缺口、变量输出、风险和下一步建议。
3. Lead `task` 事件是 `task_started/task_running/task_completed`，没有写入 GDP 的 TaskStep/TaskEvent，也没有和 `DatagenTaskPhase`、`goalStack`、`visibleVariables` 建立业务关系。
4. Lead `task` 运行在 Agent tool call 层，而 GDP 当前很多节点是确定性 service/repository 调用，不一定经过模型 tool call 层。
5. Lead 子 Agent 的恢复语义是“父工具调用还在等”；GDP 需要的是“任务中断、服务重启、用户回来后能从业务步骤继续”，这要求子任务状态可持久化、可幂等恢复。

所以 GDP 应该借鉴 Lead 子 Agent 的设计思想和局部实现，而不是把 Lead 的 `task_tool.py` 直接挂进 GDP 图。

### GDP 应建立自己的子任务模型

建议新增 GDP 专用子任务实体，例如 `DatagenTaskSubtask`，作为 TaskRun/TaskStep/TaskEvent 之外的子任务生命周期表。

建议字段：

- `taskRunId`：所属 GDP 任务。
- `subtaskId`：子任务业务 ID。
- `parentStepId`：触发该子任务的 TaskStep，可为空。
- `phase`：触发子任务时的 `DatagenTaskPhase`。
- `subagentType`：例如 `scene-discovery`、`source-designer`、`sql-analyzer`、`infra-resolver`。
- `goal`：子任务目标。
- `prompt`：实际发送给子 Agent 的自包含任务说明。
- `operationId`：幂等键，建议由 `taskRunId + phase + 子目标类型 + 输入摘要` 生成。
- `status`：`PENDING/RUNNING/COMPLETED/FAILED/CANCELLED/TIMED_OUT`。
- `inputSnapshot`：目标、约束、变量栈摘要、候选资源等输入快照。
- `resultSummary`：给主 GDP Agent 消费的短摘要。
- `resultPayload`：结构化结果，小对象可入库。
- `resultRef`：大结果、日志、文件、完整 transcript 的外部引用。
- `tokenUsage`：LLM 子任务消耗。
- `errorType/errorMessage`：失败原因。
- `startedAt/finishedAt`：生命周期时间。

TaskRun/TaskStep/TaskEvent 仍然是业务事实权威来源；Subtask 表只记录“谁被委派、跑到哪、产出了什么”。这和前面讨论的 memory 一样，不能替代业务事实。

### GDP 子 Agent 的边界应比 Lead 更业务化

Lead 的内置子 Agent 类型只有 `general-purpose` 和 `bash`，偏通用。GDP 更适合定义领域子 Agent：

- `scene-discovery-agent`：并行分析多个候选场景是否能满足当前造数目标，输出评分、缺失入参、风险。
- `source-analysis-agent`：分析已有 HTTP/SQL Source 能否支撑新场景。
- `sql-source-design-agent`：根据用户目标、库表线索、常用 SQL 和字段语义映射生成 SQL Source 草稿。
- `http-source-design-agent`：根据接口文档、历史配置和用户目标生成 HTTP Source 草稿。
- `infra-resolver-agent`：补齐系统、环境、服务端点、数据源等基础配置缺口。
- `scene-validation-agent`：验证场景定义、输入输出绑定、变量引用表达式是否闭环。
- `data-quality-reflection-agent`：对执行结果做业务目标满足度判断，输出是否继续找下一场景。

每类 GDP 子 Agent 都应该有自己的工具白名单，而不是继承父 Agent 的全量工具。例如：

- SQL 设计 Agent 只能访问 SQL Source 解析、测试、字段语义和只读 catalog 查询。
- HTTP 设计 Agent 只能访问 HTTP Source 测试、基础配置解析和接口配置保存草稿。
- Infra Agent 只能访问基础配置查询和 upsert。
- 反思 Agent 不应该有写操作工具，只能读 TaskRun、TaskStep、TaskEvent 和变量栈。

### GDP 应借鉴 Lead 的“后端等待并收敛”，但要持久化

Lead `task_tool()` 的优点是：启动子 Agent 后由后端轮询并收敛终态，模型只看到最终工具结果。GDP 可以借鉴这个机制，但等待状态必须落库。

建议 GDP 子任务执行分两种模式：

1. 短子任务：节点内部启动子任务并等待完成，完成后同步写 TaskStep/TaskEvent/Subtask，适合几十秒以内的分析。
2. 长子任务：节点只创建 Subtask 并把 TaskRun 置为 `WAITING_SUBTASK` 或类似阶段，后台 worker 执行；子任务完成后通过继续运行接口恢复 GDP 图。

长子任务不要只依赖进程内 future 或 `_background_tasks`。如果服务重启，GDP 至少要能根据 Subtask 表判断：

- 子任务已完成：读取 resultPayload/resultRef，继续下一节点。
- 子任务仍运行：继续等待或重新绑定后台 worker。
- 子任务状态未知：按 operationId 判断是否可安全重试，或标记为失败等待人工处理。

这才满足 GDP 的核心要求：异常中断后不从头开始，而是从中断步骤继续。

### GDP 需要自己的子 Agent middleware 链

建议在 `backend/app/gdp/agent/middlewares/` 下建立 GDP 物理隔离的 middleware 机制，不直接复用 Lead 的 `AgentMiddleware` 链。

推荐中间件：

- `GDPSubtaskContextMiddleware`：构造子任务上下文，强制注入原始用户目标、normalizedGoal、envCode、当前 phase、goalStack、visibleVariables 摘要、已完成步骤摘要。
- `GDPSubtaskLimitMiddleware`：限制同一 TaskRun 同时运行的子任务数量，以及单阶段最多创建多少子任务。它对应 Lead 的 `SubagentLimitMiddleware`，但约束对象应是 GDP Subtask，而不是 AIMessage tool_calls。
- `GDPSubtaskIdempotencyMiddleware`：根据 operationId 做幂等检查；已完成则直接复用结果，运行中则返回等待状态，失败则按策略重试或终止。
- `GDPSubtaskProgressEventMiddleware`：把子任务生命周期写入 TaskEvent，并向前端发 GDP 专用 stream 事件，例如 `gdp_subtask_started/running/completed/failed`。
- `GDPSubtaskResultCompactionMiddleware`：把子 Agent 完整 transcript、工具日志、大 payload 压缩为 `resultSummary + resultPayload + resultRef`，主 GDP 图只消费有界结构。
- `GDPSubtaskRecoveryMiddleware`：图恢复或服务重启后，从数据库恢复活跃子任务状态，不依赖内存任务表。
- `GDPSubtaskTokenUsageMiddleware`：记录子任务 token 消耗，方便任务成本审计。
- `GDPGoalGuardMiddleware`：子任务返回后校验结果是否仍服务于原始用户目标，避免长任务或多子任务后目标漂移。

这里可以参考甚至复制 Lead middleware 的局部源码，但应该放在 GDP 包下，并把接口改成 GDP 的 node-wrapper / graph-wrapper 形态。原因是 GDP 图不是 `create_agent(... middleware=...)`，而是 `StateGraph(GDPState)`；middleware 的切入点应该是 `before_node`、`after_node`、`on_node_error`、`before_subtask`、`after_subtask`、`after_resume`，而不是 LangChain Agent 的 `before_model/after_model/wrap_tool_call`。

### GDPState 应增加子任务相关状态，但保持轻量

GDPState 不需要继承 Lead 的 `ThreadState`，但可以增加几个生命周期清晰的字段：

- `goal_anchor`：本轮任务不可变目标锚点，来自 TaskRun.userIntent 和 normalizedGoal。
- `progress_summary`：给 LLM/子 Agent 使用的短进度摘要，由 TaskStep/TaskEvent 生成。
- `active_subtasks`：当前节点创建或等待的子任务摘要，只保存 ID、类型、状态、phase。
- `subtask_results`：刚完成、需要当前节点消费的子任务结构化结果。
- `context_refs`：大对象、完整日志、完整 transcript、文件产物引用。

这些字段都应当是数据库状态的运行时镜像，而不是唯一事实来源。checkpoint 恢复时可以从 TaskRun/Subtask/TaskStep/TaskEvent 重建它们。

### 什么时候 GDP 该用子 Agent

适合使用子 Agent 的 GDP 场景：

- 多个候选场景需要并行评估，每个候选都要分析输入缺口、风险和适配度。
- 用户目标复杂，需要先拆成多个造数子目标，例如先造用户、再造订单、再造支付流水。
- Source 缺失，需要分析接口文档、SQL、历史配置和字段语义，再生成草稿。
- 单次 SQL 测试或接口探测输出很大，需要隔离冗长日志。
- 需要调用外部代码 Agent 生成脚本、清洗样例数据或解析复杂文档。
- 反思阶段需要结合多个 TaskStep 和变量栈判断是否还要继续下一场景。

不适合使用子 Agent 的 GDP 场景：

- 单个确定性 service/repository 调用能完成的工作。
- 简单参数校验、Pydantic 校验、基础 CRUD。
- 已经有明确 sceneCode 和完整入参，只差直接执行。
- 需要立即向用户确认的审批点，应该继续走 `human_confirm` 和 `interrupt()`。

### GDP 借鉴 Lead 的最终原则

GDP 不应该追求“像 Lead 一样能开子 Agent”，而应该追求“像 Lead 一样把委派变成受控、可观察、可恢复的运行时能力”。

具体原则：

1. 子 Agent prompt 必须自包含，不能依赖父图的完整 messages。
2. 子 Agent 工具面必须收窄，默认禁止递归创建子 Agent。
3. 并发和超时必须有硬限制，不能只靠提示词。
4. 子 Agent 结果必须结构化，主 GDP 只消费摘要和引用，不吞完整 transcript。
5. 子 Agent 生命周期必须写入 GDP 业务表，不能只放内存。
6. 长子任务必须支持恢复、重试和幂等，不应该让 LangGraph 节点在内存里无界等待。
7. 子 Agent 进度要通过 GDP 专用 TaskEvent 和 stream 事件暴露给前端。
8. 每次子任务返回后都要回到原始用户目标做对齐校验，防止多轮编排后目标漂移。

如果按这个方向做，GDP 可以吸收 DeerFlow Lead 子 Agent 的设计精髓：上下文隔离、工具收窄、后端收敛、并发受控、运行可观测；同时保留 GDP 自己最重要的优势：业务状态权威、步骤可审计、异常后可从中断步骤继续。
