# 智能造数平台具体设计文档 V1

## 1. 文档目标

本文档用于将 V1 功能拆分进一步细化为可施工的设计方案。

重点说明：

- 每一类核心功能如何设计
- 关键对象如何建模
- 服务模块如何拆分
- 核心接口与流程如何落地
- 事务、快照、版本与权限如何控制

## 3. 核心对象设计

### 3.1 Task

用途：

- 探索态会话容器

设计要求：

- 不继续扩展为正式执行真相源
- 继续挂聊天消息、探索态 DAGExecution、探索态 trace

### 3.2 FlowDraft

用途：

- 探索态统一业务 DAG 根对象

建议字段：

- `id`
- `task_id`
- `status`
- `title`
- `objective`
- `business_graph_payload`
- `technical_graph_payload`
- `pending_issues_payload`
- `preflight_summary_payload`
- `input_schema_draft`
- `output_mapping_draft`
- `latest_snapshot_id`
- `created_by`
- `created_at`
- `updated_at`

### 3.3 FlowDraftSnapshot

用途：

- 保存探索态关键版本

建议字段：

- `id`
- `flow_draft_id`
- `snapshot_type`
- `business_graph_snapshot`
- `technical_graph_snapshot`
- `preflight_summary_snapshot`
- `created_by`
- `created_at`

### 3.4 Asset

#### HTTP 资产

建议字段：

- `id`
- `name`
- `description`
- `system_short`
- `base_url`
- `method`
- `path_template`
- `query_template`
- `headers_template`
- `body_template`
- `request_schema`
- `auth_type`
- `auth_config_ciphertext`
- `response_extraction_rules`
- `timeout_seconds`
- `max_response_bytes`
- `enabled`
- `owner_user_id`

#### SQL 资产

建议拆两层：

- `SqlAsset`
- `SqlAssetVersion`

其中 `SqlAssetVersion` 至少包含：

- 连接配置
- 白名单 / 黑名单
- mutation 开关
- 审核状态
- 审核信息

### 3.5 Template 与 TemplateRevision

`Template` 保存逻辑模板身份。  
`TemplateRevision` 保存可审核、可发布、可重复执行的版本内容。

`TemplateRevision` 至少承载：

- `business_graph_snapshot`
- `technical_graph`
- `input_schema`
- `output_mapping`
- `status`
- `source_run_id`

### 3.6 TemplateRevisionStep

建议字段：

- `id`
- `template_revision_id`
- `step_id`
- `step_type`
- `name`
- `depends_on`
- `design_intent`
- `resolution_rationale`
- `resolved_execution_plan`

### 3.7 Run 与 RunStep

#### Run

- `id`
- `entry_type`
- `source_task_id`
- `template_id`
- `template_revision_id`
- `initiator_user_id`
- `system_short`
- `objective`
- `input_payload`
- `resolved_input`
- `status`
- `final_output`
- `error_summary`

#### RunStep

建议字段：

- `id`
- `run_id`
- `step_id`
- `step_type`
- `step_name`
- `status`
- `depends_on`

### 3.8 AuditRecord

用途：

- 风险判定
- 用户确认
- SQL 审计
- 执行结果追溯

## 4. 视图设计

### 4.1 FlowDraftGraph

默认展示：

- `business_graph`

可切换：

- `technical_graph`

设计要求：

- 简单场景支持一对一映射
- 复杂场景支持业务节点展开到多个技术节点

### 4.2 FlowStepDesign

统一结构：

1. 资产与参数
2. 执行语句与提取规则
3. 风险与影响

### 4.3 PreflightCheckView

设计要求：

- 支持按问题类型分组
- 支持按步骤分组
- 默认按问题类型分组
- 每项问题带推荐修正路径

### 4.4 TrialExecutionDetail

设计要求：

- 整体试跑概览
- 单步执行详情
- HTTP / SQL 两类节点差异化展示

### 4.5 ToolTraceView

设计要求：

- 默认折叠
- 仅用于研发调试与高级排障

## 5. 探索态流程设计

### 5.1 初版生成

流程：

```text
聊天输入
  ↓
生成初版 FlowDraft
  ↓
展示业务意图图 + 待确认项
```

### 5.2 修正路径

#### 回聊天修正

适用于：

- 技术路线待确认
- 资产与意图不匹配
- SQL 方案明显偏离

#### 去节点详情修正

适用于：

- 参数缺失
- 输出映射不完整
- 默认值待补
- 低风险结构字段修改

### 5.3 重收敛策略

支持：

- 局部重收敛
- 全量重收敛

默认：

- 局部重收敛

建议全量重收敛的场景：

- 改了上游关键节点
- 改了资产
- 改了 SQL 主体
- 改了全局输入 schema

## 6. 预检设计

### 6.1 预检输入

- 最新 `technical_graph`
- 最新 pending flags
- 最新 resolver 输出
- governance 校验结果

### 6.2 预检输出

建议结构：

```text
PreflightResult
├── is_runnable
├── issues[]
├── grouped_by_type
├── grouped_by_step
└── suggested_actions
```

### 6.3 问题类型

- `route_pending`
- `asset_pending`
- `param_pending`
- `governance_blocked`
- `dependency_incomplete`
- `mapping_incomplete`

### 6.4 进入试跑门槛

只有满足以下条件才允许进入试跑：

- 无待确认技术路线
- HTTP / SQL 资产都已唯一确定
- 参数补齐
- SQL 风险预检通过
- 输出映射完整

## 7. Resolver / Executor 设计

### 7.1 通用契约

建议统一定义：

- `ResolverInput`
- `ResolverOutput`
- `ExecutorInput`
- `ExecutorOutput`

### 7.2 HTTPResolver

#### 输入

- `design_intent`
- `upstream_outputs`
- `user_inputs`
- `http_asset_definition`
- `template_context`
- `system_defaults`
- `history_examples`
- `history_success_mappings`

#### 输出

- `resolution_status`
- `blocking_issues`
- `resolution_rationale`
- `resolved_execution_plan`
- `editable_fields`

#### 规则

- 试跑前必须唯一确定具体 HTTP 资产
- 只允许补安全非关键默认值
- 关键参数缺失应阻塞

### 7.3 HTTPExecutor

#### 输入

- `resolved_execution_plan`
- `runtime_values`

#### 输出

- `execution_status`
- `request_snapshot`
- `response_snapshot`
- `extracted_outputs`
- `execution_metrics`
- `error_info`
- `audit_payload`

### 7.4 SQLResolver

#### 输入

- `design_intent`
- `upstream_outputs`
- `user_inputs`
- `sql_asset_definition`
- `schema_info`
- `template_context`
- `system_defaults`
- `governance_rules`
- `reference_sql_examples`
- `history_success_trials`

#### 输出

- `resolution_status`
- `blocking_issues`
- `resolution_rationale`
- `resolved_execution_plan`
- `editable_fields`

#### 规则

- 试跑前必须唯一确定具体 SQL 资产
- 设计阶段必须完成 lane / risk / confirmation 判定
- query 可以补安全默认项
- mutation 不允许用默认值补关键条件

### 7.5 SQLExecutor

#### 输入

- `resolved_execution_plan`
- `runtime_values`

#### 输出

- `execution_status`
- `sql_snapshot`
- `bound_params_snapshot`
- `result_snapshot`
- `extracted_outputs`
- `execution_metrics`
- `error_info`
- `audit_payload`

## 8. 可编辑字段设计

### 8.1 http_step

可直接编辑：

- 参数映射
- 默认值
- 提取规则

需要重收敛：

- 资产变更

### 8.2 sql_step

可直接编辑：

- 参数映射
- 输出字段
- 排序 / `limit` 等低风险结构项

需要重收敛：

- SQL 主体变更
- 资产变更
- 高风险条件结构变更

### 8.3 editable_fields 结构

建议每个字段至少包含：

- 字段名
- 可编辑方式
- 编辑控件建议
- 校验规则

## 9. RunRuntimeBridge 设计

### 9.1 职责

`RunRuntimeBridge` 至少负责：

- `task_id <-> run_id` 映射
- 从 task runtime 投影 `RunStep`
- 聚合 trace 到 run 视图
- 将执行产物挂到 `run_id`

### 9.2 V1 策略

- Task 继续承载探索态 runtime
- TraceEvent 暂时仍以 task 通道为主
- 运行态页面围绕 run 聚合视图
- 后续再评估底层双宿主化

## 10. Workspace 设计

### 10.1 TaskWorkspace

只用于：

- 探索态文件
- 聊天上传
- 试跑前中间草稿输入

### 10.2 RunWorkspace

每个 Run 独立拥有：

- input
- output
- temp

用途：

- 试跑产物隔离
- 模板正式执行产物隔离

### 10.3 文件归档

桥接层负责：

- 将探索态输入迁移或挂载到 RunWorkspace
- 将执行输出归档并与 `run_id` 关联

## 11. 权限与鉴权实现设计

### 11.1 实体

建议新增：

- `admin_system_scopes`
  - `id`
  - `user_id`
  - `system_short`
  - `created_at`

约束：

- `(user_id, system_short)` 唯一

### 11.2 执行层级

#### Middleware / Dependency

负责：

- 用户认证
- 装载用户角色
- 装载管理员 scope

#### datamakepool Service / Repository

负责：

- 对象级过滤
- 越权拦截
- 审核路由

#### API 层

只做：

- 参数解析
- service 调用
- 响应封装

## 12. 版本锁定与事务设计

### 12.1 资产版本锁定

建议以 Run 创建时为准：

- 锁定执行所需资产版本
- `RunStep` 保存 `resolved_execution_plan_snapshot`
- 必要时保存 `asset_version_snapshot_ref`

### 12.2 事务边界

以下动作必须单事务：

- `FlowDraft -> TemplateRevision + Steps + Schema + Mapping`
- SQL 资产审核通过并切换 `current_active_version_id`
- `Run + RunStep + 初始审计记录`

## 14. 施工顺序建议

建议顺序：

1. 定义 FlowDraft / Snapshot / Step schema
2. 定义 Resolver / Executor 契约
3. 建立 `admin_system_scopes`、Run、RunStep、AuditRecord
4. 落 `RunRuntimeBridge` 和 `RunWorkspace`
5. 实现 HTTP / SQL Resolver / Executor
6. 落预检与试跑
7. 打通模板草稿自动生成
8. 打通审核发布与模板执行

## 15. 结论

具体设计的核心不是页面多少，而是把以下几件事稳定下来：

- 探索态和执行态分开
- 主 DAG 与工具 DAG 分开
- 设计收敛与执行运行分开
- 可重复执行方案与运行快照分开
- 权限与审计前置进入系统设计

只要这几个点守住，后续施工就有稳定边界。
