# 07 沙箱执行系统

对应字幕：`07-沙箱执行系统_哔哩哔哩_bilibili_BV1SVdHBEEhZ_字幕.srt`

## 本章目标

这一集讲 DeerFlow 为什么需要沙箱，以及它如何控制文件和命令执行风险。

核心源码：

- `backend/packages/harness/deerflow/config/paths.py`
- `backend/packages/harness/deerflow/agents/middlewares/thread_data_middleware.py`
- `backend/packages/harness/deerflow/sandbox/middleware.py`
- `backend/packages/harness/deerflow/sandbox/tools.py`
- `backend/packages/harness/deerflow/sandbox/local/`
- `backend/packages/harness/deerflow/community/aio_sandbox/`

## 威胁模型

如果 Agent 直接操作真实文件系统，会有几类风险：

- 用户间数据泄露：一个线程读取另一个线程的上传文件。
- 系统敏感文件泄露：例如读取 `/etc/passwd`、云凭证、环境文件。
- 并发覆盖：多个线程写同一路径导致结果丢失。
- 命令执行风险：本地 `bash` 可以访问主机环境。

DeerFlow 用两个核心机制缓解：线程目录隔离和虚拟路径翻译。

## 线程目录结构

`Paths` 定义了 DeerFlow 运行时数据目录。默认基础目录来自 `DEER_FLOW_HOME` 或项目下 `.deer-flow`。

典型结构：

```text
{base_dir}/
  users/{user_id}/
    memory.json
    agents/{agent_name}/
    threads/{thread_id}/
      user-data/
        workspace/
        uploads/
        outputs/
      acp-workspace/
```

在沙箱里，线程数据统一映射为：

```text
/mnt/user-data/workspace
/mnt/user-data/uploads
/mnt/user-data/outputs
```

模型只能看到虚拟路径，不能直接看到宿主机真实路径。

## `ThreadDataMiddleware`

文件：`thread_data_middleware.py`

它在 `before_agent()` 中做几件事：

1. 从 `runtime.context.thread_id` 或 `config.configurable.thread_id` 获取线程 id。
2. 使用 `get_effective_user_id()` 得到当前用户。
3. 生成 `workspace_path`、`uploads_path`、`outputs_path`。
4. 写入 `ThreadState.thread_data`。
5. 给最后一条用户消息补充 `run_id` 和 `timestamp`。

默认 `lazy_init=True`，只计算路径，不立即创建目录。目录会在工具使用时按需创建。

## `SandboxMiddleware`

文件：`sandbox/middleware.py`

它负责沙箱生命周期：

- `lazy_init=True` 时，不在每次 Agent 调用前获取沙箱。
- `lazy_init=False` 时，在 `before_agent` / `abefore_agent` 中获取沙箱。
- `after_agent` / `aafter_agent` 释放沙箱。
- 异步路径用 `acquire_async()`，避免阻塞 ASGI 事件循环。

源码注释里说沙箱会跨同一线程多轮复用，但当前 `after_agent` 会对 state/context 中的 sandbox 执行 release。具体 provider 可实现引用计数或复用策略。

## 虚拟路径翻译

`sandbox/tools.py` 中的关键函数：

- `replace_virtual_path()`
- `_resolve_local_read_path()`
- `_resolve_and_validate_user_data_path()`
- `validate_local_tool_path()`
- `mask_local_paths_in_output()`

LocalSandbox 下，工具会把 `/mnt/user-data/...` 翻译成当前线程真实路径，并检查路径不能逃逸到其他目录。输出再反向替换为虚拟路径，避免把宿主机路径暴露给模型或用户。

## 三类可访问空间

### `/mnt/user-data`

当前线程的工作目录、上传文件和输出目录。读写权限由具体工具控制。

### `/mnt/skills`

技能目录。通常只读挂载，供 Agent 读取技能文件。`_resolve_skills_path()` 会把容器路径映射到宿主机技能目录。

### `/mnt/acp-workspace`

ACP Agent 的工作空间。`Paths.acp_workspace_dir()` 为每个线程提供独立目录，避免 ACP 调用之间互相污染。

## 沙箱 provider

### LocalSandboxProvider

直接在主机文件系统上运行。适合单用户本地开发。默认不允许 host bash。

风险：没有容器级隔离，必须依赖路径校验和工具约束。

### AioSandboxProvider

容器型沙箱，位于 `community/aio_sandbox`。适合多用户或生产隔离场景。可以基于 Docker、Apple Container 或 provisioner 管理容器/Pod。

## 文件工具的安全细节

- `read_file` 支持 `start_line/end_line`，并做最大输出截断。
- `grep` 和 `glob` 限制最大结果数。
- `bash` 在 LocalSandbox 下会校验命令中的路径，并可被 `allow_host_bash` 禁用。
- `write_file` 和 `str_replace` 使用 `get_file_operation_lock()`，避免同一路径并发写入。
- 错误信息通过 `_sanitize_error()` 屏蔽宿主机真实路径。

## 本章结论

沙箱系统的重点不是“能执行命令”，而是“在受控边界里执行命令”。DeerFlow 通过用户/线程目录、虚拟路径、路径校验、输出脱敏、host bash 开关和容器 provider，把工具执行从危险能力变成可治理能力。
