# 05 中间件链详解

对应字幕：`05-中间件链详解_哔哩哔哩_bilibili_BV1SVdHBEEhZ_字幕.srt`

## 本章目标

字幕说中间件是 DeerFlow 能力的真正来源，这个判断基本准确。LLM 调用本身只负责推理，文件、上传、记忆、标题、摘要、错误处理、安全、循环检测都通过中间件实现。

核心源码：

- `backend/packages/harness/deerflow/agents/lead_agent/agent.py`
- `backend/packages/harness/deerflow/agents/middlewares/`
- `backend/packages/harness/deerflow/sandbox/middleware.py`
- `backend/packages/harness/deerflow/guardrails/middleware.py`

## 中间件解决什么问题

如果没有中间件，Agent 主循环会变成一个巨大的过程式函数：

```text
创建线程目录 -> 注入上传文件 -> 初始化沙箱 -> 修复工具调用
-> 压缩上下文 -> 注入记忆 -> 调 LLM
-> 检查工具调用 -> 执行工具 -> 处理异常
-> 记录审计 -> 生成标题 -> 写记忆队列 -> 返回
```

这样的问题是不可测试、不可组合、不可局部关闭。中间件把这些横切关注点拆成独立类，通过 LangChain Agent middleware 钩子接入。

## 四类钩子

常见钩子可以理解为四个时机：

- `before_agent` / `after_agent`：整个 Agent 调用前后。
- `before_model` / `after_model`：每次模型调用前后。
- `wrap_tool_call` / `awrap_tool_call`：工具调用执行前后包裹。
- 其他异步版本：避免在 ASGI 事件循环里直接做阻塞 IO。

不同中间件选择不同切点。例如：

- `UploadsMiddleware.before_agent()` 改写用户消息。
- `TitleMiddleware.after_model()` 在模型第一次回复后生成标题。
- `ToolErrorHandlingMiddleware.wrap_tool_call()` 把工具异常转换成 `ToolMessage(status="error")`。
- `ClarificationMiddleware.wrap_tool_call()` 拦截 `ask_clarification` 并结束当前图执行。

## 实际链条

`lead_agent.agent._build_middlewares()` 会先调用 `build_lead_runtime_middlewares()`，基础链包含：

1. `ToolOutputBudgetMiddleware`
2. `ThreadDataMiddleware`
3. `UploadsMiddleware`
4. `SandboxMiddleware`
5. `DanglingToolCallMiddleware`
6. `LLMErrorHandlingMiddleware`
7. 可选 `GuardrailMiddleware`
8. `SandboxAuditMiddleware`
9. `ToolErrorHandlingMiddleware`

然后 Lead Agent 再追加：

10. `DynamicContextMiddleware`
11. 可选 `DeerFlowSummarizationMiddleware`
12. 可选 `TodoMiddleware`
13. 可选 `TokenUsageMiddleware`
14. `TitleMiddleware`
15. `MemoryMiddleware`
16. 条件启用 `ViewImageMiddleware`
17. 条件启用 `DeferredToolFilterMiddleware`
18. 条件启用 `SubagentLimitMiddleware`
19. 条件启用 `LoopDetectionMiddleware`
20. 自定义中间件
21. 条件启用 `SafetyFinishReasonMiddleware`
22. `ClarificationMiddleware`

具体数量会随配置变化，所以字幕里“18 个中间件”可理解为当时配置下的近似描述。源码的关键不是固定数量，而是顺序和启用条件。

## 顺序为什么重要

### `ThreadDataMiddleware` 要早

它负责写入 `thread_data`，后续上传、沙箱、文件工具都需要线程路径。

### `UploadsMiddleware` 要在模型前

上传文件信息要在 LLM 看见用户问题之前注入，否则模型不知道文件路径。

### `SandboxMiddleware` 要在沙箱工具之前

沙箱可以懒初始化，但中间件要先建立状态和 provider 约定。

### 摘要要在模型前

`DeerFlowSummarizationMiddleware` 在 `before_model` 里判断 token 是否超过触发条件，把旧消息压缩成 summary，避免后续模型调用超上下文。

### `TitleMiddleware` 在 `after_model`

标题需要用户首条消息和 AI 首次回复，必须等模型有输出后才有足够信息。

### `MemoryMiddleware` 在 Agent 结束后排队

记忆更新不应该阻塞当前响应。它读过滤后的用户消息和最终 AI 回复，加入后台防抖队列。

### `ClarificationMiddleware` 必须靠后

它拦截 `ask_clarification` 工具调用，并返回 `Command(goto=END)`。如果放太前，其他工具后处理和安全逻辑可能无法介入。

## 几个关键中间件

### `ToolErrorHandlingMiddleware`

捕获工具异常，保留 LangGraph 控制流异常 `GraphBubbleUp`，其余异常转换为错误 `ToolMessage`。这样一次工具失败不会直接让整个 run 崩溃，模型可以基于错误继续选择替代方案。

### `DeerFlowSummarizationMiddleware`

继承 LangChain `SummarizationMiddleware`，额外做了几件事：

- 摘要前触发 hook，例如记忆 flush。
- 保留动态上下文提醒，避免日期/记忆注入错位。
- 对最近读取的 skill 文件做 rescue，避免刚加载的技能被摘要掉。
- 生成 `HumanMessage(name="summary")`，前端可隐藏但模型仍能看到。

### `MemoryMiddleware`

只在 Agent 完成后把对话加入记忆队列，不直接更新文件。这样避免当前请求被记忆 LLM 调用拖慢。

### `LoopDetectionMiddleware`

检测重复工具调用循环。源码注释说明 warning 会排队到下一次模型调用，以隐藏 `HumanMessage(name="loop_warning")` 的方式追加，避免破坏工具调用与工具消息配对。

## 本章结论

中间件链是 DeerFlow 的能力装配层。理解中间件要抓三件事：它在哪个钩子执行、读写 `ThreadState` 的哪些字段、为什么必须在这个位置。只看“这个类做什么”不够，顺序才是 Agent 运行时可靠性的关键。

## 结合当前源码的逐项拆解

从源码看，Lead Agent 的 middleware 链不是“增强功能列表”，而是一个完整的运行时治理层。它解决的问题可以分成几类：

- 运行目录、上传文件、沙箱等线程运行环境。
- LLM 请求、工具调用和消息格式的稳定性。
- 上下文预算、摘要、动态上下文和记忆。
- 用户可见体验，例如标题、待办、澄清、图片注入。
- 安全、审计、循环检测、子 Agent 并发限制。

当前链路由两个函数共同组装：

- `build_lead_runtime_middlewares(app_config=..., lazy_init=True)`：构造 Lead/Subagent 共享的基础运行时链。
- `_build_middlewares(config, model_name, agent_name, app_config=...)`：追加 Lead Agent 专属能力链。

### 基础运行时链

基础链来自 `backend/packages/harness/deerflow/agents/middlewares/tool_error_handling_middleware.py`。

| 顺序 | Middleware | 主要钩子 | 解决的问题 | 关键状态/依赖 |
| --- | --- | --- | --- | --- |
| 1 | `ToolOutputBudgetMiddleware` | `wrap_tool_call` / `wrap_model_call` | 单个工具结果过大时写入 outputs 文件，只把预览和文件路径放回上下文；历史 `ToolMessage` 仍过大时兜底截断。 | 依赖 `thread_data.outputs_path`，但它在工具调用阶段已由 `ThreadDataMiddleware.before_agent()` 写入。 |
| 2 | `ThreadDataMiddleware` | `before_agent` | 根据 `thread_id/user_id` 计算或创建 workspace/uploads/outputs 路径，并给最后一条用户消息补 `run_id/timestamp`。 | 写 `thread_data`，要求 runtime context 或 configurable 中有 `thread_id`。 |
| 3 | `UploadsMiddleware` | `before_agent` | 扫描本轮上传和历史上传，把 `<uploaded_files>` 注入最后一条 HumanMessage，让模型知道文件路径、大小、文档 outline。 | 写 `uploaded_files`，依赖 thread uploads 目录和消息 `additional_kwargs.files`。 |
| 4 | `SandboxMiddleware` | `before_agent` / `after_agent` | 为线程分配沙箱环境，支持 lazy/eager 初始化，工具执行时可绑定同一个 sandbox。 | 写 `sandbox.sandbox_id`，依赖 `thread_id` 和 sandbox provider。 |
| 5 | `DanglingToolCallMiddleware` | `wrap_model_call` | 修复被取消/中断后遗留的 AI tool call，没有对应 ToolMessage 会导致下一次模型请求被 provider 拒绝。 | 扫描 `AIMessage.tool_calls/invalid_tool_calls` 和 `ToolMessage`，按因果顺序插入合成错误 ToolMessage。 |
| 6 | `LLMErrorHandlingMiddleware` | `wrap_model_call` | 对 transient/busy 错误做重试和退避；对持续失败启用 circuit breaker；认证/额度等错误转成用户可见 AIMessage。 | 依赖 `app_config.circuit_breaker`，保护模型调用而不是工具调用。 |
| 7 | `GuardrailMiddleware` | `wrap_tool_call` | 工具执行前调用外部 guardrail provider，拒绝高风险工具输入。provider 异常时按 fail-closed/fail-open 策略处理。 | 构造 `GuardrailRequest(tool_name, tool_input, agent_id, timestamp)`。 |
| 8 | `SandboxAuditMiddleware` | `wrap_tool_call` | 对 `bash` 命令做输入校验、风险分类、审计日志。高风险阻断，中风险执行但在结果里追加警告。 | 依赖 `ThreadState`、`thread_id`、bash tool 参数。 |
| 9 | `ToolErrorHandlingMiddleware` | `wrap_tool_call` | 捕获工具异常并转换为 `ToolMessage(status="error")`，让模型有机会基于错误继续，而不是整个 run 崩溃。 | 保留 `GraphBubbleUp`，避免吞掉 interrupt/pause/resume。 |

这里有一个容易误解的点：`ToolOutputBudgetMiddleware` 排在 `ThreadDataMiddleware` 前面，但它不是在 `before_agent` 阶段读取 `thread_data`，而是在工具调用返回后从 `request.runtime.state.thread_data` 取 outputs 路径。到工具调用阶段，`ThreadDataMiddleware.before_agent()` 已经执行过，所以顺序是成立的。

### Lead Agent 专属链

Lead Agent 在基础链之后追加自己的 middleware，源码在 `backend/packages/harness/deerflow/agents/lead_agent/agent.py` 的 `_build_middlewares()`。

| 顺序 | Middleware | 主要钩子 | 解决的问题 | 关键状态/依赖 |
| --- | --- | --- | --- | --- |
| 10 | `DynamicContextMiddleware` | `before_agent` | 把当前日期和可注入 memory 作为隐藏 `<system-reminder>` 注入 HumanMessage，保持系统 Prompt 静态以提高 prefix cache 命中。跨午夜时补充日期更新提醒。 | 读 memory 配置和 agent_name；写隐藏 HumanMessage，不直接改 system prompt。 |
| 11 | `DeerFlowSummarizationMiddleware` | `before_model` | 上下文超阈值时压缩旧消息；摘要前触发 hook；保留动态上下文提醒；保留最近读取的 skill 文件。 | 写 `HumanMessage(name="summary")`，用 `RemoveMessage(REMOVE_ALL_MESSAGES)` 重建消息窗口。 |
| 12 | `TodoMiddleware` | `before_model` / `after_model` / `wrap_model_call` | plan mode 下维护 todo。摘要导致 `write_todos` 消失时重新提醒；模型想提前结束但 todo 未完成时跳回 model。 | 读写 `todos`；用 thread_id/run_id 管理临时 completion reminder。 |
| 13 | `TokenUsageMiddleware` | `after_model` | 记录 token usage，给 AIMessage 追加 `token_usage_attribution`，把 `write_todos/task/present_files/ask_clarification` 归因成前端可展示的步骤类型。 | 写 AIMessage `additional_kwargs`；还能合并 subagent token usage。 |
| 14 | `TitleMiddleware` | `after_model` | 首轮用户消息和 AI 回复完成后生成 thread title；失败时用用户消息本地截断兜底。 | 写 `title`；内部 LLM 调用必须 `attach_tracing=False`。 |
| 15 | `MemoryMiddleware` | `after_agent` | Agent 结束后过滤用户输入和最终回复，异步加入 memory 更新队列；识别 correction/reinforcement。 | 不直接改 state；捕获 `user_id`，按 agent_name 做 per-agent memory。 |
| 16 | `ViewImageMiddleware` | `before_model` | `view_image` 工具完成后，把已查看图片的 base64 和 mime 信息注入下一次模型请求。 | 读 `viewed_images` 和最近的 view_image tool call；要求当前模型支持 vision。 |
| 17 | `DeferredToolFilterMiddleware` | `wrap_model_call` / `wrap_tool_call` | tool_search 开启时，从模型绑定 schema 中移除 deferred tools；如果模型直接调用未 promoted 的 deferred tool，则返回错误 ToolMessage。 | 依赖 deferred tool registry。 |
| 18 | `SubagentLimitMiddleware` | `after_model` | 模型一次性生成超过上限的 `task` tool call 时，直接截断超额调用。 | 修改最后一个 AIMessage 的 tool_calls；上限 clamp 到 2-4。 |
| 19 | `LoopDetectionMiddleware` | `after_model` / `wrap_model_call` / `after_agent` | 识别重复工具调用和过高工具频率。warning 延迟到下一次模型请求注入，hard stop 直接清空 tool_calls 迫使收尾。 | 按 thread/run 管理滑动窗口和临时 warning。 |
| 20 | 自定义 middleware | 取决于实现 | 给外部扩展点，但必须接受现有顺序约束。 | 在 `SafetyFinishReasonMiddleware` 和 `ClarificationMiddleware` 前。 |
| 21 | `SafetyFinishReasonMiddleware` | `after_model` | provider 因 safety/content_filter/refusal 停止但仍返回半截 tool_calls 时，清空 tool_calls，避免执行不完整参数。 | 写 `additional_kwargs.safety_termination`，发 stream/audit 事件。 |
| 22 | `ClarificationMiddleware` | `wrap_tool_call` | 拦截 `ask_clarification`，生成稳定 ToolMessage，并返回 `Command(goto=END)` 中断当前图。 | 必须保留 `GraphBubbleUp` 语义，澄清后等待用户继续。 |

`SafetyFinishReasonMiddleware` 的位置尤其细。源码注释说明 LangChain 对 `after_model` 是按 middleware 列表反向连边的，所以后注册的 Safety 会先看到模型原始输出。它先清理 safety 触发的半截 tool_calls，然后 Loop/Subagent 等后续 after-model 逻辑看到的是已经清洗过的 AIMessage，避免误判或继续执行危险工具。

### 每个 middleware 的设计精髓

`ToolOutputBudgetMiddleware` 的重点不是简单截断，而是把“大结果”从模型上下文迁移到 outputs 文件。这样模型仍能通过文件路径继续精确读取，同时避免一次 SQL、bash、read_file 结果撑爆上下文。

`ThreadDataMiddleware` 把线程运行目录变成 state 生命周期的一部分。后续上传、沙箱、文件输出、工具结果外置都不需要自己重复推导路径。

`UploadsMiddleware` 的设计重点是“用户上传文件必须在模型首次理解任务前可见”。它不是工具调用后的补充信息，而是用户消息的一部分。

`SandboxMiddleware` 把执行环境和 Agent state 绑定，避免工具层各自创建执行环境。沙箱生命周期和 thread/run 生命周期要分开考虑，不能让每个工具调用都创建一次沙箱。

`DanglingToolCallMiddleware` 解决的是 provider 消息协议问题。OpenAI/Moonshot 这类接口要求 `AIMessage(tool_calls)` 后面必须紧跟对应 `ToolMessage`。如果中断、取消、非法 tool call 破坏了配对关系，下一次模型调用会直接失败。

`LLMErrorHandlingMiddleware` 把模型供应商不稳定从业务逻辑中隔离出来。节点和工具不需要关心 429/5xx/busy/auth/quota 的分类。

`GuardrailMiddleware` 和 `SandboxAuditMiddleware` 都是工具安全，但层次不同。前者是通用 provider 策略，后者是 DeerFlow 对 bash 命令的本地审计和阻断。

`ToolErrorHandlingMiddleware` 的关键是把工具错误“对话化”。错误成为 ToolMessage，模型可以继续推理。只有 LangGraph 控制流异常不能吞。

`DynamicContextMiddleware` 的精髓是静态 Prompt 和动态上下文分离。系统 Prompt 保持稳定，日期、memory 这类会变的信息以隐藏 HumanMessage 注入。

`DeerFlowSummarizationMiddleware` 的重点不是“少点 token”，而是压缩时保护运行时语义：动态提醒不能被摘要错位，刚读取的 skill 不能被摘要掉，memory flush hook 要在历史被移除前有机会处理。

`TodoMiddleware` 说明 DeerFlow 不只靠 Prompt 让模型“记得计划”。它把 todo 存在 state 中，发现上下文里原始 todo 工具调用消失时重新提醒；模型想带着未完成 todo 结束时，用 `jump_to="model"` 做硬约束。

`TokenUsageMiddleware` 是观测和前端体验层。它把一条 AIMessage 的工具调用解释成“todo 更新、子 Agent 分发、文件呈现、澄清、普通工具批次”等语义，便于 RunJournal 或 UI 展示。

`TitleMiddleware` 是非核心能力，但它体现了 middleware 内部 LLM 调用的 tracing 约束：必须继承父 RunnableConfig，并且 `create_chat_model(... attach_tracing=False)`，否则会制造重复 trace。

`MemoryMiddleware` 不把 memory 更新放在主链路同步执行，而是 after_agent 入队。这样当前响应不被 memory LLM 调用拖慢，也减少 memory 写入和主任务执行之间的耦合。

`ViewImageMiddleware` 把工具产物转成下一轮模型可消费的 multimodal 输入。它只在模型支持 vision 时启用，否则注入 base64 没有意义还会浪费上下文。

`DeferredToolFilterMiddleware` 解决的是工具 schema 膨胀。工具可以注册给 ToolNode 执行，但不一定全部暴露给模型 bind_tools；模型需要先用 tool_search 发现和提升。

`SubagentLimitMiddleware` 是 prompt 约束的运行时兜底。即使 Prompt 要求最多 N 个 `task`，模型仍可能超发，middleware 直接截断超额调用。

`LoopDetectionMiddleware` 的重点是“不破坏 tool call 配对”。warning 不能在 `after_model` 直接插入，因为那时 ToolMessage 还没生成；所以它先排队，在下一次 `wrap_model_call` 时追加到消息末尾。hard stop 则直接清空最后 AIMessage 的 tool_calls，因为清空后不再需要 ToolMessage 配对。

`SafetyFinishReasonMiddleware` 解决的是 provider 返回“被安全终止但仍带半截 tool call”的异常形态。半截写文件、半截 bash、半截 JSON 参数都不能执行。

`ClarificationMiddleware` 把澄清从普通工具调用升级成图中断。它不是让模型继续执行一个返回文本的工具，而是让 run 停在明确的用户输入点。

## 为什么 Lead middleware 链没有用于 GDP 图

当前 GDP Agent 在 `backend/app/gdp/agent/graph.py` 中使用的是 `StateGraph(GDPState)`，不是 `langchain.agents.create_agent()`。这决定了 Lead Agent 的 `AgentMiddleware` 链不能直接挂上去。

### 类型和生命周期不匹配

Lead middleware 的生命周期围绕 LangChain Agent 主循环：

```text
before_agent -> before_model/wrap_model_call -> after_model
-> wrap_tool_call -> after_agent
```

GDP 图的生命周期是业务节点流转：

```text
intake -> scene_fulfillment -> scene_design/source_config/infra_config
-> human_confirm -> scene_execute -> progress_reflection
```

GDP 节点直接调用 Python service 和业务 tool 函数，返回的是 `GDPState` dict。Lead middleware 处理的是 `AIMessage`、`ToolMessage`、`tool_calls`、`ModelRequest`、`ToolCallRequest`。两者的插入点不是同一类对象。

### 状态权威来源不同

Lead Agent 的权威上下文主要在 `ThreadState.messages` 和一组 reducer state 中：

- `thread_data`
- `sandbox`
- `uploaded_files`
- `viewed_images`
- `todos`
- `artifacts`
- `title`

GDP 的权威业务状态在数据库：

- `TaskRun`：目标、阶段、状态、环境、计划、变量栈、中断、反思、完成/失败。
- `TaskStep`：每一个业务步骤的输入、输出、状态、资源选择。
- `TaskEvent`：审计事件和可回放事件流。
- `GDPState`：图运行时轻量路由状态，例如 `task_run_id/current_phase/last_tool_result/pending_confirmation`。

如果把 Lead middleware 原样挂到 GDP 上，很容易把业务事实错误地迁移到 `messages` 或 `ThreadState`，反而削弱 TaskRun/TaskStep/TaskEvent 的权威性。

### 很多 Lead middleware 依赖通用 Agent 的消息协议

这些 middleware 基本不能直接复用到 GDP 主图：

- `DanglingToolCallMiddleware`：只处理 `AIMessage.tool_calls` 和 `ToolMessage` 配对。
- `TodoMiddleware`：依赖 `write_todos` 工具调用和 `ThreadState.todos`。
- `ViewImageMiddleware`：依赖 `view_image` 工具、`viewed_images` 和 vision model。
- `DeferredToolFilterMiddleware`：依赖模型 bind_tools 和 ToolNode 工具注册。
- `SubagentLimitMiddleware`：依赖模型输出的 `task` tool_calls。
- `LoopDetectionMiddleware`：检测的是模型重复工具调用，不是 GDP 业务节点重复执行。
- `ClarificationMiddleware`：拦截 `ask_clarification` tool call；GDP 已经通过 `human_confirm` 节点和 `interrupt(payload)` 管理等待用户。
- `SafetyFinishReasonMiddleware`：处理 provider safety finish 后的半截 tool call；GDP 主图目前不是模型直接发工具调用。

这些不是“没用上”的问题，而是运行时模型不同。GDP 可以学习它们的思想，但不能把类直接塞进 `StateGraph(GDPState)`。

### 有些能力可以抽思想，不能复制状态

这些 middleware 对 GDP 有明显参考价值，但需要 GDP 版本：

- `ToolOutputBudgetMiddleware`：GDP 也可能有大 SQL 结果、场景执行结果、接口响应。GDP 应该把大结果外置到 TaskStep output 的 storageRef 或专用结果表，而不是写到 `/mnt/user-data/outputs` 后塞回 ToolMessage。
- `LLMErrorHandlingMiddleware`：如果 GDP 后续节点调用 LLM 做场景设计、字段映射、反思，可以复用错误分类和 retry/circuit breaker 思想，但作用点应是 GDP LLM client wrapper 或 GDP node middleware。
- `GuardrailMiddleware`：GDP 更需要业务 guardrail，例如生产环境写操作审批、敏感字段脱敏、危险 SQL 阻断、跨系统调用白名单，而不是只检查通用 tool_name。
- `SandboxAuditMiddleware`：GDP 不一定执行 bash，但应该有 Source/SQL/HTTP/场景执行审计。
- `DynamicContextMiddleware`：GDP 可以注入用户常用环境、系统别名、字段语义映射、常用 SQL、常用造数场景，但应注入到 GDP 的 LLM 节点上下文，而不是每个业务节点。
- `DeerFlowSummarizationMiddleware`：GDP 需要压缩的是长任务执行摘要、子任务结果和消息历史，而不是替代 TaskRun/TaskStep/TaskEvent。
- `MemoryMiddleware`：GDP memory 应记录跨任务偏好和可复用知识，不替代任务历史。
- `SubagentLimitMiddleware`：GDP 子任务也需要硬并发上限、批次限制和结果归并策略。
- `LoopDetectionMiddleware`：GDP 应检测阶段振荡、重复执行同一场景、重复要求用户补同一字段，而不只是重复 tool call hash。

## GDP 应该怎样借鉴 middleware 机制

GDP 不建议直接使用 Lead 的 `AgentMiddleware` 类型，而应该做物理隔离的 GDP middleware 链，例如放在：

```text
backend/app/gdp/agent/middlewares/
```

可以定义自己的协议：

```python
class GDPGraphMiddleware(Protocol):
    async def before_graph(self, state: GDPState, config: RunnableConfig) -> GDPState | None: ...
    async def before_node(self, node_name: str, state: GDPState, config: RunnableConfig) -> GDPState | None: ...
    async def after_node(self, node_name: str, state: GDPState, result: GDPState, config: RunnableConfig) -> GDPState | None: ...
    async def on_node_error(self, node_name: str, state: GDPState, error: Exception, config: RunnableConfig) -> GDPState | None: ...
```

注册节点时用 wrapper 包起来：

```python
workflow.add_node(
    "scene_execute",
    wrap_gdp_node("scene_execute", build_scene_execute_node(...), middlewares),
)
```

这样 GDP 能获得 middleware 的顺序治理和横切能力，但不会依赖 LangChain Agent 的消息/工具协议。

### 建议的 GDP middleware 链

GDP 的链路可以按业务图生命周期设计：

| 建议顺序 | GDP middleware | 解决的问题 | 对应借鉴 |
| --- | --- | --- | --- |
| 1 | `GDPRuntimeContextMiddleware` | 合并 `configurable/context`，解析 `thread_id/run_id/user_id/task_run_id/env_code/model_name`。 | 借鉴 `_get_runtime_config()` 和 metadata 注入。 |
| 2 | `GDPTaskRunSyncMiddleware` | 节点前加载 TaskRun，恢复 `user_intent/env_code/current_phase/user_inputs`；节点后同步 `lastCheckpointId/deerflowRunId`。 | 借鉴 `ThreadDataMiddleware` 的“运行上下文先入 state”。 |
| 3 | `GDPGoalGuardMiddleware` | 每个节点执行前确认当前任务目标、父目标、阶段没有丢失；子任务返回后校验是否仍服务主目标。 | 借鉴 Todo/Loop 的“不能偏离未完成目标”。 |
| 4 | `GDPTaskEventAuditMiddleware` | 统一记录 NODE_STARTED/NODE_FINISHED/NODE_FAILED，避免每个节点手写重复审计。 | 借鉴 `SandboxAuditMiddleware` 和 `TokenUsageMiddleware` 的观测归因。 |
| 5 | `GDPIdempotencyMiddleware` | 对场景执行、Source 保存、Infra 保存等副作用节点做幂等键检查，恢复时从中断步骤继续。 | 借鉴 tool error/budget 的副作用隔离思想。 |
| 6 | `GDPBusinessGuardrailMiddleware` | 阻断未审批写操作、生产环境危险造数、敏感字段明文注入、危险 SQL。 | 借鉴 `GuardrailMiddleware`，但 policy 输入要是业务动作。 |
| 7 | `GDPMemoryInjectionMiddleware` | 在需要 LLM 的节点注入用户常用环境、系统别名、字段映射、常用 SQL、常用配置。 | 借鉴 `DynamicContextMiddleware`，但不污染 TaskRun 权威事实。 |
| 8 | `GDPContextCompressionMiddleware` | 长任务/子任务结果多时生成“任务执行摘要”，保留目标、计划、变量栈、未完成事项和中断点。 | 借鉴 `DeerFlowSummarizationMiddleware`。 |
| 9 | `GDPSubtaskMiddleware` | 管理子 Agent/子任务创建、并发上限、等待、超时、结果归并、失败回退。 | 借鉴 `SubagentLimitMiddleware`，但结果写 TaskStep/Event。 |
| 10 | `GDPProgressLoopDetectionMiddleware` | 检测阶段振荡、重复候选确认、重复执行同一场景、重复补同一配置。 | 借鉴 `LoopDetectionMiddleware`。 |
| 11 | `GDPErrorHandlingMiddleware` | 节点异常分类，业务异常进入 WAITING_USER/FAILED，可恢复异常保留 checkpoint 后等待继续。 | 借鉴 `LLMErrorHandlingMiddleware` 和 `ToolErrorHandlingMiddleware`。 |
| 12 | `GDPInterruptMiddleware` | 统一把 WAITING_USER、pendingInterrupts、`interrupt(payload)`、用户 resume 合并成一致协议。 | 借鉴 `ClarificationMiddleware`，但以 GDP 的 human_confirm 为核心。 |

这个顺序的原则是：

- 先建立运行上下文，再执行业务节点。
- 先恢复 TaskRun，再判断是否允许执行。
- 审计和幂等包住所有副作用节点。
- memory 和压缩只服务 LLM-heavy 节点，不替代数据库业务事实。
- 子任务、循环检测、错误处理和中断恢复要保证主目标不丢。

### 当前 GDP 已经隐含了哪些 middleware 职责

现在 GDP 没有独立 middleware 链，但很多职责已经散落在节点里：

- `intake`：创建 TaskRun、绑定 DeerFlow thread、初始化目标和计划。
- `scene_fulfillment`：搜索候选场景、记录事件、判断是否等待用户、执行场景。
- `scene_design`：搜索 Source、候选确认、发布场景。
- `source_config`：提取 Source 配置、Pydantic 校验、等待用户补配置、保存 Source。
- `infra_config`：解析基础配置缺口、等待用户补充、保存系统/环境/端点/数据源。
- `human_confirm`：调用 `interrupt(payload)`，恢复后合并用户输入并迁移阶段。
- `progress_reflection`：反思场景结果，完成任务或回到场景满足阶段。

这说明 GDP 的业务链路方向是对的：TaskRun/TaskStep/TaskEvent 是权威事实，`GDPState` 只是图推进状态。但横切逻辑如果继续留在节点里，会逐渐出现这些问题：

- 每个节点都要自己记录事件，事件字段和错误分类会不一致。
- 每个节点都要自己决定 WAITING_USER，恢复协议容易分叉。
- 副作用节点的幂等和恢复逻辑容易遗漏。
- 长任务和子任务接入后，主目标保护和进度摘要会散落。
- memory、压缩、tracing、token 统计会各写各的。

所以 GDP 引入 middleware 的目的不是“像 Lead Agent 一样字段多”，而是把这些横切生命周期集中管理。

## 对 GDP 的结论

Lead Agent middleware 链没有用于 GDP，不是简单漏接，而是两者运行时模型不同。Lead 链围绕通用 Agent 的 `messages -> model -> tool_calls -> ToolMessages` 循环；GDP 围绕 `TaskRun -> 业务节点 -> TaskStep/TaskEvent -> 中断/恢复` 的业务状态机。

GDP 后续应该尽量参考 Lead Agent middleware 链的设计精髓：

- 每个横切能力独立成类。
- 每个类明确读写哪些状态。
- 链路顺序是契约，不是 append 列表。
- Prompt 约束必须有运行时硬约束兜底。
- 中断、恢复、错误、审计、压缩、记忆不能散落在节点里。

但 GDP middleware 必须物理隔离、状态隔离、生命周期隔离。可以复制 Lead middleware 的局部实现思路，比如 retry、loop detection、guardrail、上下文压缩、subtask 限流，但不应该直接复用 `AgentMiddleware` 类和 `ThreadState` 字段。GDP 的核心约束始终应该是：业务事实以 TaskRun/TaskStep/TaskEvent 为准，图状态只保存推进所需的最小运行态。
