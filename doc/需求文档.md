# 智能造数平台需求文档

## 1. 文档目标

本文档用于从现有 `spec` 文档中提炼稳定需求，形成后续架构、功能拆分和实施设计的统一输入。

本文档回答的是：

- 智能造数平台到底是什么
- 平台当前代码现状是什么
- 平台 V1 必须解决哪些核心问题
- 哪些是已确认的硬需求
- 哪些约束和边界必须在后续设计中被保留

## 2. 系统当前现状

基于当前 Xagent 代码库与既有 `spec` 文档，现状可以概括为：

- Xagent 已具备较完整的通用 Agent 平台能力
  - 用户与登录体系
  - Task 模型
  - WebSocket 过程回显
  - DAG 可视化前端
  - 工具体系与 Workspace
  - Trace 与历史恢复机制
- 统一造数平台业务域尚未真正成型
  - 缺独立资产中心
  - 缺统一业务 DAG 模型
  - 缺模板草稿、版本、审核发布链
  - 缺正式 Run 中心
  - 缺 SQL 风险确认与审计闭环

也就是说：

- Xagent 底座已有
- 统一造数平台业务内核还需单独建设

## 3. 平台定位

智能造数平台不是通用 Agent 对话产品，也不是单纯后台配置系统。

平台本质是：

```text
聊天澄清需求
  ↓
生成业务 DAG 草案
  ↓
受控试跑
  ↓
沉淀模板草稿
  ↓
审核发布
  ↓
标准化重复执行
```

平台的核心价值不是“帮用户跑一次”，而是：

- 帮用户把需求澄清清楚
- 把需求变成完整可编辑的流程
- 用真实资产验证流程
- 把成功流程沉淀成模板
- 让后续同类需求走标准化执行

## 4. V1 正式范围

V1 必须包含以下能力：

- 聊天探索入口
- 模板执行入口
- HTTP 资产中心
- SQL 资产中心
- 统一业务 DAG 草案
- 受控试跑机制
- 模板草稿自动生成
- 模板版本、审核、发布
- 正式 Run 中心
- SQL 风险确认与审计
- 后台管理台

V1 关注的是：

- 让平台真正形成“探索 -> 试跑 -> 沉淀 -> 复用”的闭环

## 5. 已确认硬需求

### 5.1 入口与流程

- 平台必须有聊天探索入口
- 平台必须有模板执行入口
- 聊天探索入口必须支持多轮澄清需求
- 聊天探索阶段先整体生成一版 `FlowDraft`
- 初版生成后优先展示业务意图图和待确认项

### 5.2 FlowDraft 与业务 DAG

- 聊天探索、试跑、模板沉淀必须共用同一种业务 DAG 模型
- 主 DAG 是业务 DAG，不是工具调用 DAG
- 平台必须区分：
  - `business_graph`
  - `technical_graph`
- 默认主图展示业务意图图
- 用户可以切换技术实现图

### 5.3 技术节点

V1 技术执行节点至少包含：

- `http_step`
- `sql_step`
- `confirm`
- `mapping`
- `start`
- `end`

规则：

- `http_step` 试跑前必须唯一确定具体 HTTP 资产
- `sql_step` 试跑前必须唯一确定具体 SQL 资产
- `sql_step` 在设计阶段必须完成：
  - query/mutation 判定
  - 风险等级判定
  - 是否需要确认
  - 命中对象分析

### 5.4 试跑与固化

- 聊天阶段允许智能收敛执行方案
- 试跑允许调用真实 HTTP / SQL 资产
- 试跑前必须完成预检与收敛检查
- 只要存在待确认项，就不能进入试跑
- 试跑成功后必须把执行方案固化
- 已发布模板按固化方案执行，不再运行时高度自由发挥

### 5.5 模板

- 模板不是 prompt 预设，而是可发布、可复用的流程对象
- 模板草稿必须支持步骤级编辑
- 模板必须有版本流
- 模板发布是正式安全门
- 已发布模板执行时不再逐次确认危险 SQL

### 5.6 权限与审核

平台 V1 固定三类角色：

- 系统管理员
- 普通管理员
- 用户

必须保留：

- `systemShort` 作为对象归属与审核路由字段
- 普通管理员可绑定多个 `systemShort`
- SQL 资产新增/修改必须审核后生效
- 模板审核只按模板主 `systemShort` 路由
- Run 查看范围受角色与归属限制
- 审计页仅系统管理员可见

### 5.7 宿主与运行时

- `Task` 承担探索态宿主
- `Run` 承担执行态宿主
- V1 不要求底层基础设施一步到位做真正双宿主
- 但必须在业务层明确区分探索态和执行态

## 6. 非硬约束

以下内容不是当前必须坚持的业务前提：

- 必须沿用旧方案里所有对象命名
- 必须把 `Task` 继续当作唯一执行中心
- 必须把当前 Xagent 的工具 DAG 直接当平台主图

也就是说：

- 方案可以重构实现
- 但不能偏离已经澄清的业务目标与权限边界

## 7. 关键用户旅程

### 7.1 聊天探索旅程

```text
用户描述目标
  ↓
系统多轮澄清
  ↓
生成初版 FlowDraft
  ↓
展示业务意图图 + 待确认项
  ↓
聊天修正 / 节点详情修正
  ↓
通过预检
  ↓
受控试跑
  ↓
成功沉淀模板草稿
```

### 7.2 模板执行旅程

```text
用户选择已发布模板
  ↓
填写结构化表单
  ↓
创建 Run
  ↓
执行流程
  ↓
查看运行详情与结果
```

## 8. 关键风险

当前设计和实现最需要重点关注的风险包括：

- Task 与 Run 的双宿主语义冲突
- 现有 Trace / WebSocket 强绑定 task_id
- 现有 Workspace 强绑定 task/workspace id
- SQL 资产审核后生效带来的版本锁定问题
- 模板草稿、Run 初始化等多表写入的一致性问题

## 9. 需求总结

一句话总结：

智能造数平台要建设的不是一个更花哨的聊天页面，
而是一套以聊天探索为入口、以业务 DAG 为核心、以试跑验证为中枢、以模板复用为目标、以权限和审计为边界的统一造数平台。
