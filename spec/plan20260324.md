# 2026-03-24 基于 Xagent 的统一造数平台整体方案

## 1. 文档目标

本文档用于把前面已经确认过的需求，
重新组织成适合 `Xagent + datamakepool` 架构的产品边界文档。

本文档回答的是：

- 平台到底是什么
- `Xagent` 作为底座后，V1 做什么、不做什么
- 双入口如何协作
- 核心对象如何分层
- 角色、`systemShort`、审核、安全门如何放置

本文档不展开：

- 详细 API
- 详细表结构
- 详细实施任务

## 2. 平台定位与 V1 范围

### 2.1 平台定位

统一造数平台是一个前后台一体的平台型产品，
不是单纯聊天 Agent，也不是单纯后台配置页。

它有两个正式入口：

- **聊天探索页**
  - 适合用户不明确目标时，通过对话澄清需求、修正流程、试跑流程
- **模板执行页**
  - 适合用户目标明确时，直接按模板表单高效发起执行

平台的核心价值不是“跑一次任务”，而是：

- 聊天澄清需求
- 生成并修正业务 DAG 草案
- 受控试跑
- 自动沉淀模板草稿
- 审核发布
- 标准化重复执行

### 2.2 V1 正式范围

V1 正式交付范围包括：

- 聊天探索入口
- 模板执行入口
- HTTP 资产中心
- SQL 资产中心
- SQL 资产审核后生效
- 模板版本与审核发布
- 独立 Run 与运行详情
- SQL 风险控制与审计
- 关系视图与影响范围提示
- 后台管理台

V1 正式能力域：

- `newapigen`
- `nlp2sqlgen`

### 2.3 V1 非范围

V1 明确不做：

- `hisgen` 正式接入
  - 只保留扩展位
- 多租户隔离
- 模板条件分支
- 模板循环执行
- 外部 Secret Manager / Vault / KMS 集成
- 资产快照冻结
- 多管理员会签

## 3. 核心对象分层

切到 `Xagent` 之后，不建议推翻前面已经确认过的对象语义，
而是做一层明确映射。

### 3.1 探索前台对象

| 对象 | 作用 |
|---|---|
| `ConversationSession(Task)` | 承载聊天过程 |
| `FlowDraft` | 聊天中逐步收敛出的业务 DAG 草案 |

### 3.2 平台业务对象

| 对象 | 作用 |
|---|---|
| `Asset` | 平台可调用的能力资源 |
| `Template` | 可复用标准流程 |
| `TemplateVersion` | 模板某一版可审核、可发布内容 |
| `Run` | 一次正式执行单 |
| `AuditRecord` | 风险确认与执行审计记录 |

### 3.3 对象关系

- 聊天探索阶段的主对象是 `ConversationSession + FlowDraft`
- 一旦进入正式执行，创建 `Run`
- 模板页执行天然创建 `Run`
- 聊天试跑成功后，自动从 `FlowDraft` 生成 `TemplateVersion` 草稿

## 4. 双入口模型

### 4.1 聊天探索入口

聊天探索入口的作用不是生成一段文本回复，
而是逐步收敛出一份可试跑的业务 DAG 草案。

聊天链路：

```text
用户描述目标
  ↓
Xagent 多轮澄清
  ↓
生成 / 修正 FlowDraft
  ↓
用户继续聊天或可视化编辑
  ↓
进入受控试跑
  ↓
创建 Run
  ↓
成功后自动生成模板草稿
```

### 4.2 模板执行入口

模板执行入口是高效率正式入口。

模板链路：

```text
选择已发布模板
  ↓
填写强结构化表单
  ↓
创建 Run
  ↓
执行
  ↓
查看运行详情与审计
```

### 4.3 两个入口的统一关系

虽然聊天页和模板页是两个入口，
但正式执行统一收口到 `Run`。

因此：

- 运行详情统一看 `Run`
- SQL 审计统一挂在 `run_id`
- 模板草稿来源统一挂在 `source_run_id`

## 5. 角色模型与 `systemShort`

### 5.1 三类角色

V1 固定为三类系统角色：

- **系统管理员**
  - 最高权限
  - 全量查看、全量修改、全量审核
  - 可看审计页
- **普通管理员**
  - 绑定一个或多个 `systemShort`
  - 在自己负责的 `systemShort` 范围内承担审核职责
  - 不能直接修改别人对象
- **用户**
  - 创建、修改自己的资产
  - 创建、修改自己的模板草稿
  - 发起聊天探索
  - 执行已发布模板

### 5.2 `systemShort` 的定位

`systemShort` 是对象归属与审核路由字段。

规则：

- SQL 资产必须带 `systemShort`
- 模板必须带主 `systemShort`
- HTTP 资产建议也带 `systemShort`
- 模板可以跨 `systemShort` 引用资产
- 模板审核只按模板主 `systemShort` 路由

`systemShort` 的作用是：

- 决定对象归属
- 决定普通管理员审核边界
- 决定普通管理员查看边界

`systemShort` **不限制普通用户使用范围**。

### 5.3 普通管理员边界

普通管理员在自己负责的 `systemShort` 范围内：

- 可查看全部资产
- 可查看全部模板
- 可查看相关 Run
- 可审核 SQL 资产
- 可审核模板发布
- 允许自审自己提交、且归属于自己负责 `systemShort` 的对象

## 6. 对象权限与可见性

### 6.1 对象权限

| 对象 | 可见范围 | 修改范围 | 审核范围 | 正式使用/执行范围 |
|---|---|---|---|---|
| HTTP 资产 | 团队可见 | 创建者 + 系统管理员 | 不走审核主线 | 团队可用 |
| SQL 资产 | 团队可见 | 创建者 + 系统管理员 | 对应 `systemShort` 普通管理员 + 系统管理员 | 仅当前生效版本可正式使用 |
| 草稿模板 / 待审核模板版本 | 团队可见 | 创建者 + 系统管理员 | 对应 `systemShort` 普通管理员 + 系统管理员 | 不可执行 |
| 已发布模板 | 团队可见 | 不可直接修改 | 新版本发布由对应 `systemShort` 的普通管理员或系统管理员审核 | 团队可执行 |
| 审计日志 | 仅系统管理员可见 | 仅系统写入 | 不适用 | 不适用 |

### 6.2 Run 查看权限

| 角色 | Run 查看范围 |
|---|---|
| 系统管理员 | 全量 Run |
| 普通管理员 | 自己负责 `systemShort` 下的相关 Run |
| 用户 | 仅自己发起的 Run |

补充规则：

- 模板创建者不能查看别人执行自己模板产生的完整 Run
- 模板页只展示聚合信息，例如最近执行时间

### 6.3 待审核与驳回规则

规则：

- 待审核 SQL 资产对所有用户可见，但不可正式使用
- 待审核 SQL 资产仅创建者可测试
- 待审核模板版本对所有用户可见，但不可执行
- 驳回对象保留原对象，创建者可继续修改并重新提交
- 驳回原因只对以下角色可见：
  - 创建者
  - 负责该 `systemShort` 的普通管理员
  - 系统管理员

### 6.4 复制规则

规则：

- 用户可以复制别人已发布模板为自己的个人草稿
- 该草稿不属于原模板版本线
- 原模板的新版本只能由原模板创建者或系统管理员发起
- 用户可以复制别人资产为自己的新资产
- 复制对象默认继承原 `systemShort`
- 创建者复制后可以修改 `systemShort`
- 审核路由按修改后的新 `systemShort` 走

对于 SQL 资产：

- 复制 SQL 资产等于新建 SQL 资产
- 复制后仍需重新审核

## 7. 风险控制与审核模型

### 7.1 聊天探索态

聊天探索态是高风险阶段。

规则：

- query SQL 命中黑名单表时，必须满足权限规则
- query / mutation 都只允许使用 SQL 资产的当前生效版本
- mutation SQL 必须强校验
- 危险 SQL 必须由当前用户自己确认
- 审计日志必须完整记录

### 7.2 模板执行态

模板发布是正式安全门。

规则：

- 模板发布前由对应 `systemShort` 的普通管理员或系统管理员审核
- 已发布模板执行时不再逐次确认危险 SQL
- 即使模板中包含危险 SQL / mutation，也允许团队成员直接执行
- 仍然保留完整审计日志
- 若引用 SQL 资产，则只使用当前生效版本

### 7.3 资产审核态

V1 中，高风险资产审核仅针对 SQL 资产。

规则：

- 所有 SQL 资产新增/修改都要审核
- HTTP 资产默认不进入审批主线

## 8. 生命周期总览

### 8.1 HTTP 资产生命周期

```text
用户创建 HTTP 资产
  ↓
资产默认团队可见可用
  ↓
创建者或系统管理员修改资产
  ↓
修改直接生效
  ↓
引用该资产的模板后续执行使用当前定义
```

### 8.2 SQL 资产生命周期

```text
用户创建 SQL 资产
  ↓
填写 systemShort
  ↓
进入待审核
  ↓
对应 systemShort 普通管理员 / 系统管理员审核
  ↓
审核通过后成为当前生效版本
  ↓
后续修改生成新的待审核版本
  ↓
审核通过前，旧生效版本继续正式使用
```

### 8.3 模板生命周期

```text
聊天成功 Run
  ↓
自动生成模板草稿
  ↓
填写 / 继承主 systemShort
  ↓
对应 systemShort 普通管理员 / 系统管理员审核
  ↓
发布 v1
  ↓
模板页按最新发布版本执行
  ↓
修改生成 v2 草稿
  ↓
再次审核
  ↓
发布 v2
```

### 8.4 Run 生命周期

```text
聊天澄清阶段
  ↓
进入正式执行
  ↓
创建 Run
  ↓
步骤执行 / 结果输出 / SQL 审计
  ↓
成功 / 失败 / 终止
```

## 9. 可见性与提示原则

由于 V1 采用：

- HTTP 资产直接生效
- SQL 资产审核后生效
- 模板始终引用资产当前生效版本

平台必须提供足够好的关系可见性与变更提示能力。

具体原则：

- 模板详情页展示最近发生变更的引用资产
- 该提示是告知性提示，不打断模板执行
- 用户可以从模板详情页跳到资产详情
- 资产详情页需要反向展示引用它的模板列表
- 列表至少包含：
  - 模板名
  - 当前发布版本
  - 最近执行时间

这些提示的作用是：

- 帮助用户理解“为什么模板没改，但执行结果可能变了”
- 帮助维护者评估改动影响范围
- 帮助排障与复盘

## 10. Xagent 与 datamakepool 的职责边界

### 10.1 Xagent 负责

- 聊天探索
- 多轮澄清
- `FlowDraft` 草案生成
- DAG 可视化与交互修正
- 试跑过程回显

### 10.2 datamakepool 负责

- 统一业务 DAG 约束
- 资产中心
- SQL 资产审核后生效
- 模板草稿、模板版本、模板发布
- 正式 Run 执行
- 风险控制
- 用户确认
- 审计
- 关系视图

## 11. 设计总结

基于前面已经确认过的需求，切到 `Xagent` 后的正确产品方案应该是：

- `Xagent` 承担探索前台
- `datamakepool` 承担平台业务内核
- `FlowDraft` 承担探索态流程对象
- `Asset / Template / Run / Audit` 承担正式平台语义
- `systemShort` 承担归属与审核路由

这份 `plan` 已经吸收了前面多轮需求沟通结论，
不需要再从头重做一遍需求澄清。
