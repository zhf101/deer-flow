# 2026-03-24 基于 Xagent 的统一造数平台正式方案

## 1. 文档目标

本文档的目标不是重新发明一套需求，而是把前面在 `deer-flow` 方案中
已经多轮确认过的需求结论，完整迁移并重构到 `Xagent` 底座上。

这份文档回答的是：

- 前面已经澄清过的需求，在切到 `Xagent` 后哪些仍然成立
- 统一造数平台为什么更适合采用 `Xagent + datamakepool` 的双层结构
- `Xagent` 在平台中到底承担什么角色
- `datamakepool` 业务内核要承接哪些平台语义
- 后续 `plan / design / task` 三份规格文档如何围绕这份总方案展开

这份文档不再重走一遍需求沟通，而是把已确认结论重新组织成适合
`Xagent` 的正式方案。

## 2. 已确认需求沉淀

### 2.1 平台定位

统一造数平台不是通用 Agent 系统，也不是单纯后台配置系统，
而是一个：

```text
对话式流程设计
      ↓
受控试跑
      ↓
模板沉淀
      ↓
标准化重复执行
```

平台的核心价值不是“执行一次任务”，而是：

- 通过聊天帮助用户澄清需求
- 把需求沉淀成完整可编辑的业务流程
- 用真实资产做受控试跑
- 把成功流程沉淀成模板
- 让后续大多数同类需求走模板页高效执行

### 2.2 双入口

平台必须同时存在两个正式入口：

- **聊天探索入口**
  - 适合用户不清楚自己要什么
  - 通过聊天逐步澄清、修正、试跑流程
- **模板执行入口**
  - 适合用户目标明确
  - 直接按模板表单高效发起执行

### 2.3 核心对象

前面已经确认的核心对象语义仍然保留，只是在 `Xagent` 中要重新映射：

- `Asset`
  - 平台可调用的能力资源
  - V1 包含 HTTP 资产、SQL 资产
- `Template`
  - 可复用的标准流程
  - 有草稿、待审核、已发布、历史版本
- `Run`
  - 一次正式执行单
  - 承载步骤、状态、结果、审计
- 聊天探索态还需要一个更适合 `Xagent` 的前台对象：
  - `FlowDraft`

### 2.4 权限与审核

前面已确认的权限与审核模型如下：

- 角色固定为三类：
  - 系统管理员
  - 普通管理员
  - 用户
- 引入 `systemShort`
  - 用于对象归属与审核路由
  - 不限制普通用户使用范围
- SQL 资产属于高风险资产
  - 新增/修改都要审核
  - 待审核状态可见但不可正式使用
  - 创建者可测试
  - 审核通过后才成为当前生效版本
- 模板带主 `systemShort`
  - 可跨系统引用资产
  - 但审核只按模板主 `systemShort` 路由
- 普通管理员：
  - 绑定一个或多个 `systemShort`
  - 负责该范围下的 SQL 资产审核和模板发布审核
  - 允许自审自己提交的对象
  - 不能直接改别人对象
- 用户：
  - 创建、修改自己的资产
  - 创建、修改自己的模板草稿
  - 执行已发布模板
  - 只能看自己发起的 Run

### 2.5 风险控制

前面已确认的风险控制模型：

- 聊天探索态：
  - query SQL 受黑名单/权限组约束
  - mutation SQL 要强校验 + 用户确认 + 审计
- 模板执行态：
  - 模板发布是正式安全门
  - 已发布模板执行时不再逐次确认危险 SQL
  - 但必须保留完整审计

### 2.6 模板与资产关系

前面已确认的边界：

- **资产修改**
  - 改的是某个能力资源本身定义
- **模板修改**
  - 改的是模板引用关系或编排关系
- 资产修改可能影响模板后续执行结果
  - 但不等于模板修改

资产生效语义需要分开理解：

- HTTP 资产修改后可直接生效
- SQL 资产修改后先形成待审核新版本
  - 审核通过前由旧生效版本继续承担正式执行

## 3. 为什么采用 Xagent 作为底座

### 3.1 不是因为聊天更强，而是因为平台壳更近

从现有代码结构看，`Xagent` 已经具备：

- 用户体系
- Web API
- 任务对象
- 数据库迁移
- WebSocket 与过程回传
- DAG 可视化前端

这些能力让它更接近“平台前后台骨架”。

而当前统一造数平台最重的工作量，不在纯 Agent runtime，
而在：

- 角色与审核
- `systemShort`
- 资产与模板中心
- Run 查看权限
- 管理台
- 图形化探索前台

这些能力和 `Xagent` 的现有平台骨架天然更接近。

### 3.2 Xagent 的优势

`Xagent` 最适合承担的，是探索前台这部分：

- 聊天多轮澄清
- 流程设计过程可视化
- DAG 草案展示
- WebSocket 过程回传
- 任务进度与图结构展示

### 3.3 Xagent 的边界

但 `Xagent` 不应该直接等同于平台业务内核。

它不应该成为以下能力的权威承载者：

- 资产中心定义
- SQL 资产审核后生效
- 模板版本与审核发布
- 审计日志权威落库
- 平台正式执行语义

这些应该下沉到独立的 `datamakepool` 业务域。

## 4. 最终推荐架构

### 4.1 总体结构

最终推荐采用：

```text
Xagent 作为探索前台主骨架
+ datamakepool 作为统一造数平台业务内核
```

其中：

- `Xagent`
  - 负责探索态、交互态、展示态
- `datamakepool`
  - 负责平台态、执行态、治理态、模板态

### 4.2 统一的对象分层

我建议不要在切到 `Xagent` 后推翻前面已经确认的对象语义，
而是做一层映射：

| 层次 | 对象 | 作用 |
|---|---|---|
| 探索前台 | `ConversationSession(Task)` | 承载聊天过程 |
| 探索前台 | `FlowDraft` | 聊天中逐步收敛出来的业务 DAG 草案 |
| 平台业务域 | `Asset` | 提供能力资源 |
| 平台业务域 | `Template / TemplateVersion` | 承载流程沉淀与发布 |
| 平台业务域 | `Run` | 承载正式执行单 |
| 平台业务域 | `AuditRecord` | 承载审计与风险确认 |

### 4.3 一个关键判断

这意味着：

- `Xagent Task` 不应该直接等于平台最终 `Run`
- 它更适合作为探索会话容器
- 真正的正式执行仍然应该落到 `datamakepool.Run`

这样能同时保住：

- `Xagent` 擅长的探索前台
- 平台真正需要的审核、版本、审计、权限模型

## 5. Xagent 需要承接的具体职责

`Xagent` 最适合承接：

- 多轮澄清需求
- 输出结构化澄清结果
- 生成业务 DAG 草案
- 把 DAG 草案可视化展示
- 支持用户继续通过聊天修 DAG
- 支持用户在界面上微调 DAG
- 展示试跑过程中的状态、错误、确认提示和结果

可以把 `Xagent` 理解为：

- 对话式流程设计前台
- 业务 DAG 设计与修正前台
- 试跑过程控制台

## 6. datamakepool 需要承接的具体职责

`datamakepool` 业务域负责：

- 统一业务 DAG schema
- HTTP / SQL 资产中心
- SQL 资产审核后生效
- 模板草稿与模板版本
- 模板审核与发布
- 正式 Run 执行
- SQL 风险控制
- 用户确认
- 审计日志
- 关系视图与影响范围查询

一句话：

> `Xagent` 负责把流程“想清楚、展示清楚、试跑清楚”，  
> `datamakepool` 负责把流程“管得住、审得过、跑得稳、可追溯”。

## 7. 统一业务 DAG 仍然是核心

不管换不换底座，统一造数平台都必须坚持一条原则：

- 聊天探索看到的是业务 DAG 草案
- 试跑执行的是同一份业务 DAG 草案
- 成功后沉淀成模板草稿的还是同一份业务 DAG

不能出现：

- 聊天一套流程表示
- 试跑一套执行表示
- 模板又一套发布表示

这也是为什么即便切到 `Xagent`，仍然必须保留
`FlowDraft -> TemplateVersion -> Run` 这条清晰链路。

### 7.1 业务 DAG 不等于工具调用 DAG

这里需要明确一个关键判断：

- `Xagent` 当前默认暴露出来的 DAG，更接近“步骤内部用了哪些工具”的执行计划视图
- 这不应该直接成为统一造数平台对用户展示的主 DAG

统一造数平台的主 DAG 必须表达：

- 业务流程如何被组织
- 每一步是什么业务动作或技术执行单元
- 依赖关系是什么
- 关键输入输出是什么
- 哪些步骤有确认或风险要求

而不是直接表达：

- 内部 agent 用了哪些工具
- 内部推理链具体分了几轮

因此：

- 工具调用 DAG 只能作为调试 trace 视图
- 平台主 DAG 必须是 `FlowDraft` 驱动的业务 DAG

## 8. 五层视图与分层 DAG

### 8.1 统一采用五层视图

基于当前需求，平台前台不应只有“一张 DAG 图”，而应明确区分五层视图：

1. `FlowDraftGraph`
   - 聊天探索页主图
   - 用于展示整体流程
2. `FlowStepDesign`
   - 节点设计详情
   - 用于展示某一步被设计成如何执行
3. `PreflightCheckView`
   - 试跑前收敛检查
   - 用于集中展示阻塞试跑的问题
4. `TrialExecutionDetail`
   - 试跑 / 正式执行详情
   - 用于展示实际发生了什么
5. `ToolTraceView`
   - 调试视图
   - 用于展示步骤内部 agent / tool 的调用细节

### 8.2 `FlowDraftGraph` 默认展示业务意图图

聊天探索页的默认主图应为业务意图图。

它主要展示：

- 流程步骤
- 依赖关系
- 步骤类型
- 关键输入 / 输出摘要
- 风险或确认标记

它不直接展示：

- 完整 HTTP 请求体
- 完整 SQL 文本
- 完整响应报文
- 内部工具调用链

这层的目标是让用户快速看懂：

> 这个造数流程准备怎么做

### 8.3 主图允许切换到技术实现图

除了业务意图图，用户还应能切换到技术实现图。

技术实现图的节点粒度是“执行单元”，而不是工具步骤。

也就是说，这张图里的节点应是：

- 某个 HTTP 执行步骤
- 某个 SQL 执行步骤
- 某个确认节点
- 某个结果映射节点

而不是：

- 选资产
- 组参数
- 查 schema
- 生成 SQL
- 提取字段

这些更细粒度动作只属于步骤内部智能过程，不属于主图节点。

### 8.4 业务意图图与技术实现图的关系

平台采用分层 DAG：

- 默认主图是业务意图图
- 用户可切换到技术实现图

并且：

- 简单场景下，一个业务节点可以对应一个技术节点
- 复杂场景下，一个业务节点可以展开成多个技术子步骤

因此，平台需要支持：

- 一对一映射
- 一对多展开

但不要求在默认主图中直接暴露全部技术细节。

### 8.5 `ToolTraceView` 只作为下钻调试视图

`ToolTraceView` 的定位必须明确：

- 它不是主图
- 它不是模板发布审核图
- 它不是业务用户的默认视图

它只用于：

- 开发调试
- 高级排障
- 管理员/研发查看步骤内部 agent 行为

## 9. 技术执行节点与步骤设计

### 9.1 技术执行节点先不绑定具体执行器命名

当前设计不要求一开始就把执行器固定命名为 `newapigen` 或 `nlp2sqlgen`。

真正需要先固定的是：

- 节点语义
- 节点边界
- 步骤设计结构
- 试跑阶段与发布阶段的智能边界

执行器后续可以继续澄清命名和拆分方式，
但节点设计必须先稳定。

### 9.2 V1 技术执行节点类型

V1 技术执行节点至少拆成两类：

- `http_step`
- `sql_step`

同时保留控制节点：

- `confirm`
- `mapping`
- `start`
- `end`

### 9.3 `FlowStepDesign` 的展示优先级

对 `http_step` 与 `sql_step` 来说，节点设计详情层应按以下优先级组织信息：

1. 资产与参数
2. 执行语句与提取规则
3. 风险与影响

也就是说，用户在看某一步时，最先应该看见：

- 用哪个资产
- 需要哪些参数
- 参数从哪里来

然后才是：

- HTTP 请求细节 / SQL 方案
- 返回结果如何提取

最后才是：

- 风险级别
- 影响范围
- 是否需要确认

### 9.4 每个技术节点都必须保留两层信息

每个技术执行节点都应同时保留两类信息：

#### `design_intent`

表达：

- 这一步想完成什么
- 为什么需要这一步
- 依赖什么输入
- 期望什么输出

#### `resolved_execution_plan`

表达：

- 最终选定的资产
- 最终参数模板
- 最终 SQL / HTTP 执行方案
- 最终提取规则
- 最终输出映射
- 风险与确认策略

这样才能同时满足：

- 聊天探索时看得懂设计意图
- 模板草稿里看得见最终固化方案
- 生产回溯时能比较“原始设计意图”和“最终执行方案”

### 9.5 `http_step` 的规则

`http_step` 的规则应明确为：

- 试跑前必须唯一确定具体 HTTP 资产
- 不允许试跑时在多个 HTTP 资产中动态选择
- 不走 SQL 风险判断主链
- 智能能力主要负责：
  - 参数组装
  - 请求渲染
  - 响应提取
  - 输出映射

也就是说：

- HTTP 节点的“资产选择”是试跑前收敛问题
- HTTP 节点的“参数与提取智能”是试跑阶段能力

### 9.6 `sql_step` 的规则

`sql_step` 的规则应明确为：

- 试跑前必须唯一确定具体 SQL 资产
- 节点类型不再继续拆成 `query_step` 和 `mutation_step`
- 但在设计阶段必须提前完成：
  - `query / mutation` 判定
  - 风险等级判定
  - 是否需要确认
  - 命中对象分析
  - 治理检查结果

也就是说：

- `sql_step` 虽然只有一种节点类型
- 但不能等到真正执行时才第一次发现它是高风险 SQL

建议在 `resolved_execution_plan` 中至少保留：

- `sql_lane`
- `risk_level`
- `requires_confirmation`
- `target_tables`
- `governance_check_result`

### 9.7 试跑阶段智能、模板发布后固化

必须明确：

- 试跑阶段允许执行器使用智能能力帮助把步骤跑通
- 一旦试跑成功，最终具体方案必须回写并固化进 `FlowStepDesign`
- 已发布模板执行时按固化方案执行

也就是说，智能能力主要服务于：

- 澄清需求
- 设计流程
- 跑通试跑
- 自动产出可发布草稿

而不是让生产模板在运行时继续高度自由发挥。

## 10. 试跑前收敛检查

### 10.1 探索态 `FlowDraft` 与可试跑 `FlowDraft`

聊天探索阶段允许存在不完整 DAG。

因此必须区分：

- 探索态 `FlowDraft`
- 可试跑 `FlowDraft`

探索态允许存在：

- 技术路线待确认
- 资产待确认
- 参数待补齐
- 治理待确认

但一旦要进入试跑，就必须先完成一次收敛检查。

### 10.2 必须新增 `PreflightCheckView`

在聊天探索页里，`FlowDraft` 从生成到试跑之间，
必须有独立的“收敛检查 / 预检结果”层。

它的职责是：

- 判断 DAG 是否已经从探索态进入可试跑态
- 集中展示阻塞项
- 引导用户回聊天修正或在节点详情里修正

### 10.3 阻塞试跑的典型问题

预检至少需要覆盖：

- `route_pending`
  - 技术路线待确认
- `asset_pending`
  - 资产未确定
- `param_pending`
  - 参数未补齐
- `governance_blocked`
  - SQL 风险未通过
- `dependency_incomplete`
  - 必要依赖不完整
- `mapping_incomplete`
  - 输出映射不完整

### 10.4 试跑门槛

只要还有待确认项，`FlowDraft` 就不能进入试跑。

也就是说：

- 探索态 DAG 可以不完整
- 可试跑 DAG 必须完整、唯一、可校验

这条规则对于平台治理是强约束，不应弱化成运行时兜底逻辑。

### 10.5 缺少资产时的处理方式

如果聊天探索过程中系统发现缺少合适资产，正确处理方式应为：

- 创建“资产待补全项”
- 在补全并确认具体资产前，不允许进入试跑

不允许：

- 临时绕过资产中心直接直连试跑
- 运行时在多个候选资产之间自由选一个再说

## 11. 探索态交互与重收敛

### 11.1 先整体生成，再逐步修正

聊天探索阶段不采用“边聊边长出一小块 DAG”的模式，
而采用：

- 先整体生成一版初始 `FlowDraft`
- 再围绕这版草案逐步修正

这样做的好处是：

- 用户能先看见全局流程轮廓
- 系统能集中暴露待确认项
- 后续修正有稳定参照物

### 11.2 初版 `FlowDraft` 的默认展示

初版 `FlowDraft` 生成后，系统最先应向用户展示：

- 业务意图图
- 待确认项清单

不应默认先把用户带入：

- 纯文字摘要确认流
- 技术实现图
- 工具级 trace 视图

### 11.3 待确认项清单的组织方式

待确认项清单应同时支持：

- 按问题类型分组
- 按步骤分组

默认先按问题类型分组。

这样用户可以先快速理解：

- 当前阻塞试跑的主要问题有哪些

然后再按步骤下钻处理。

### 11.4 待确认项的默认修正路径

修正路径两条都必须支持：

- 回聊天修正
- 去节点详情面板修正

系统应按问题类型推荐默认路径：

- 需要重新思考方案的问题
  - 默认走聊天修正
- 结构化配置补全的问题
  - 默认走节点详情修正

前者典型包括：

- 技术路线待确认
- 资产与意图不匹配
- SQL 方案明显偏离意图

后者典型包括：

- 参数缺失
- 输出映射不完整
- 默认值待补
- 低风险结构字段调整

### 11.5 局部重收敛与全量重收敛

系统需要同时支持：

- 局部重收敛
- 全量重收敛

默认优先采用：

- 受影响步骤或子图的局部重收敛

但在以下场景中，应建议全量重收敛：

- 修改了上游关键节点
- 修改了资产
- 修改了 SQL 主体
- 修改了全局输入 schema
- 其他明显会影响全局语义的结构项

### 11.6 `FlowDraft` 关键版本快照

探索态不只要保存当前最新草案，
还应保留关键版本快照。

关键快照点包括：

- 初版 `FlowDraft` 生成后
- 每次完成一轮重收敛后
- 进入试跑前
- 试跑成功后

其中：

- “每次重收敛后”应支持节流和合并
- 不要求把每个细碎改动都永久固化成独立版本

### 11.7 版本 diff

`FlowDraft` 版本快照在前台应支持直接比较。

diff 需要同时支持两层：

- 业务意图层差异
- 技术实现层差异

默认先展示：

- 业务意图层差异

再支持下钻：

- 技术实现层差异

## 12. Resolver / Executor 契约总览

### 12.1 两类技术节点明确拆开

V1 技术节点明确拆成两类：

- `http_step`
- `sql_step`

并且分别拥有独立的：

- Resolver
- Executor

不采用一个模糊的统一策略层去兼容两类完全不同的技术语义。

### 12.2 设计层与执行层必须分离

对 `http_step` 和 `sql_step` 都必须坚持：

- Resolver 负责把步骤从设计意图收敛成可执行方案
- Executor 只按已收敛方案执行

因此执行层不再读取：

- `design_intent`

Executor 输入只能依赖：

- `resolved_execution_plan`
- `runtime_values`

### 12.3 每个技术节点的三层结构

每个技术节点最终至少应保留三层内容：

#### `design_intent`

表达：

- 这一步想完成什么
- 为什么需要这一步
- 依赖什么输入
- 期望什么输出

#### `resolution_rationale`

表达：

- 方案为何这样收敛
- 依据了哪些输入、样例、schema、规则、默认值

它面向两类人：

- 模板审核人
- 研发 / 排障人员

但对审核人默认展示为：

- 结构化依据级

而不是自由文本长推理链。

#### `resolved_execution_plan`

表达：

- 最终可执行且可固化的具体方案

### 12.4 Resolver 的统一输出形态

不论 HTTP 还是 SQL，Resolver 都应返回统一形态的收敛结果对象，
至少包含：

- `resolution_status`
- `blocking_issues`
- `resolution_rationale`
- `resolved_execution_plan`
- `editable_fields`

### 12.5 `HTTP Resolver` 的输入来源

`HTTP Resolver` 最少应读取：

- `design_intent`
- 上游步骤输出
- 用户输入
- 已选 HTTP 资产定义
- 模板上下文
- 系统默认值
- 历史试跑样例
- 历史成功映射

### 12.6 `HTTP resolved_execution_plan` 的冻结粒度

HTTP 节点在试跑成功后，至少冻结到：

- 已选 HTTP 资产
- 参数模板
- 参数来源标注
- 默认值与默认值来源
- 请求渲染结果结构
- 提取规则
- 输出映射
- 运行配置

并且：

- 试跑前必须唯一确定具体 HTTP 资产
- 试跑时不允许在多个 HTTP 资产之间动态选择

### 12.7 `HTTP Resolver` 的阻塞项与可编辑字段

HTTP 节点的典型阻塞项至少包括：

- 资产缺失
- 关键参数缺失
- 提取规则不完整
- 参数来源冲突
- 输出映射冲突
- 资产与设计意图不匹配
- 历史样例不足，无法可靠收敛

`editable_fields` 不能只是字段名列表，
而应至少包含：

- 字段名
- 可编辑方式
- 编辑控件建议
- 校验规则

### 12.8 `HTTP Executor` 的输入与输出

`HTTP Executor` 输入仅包含：

- `resolved_execution_plan`
- `runtime_values`

其中 `runtime_values` 主要来自：

- 用户表单输入
- 上游步骤输出
- 系统默认运行参数
- 人工确认结果
- 运行时上下文信息

`HTTP Executor` 输出至少包含：

- `execution_status`
- `request_snapshot`
- `response_snapshot`
- `extracted_outputs`
- `execution_metrics`
- `error_info`
- `audit_payload`

### 12.9 `SQL Resolver` 的输入来源

`SQL Resolver` 最少应读取：

- `design_intent`
- 上游步骤输出
- 用户输入
- 已选 SQL 资产定义
- schema 信息
- 模板上下文
- 系统默认值
- 治理规则
- 历史参考 SQL
- 历史成功试跑样例

### 12.10 `SQL resolved_execution_plan` 的冻结粒度

SQL 节点在试跑成功后，至少冻结到：

- 已选 SQL 资产
- SQL 模板 / 查询方案
- 参数模板
- 输出字段
- `sql_lane`
- 风险判断结果
- 命中对象
- 确认策略
- 参考依据绑定
- 允许变动边界
- 执行配置

其中允许变动边界同时约束：

- SQL 文本可变范围
- 结构性边界

但以结构性边界为主：

- 表范围
- 字段范围
- 参数范围
- 操作类型
- 风险等级边界

并且：

- 试跑前必须唯一确定具体 SQL 资产
- 虽然节点不再细拆成 `query_step` / `mutation_step`
- 但必须在设计阶段提前完成：
  - lane 判定
  - 风险判定
  - 确认要求判定

### 12.11 `SQL Resolver` 的阻塞项与可编辑字段

SQL 节点的典型阻塞项至少包括：

- schema 信息不足
- 关键条件缺失
- `lane` 无法判定
- 风险未通过
- 命中对象不允许
- 参考 SQL 不足
- 字段映射冲突
- 输出不可验证
- 设计意图与 SQL 资产不匹配

SQL 节点更适合直接编辑的字段包括：

- 参数映射
- 输出字段
- 排序 / `limit` 等低风险结构项

但以下改动应触发重新收敛：

- SQL 主体变更
- SQL 资产变更
- 高风险条件结构变更

### 12.12 `SQL Executor` 的输入与输出

`SQL Executor` 输入仅包含：

- `resolved_execution_plan`
- `runtime_values`

其中 `runtime_values` 主要来自：

- 用户表单输入
- 上游步骤输出
- 系统默认运行参数
- 人工确认结果
- 运行时上下文信息

`SQL Executor` 输出至少包含：

- `execution_status`
- `sql_snapshot`
- `bound_params_snapshot`
- `result_snapshot`
- `extracted_outputs`
- `execution_metrics`
- `error_info`
- `audit_payload`

## 14. 推荐的第一阶段建设顺序

2. 再完成 `datamakepool` 的资产、模板、Run、审核基础模型

## 15. 最终结论

从当前已经确认的需求出发，统一造数平台最合理的技术落点不是：

- 继续沿旧 `deer-flow` 思路硬拉整个平台能力

也不是：

- 把平台完整语义直接塞进 `Xagent` 现有对象模型

而是：

- 用 `Xagent` 承担探索前台
- 用 `datamakepool` 承担平台业务内核
- 用统一的 `FlowDraft` 业务 DAG 模型打通聊天、试跑、模板三条主链
- 用五层视图把“设计态、预检态、运行态、调试态”分开
- 用 `Asset / Template / Run / Audit` 承担正式平台治理语义

这份方案已经吸收了前面多轮澄清结论，
后续不需要再从头重做一轮需求确认。
