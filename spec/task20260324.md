# 2026-03-24 基于 Xagent 的统一造数平台实施任务拆解

## 1. 当前代码现状

基于 `Xagent` 当前仓库结构，现状可以概括为：

- 已有用户体系、Web API、数据库迁移、任务模型、DAG 可视化前端
- 还没有统一造数平台的业务内核
- 当前最接近的现有能力有两类：
  - `Task / DAGExecution / TraceEvent`
  - `text2sql` 用户数据库配置

### 1.1 已有可复用能力

| 模块 | 当前状态 | 后续用途 |
|---|---|---|
| `Task / DAGExecution / TraceEvent` | 已有 | 可承接探索会话容器、过程回显、前端任务展示 |
| 用户体系与登录 | 已有 | 可承接系统管理员 / 普通管理员 / 用户三类角色扩展 |
| Alembic 迁移 | 已有 | 适合落地平台新表 |
| Web API 与 DB Session | 已有 | 可承接 datamakepool 路由 |
| DAG 前端与 WebSocket | 已有 | 可承接 FlowDraft 可视化与试跑过程展示 |

### 1.2 当前明显缺失

| 模块 | 当前状态 | 影响 |
|---|---|---|
| `systemShort` 审核路由 | 缺失 | 无法实现分域普通管理员审核 |
| HTTP 资产中心 | 缺失 | 无法复用外部 HTTP 能力 |
| SQL 资产审核后生效 | 缺失 | 当前 text2sql 只是用户配置，不是平台资产 |
| 模板版本与审核流 | 缺失 | 现有模板能力不是平台模板中心 |
| `FlowDraft` | 缺失 | 聊天探索没有统一业务 DAG 草案模型 |
| 独立 Run 中心 | 缺失 | 无法统一运行详情、模板执行和审计 |
| 审计页 | 缺失 | 无法做平台级 SQL 审计治理 |

## 2. 目标差距

当前 xagent 已经有平台壳，但还没有统一造数业务内核。

### 2.1 目标状态

V1 目标平台需要同时具备：

- 聊天探索入口
- `FlowDraft` 业务 DAG 草案
- 模板执行入口
- HTTP 资产中心
- SQL 资产中心
- SQL 资产审核后生效
- 模板版本与审核发布
- 独立 Run 中心
- SQL 风险确认与审计
- `systemShort` 分域审核与查看

### 2.2 差距判断

| 层次 | 当前完成度 | 说明 |
|---|---:|---|
| Xagent 平台壳 | 65% | 用户、任务、Web、DAG UI 已有 |
| 统一造数业务域 | 10% | 基本未开始 |
| 权限与审核模型 | 10% | 只有 `is_admin`，没有分域审核 |
| V1 整体完成度 | 30% | 比 deer-flow 更接近平台，但业务内核仍缺失 |

## 5. 详细实施任务

### 5.1 P0 Xagent 语义对齐与脚手架

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P0-1 | 建立 `core/datamakepool` 骨架 | 在 Xagent core 下新增统一造数业务域目录 | 不把平台业务逻辑散落在现有 `web/api` 和 `core/agent` 中 |
| P0-2 | 建立 `web/models` 平台新模型 | 增加 `dm_*` 模型文件，避免和现有通用模板/text2sql 冲突 | 新业务模型统一加前缀，减少歧义 |
| P0-3 | Task 语义对齐 | 明确 `Task` 在平台里承担 `ConversationSession` | 避免把 Task 直接混成正式 Run |
| P0-4 | `FlowDraft` 初始契约 | 定义统一业务 DAG 草案 schema | 让聊天、试跑、模板草稿共用一套流程模型 |

### 5.2 P1 角色、`systemShort`、审核基础能力

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P1-1 | 角色模型扩展 | 在 `User.is_admin` 基础上引入普通管理员语义 | 系统管理员仍用 `is_admin`，普通管理员通过新表表达 |
| P1-2 | `admin_system_scopes` | 建立普通管理员与多个 `systemShort` 的映射表 | 支撑审核路由与查看边界 |
| P1-3 | 鉴权扩展 | 新增系统管理员 / 普通管理员 / 用户三类鉴权逻辑 | API 和页面都要统一使用 |
| P1-4 | 驳回原因可见性 | 落地创建者 / 普通管理员 / 系统管理员三类可见范围 | 防止驳回原因全员泄露 |

### 5.3 P2 FlowDraft 与探索前台接入

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P2-1 | `dm_flow_drafts` | 建立探索态 DAG 草案对象表 | 每个 Task 对应当前 FlowDraft 或其版本 |
| P2-2 | 聊天转 FlowDraft | 让聊天澄清结果能不断更新 FlowDraft | 聊天产物不再只是文本，而是结构化草案 |
| P2-3 | DAG UI 适配 | 复用 Xagent 现有 DAG 可视化前端承载 FlowDraft | 避免重做图形前端 |
| P2-4 | 试跑前修正流 | 支持聊天修 DAG 和界面修 DAG 并存 | 这是 xagent 作为前台的核心价值 |

### 5.4 P3 HTTP 资产中心

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P3-1 | `dm_http_assets` | 建立 HTTP 资产模型 | 带 `system_short`、凭证密文、请求模板等字段 |
| P3-2 | HTTP 资产服务 | 支撑请求渲染、响应提取、资产测试执行 | 执行逻辑服务化，供聊天/模板/测试共用 |
| P3-3 | HTTP 资产 API | 提供列表、详情、创建、更新、删除、测试 | 不走审核主线 |
| P3-4 | HTTP 资产复制 | 支持复制别人 HTTP 资产为自己的新资产 | 默认继承原 `systemShort`，允许修改 |

### 5.5 P4 SQL 资产中心与审核后生效

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P4-1 | `dm_sql_assets` | 建立 SQL 资产逻辑对象表 | 负责名称、归属、当前生效版本指针 |
| P4-2 | `dm_sql_asset_versions` | 建立 SQL 资产版本表 | 负责连接信息、白名单黑名单、mutation 规则、审核状态 |
| P4-3 | 草稿与测试流 | 创建者可在草稿/待审核阶段测试连接与 schema | 但不能正式用于聊天执行或模板执行 |
| P4-4 | SQL 审核流 | 对应 `systemShort` 的普通管理员/系统管理员审核通过后生效 | 驳回后可继续改再提审 |
| P4-5 | 生效版本引用规则 | 聊天和模板正式执行只用当前生效版本 | 待审核版本不进入正式执行链 |
| P4-6 | SQL 复制规则 | 复制 SQL 资产视为新对象并重新审核 | 默认继承原 `systemShort`，允许修改，审核按新归属走 |

### 5.6 P5 模板中心与版本审核

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P5-1 | `dm_templates` 与 `dm_template_versions` | 建立逻辑模板与版本模型 | 模板带主 `system_short` |
| P5-2 | `dm_template_version_steps` | 建立受控 DAG 步骤模型 | 支持多前置依赖汇总，不支持条件分支和循环 |
| P5-3 | 模板草稿生成 | 从成功聊天 Run / FlowDraft 自动生成模板草稿 | 吸收前面已澄清的沉淀机制 |
| P5-4 | 模板审核流 | 模板按主 `systemShort` 提交审核并发布 | 普通管理员按系统范围审核 |
| P5-5 | 模板复制规则 | 普通用户可复制别人已发布模板为自己的个人草稿 | 不挂到原模板版本线 |
| P5-6 | 模板新版本规则 | 原模板新版本只能由原创建者或系统管理员发起 | 保护原模板版本线 |

### 5.7 P6 Run 中心与权限可见范围

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P6-1 | `dm_runs` | 建立正式执行单模型 | 模板页执行和聊天正式试跑统一落到 Run |
| P6-2 | `dm_run_steps` | 建立步骤快照与状态模型 | 便于运行详情页复盘 |
| P6-3 | Run 权限模型 | 系统管理员全量、普通管理员看负责系统、用户只看自己 | 必须和前端页面统一 |
| P6-4 | Run API | 提供详情、步骤、结果、确认危险 SQL 等接口 | 鉴权逻辑按角色和 `systemShort` 落地 |

### 5.8 P7 前端管理台与关系视图

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P7-1 | DataMakePool 管理台入口 | 在 Xagent 前端中新增统一造数平台入口 | 保持 `task` 页作为聊天探索入口 |
| P7-2 | SQL 审核状态展示 | 明确展示草稿、待审核、生效、驳回状态 | 待审核对象可见但不可正式使用 |
| P7-3 | 模板状态展示 | 草稿、待审核、已发布、历史版本清晰分层 | 待审核模板可见但不可执行 |
| P7-4 | 模板详情变更提示 | 展示最近变更的引用资产，并可跳转资产详情 | 仅告知，不拦截执行 |
| P7-5 | 资产详情关系视图 | 展示引用该资产的模板列表、版本、最近执行时间 | 帮助维护者评估影响范围 |
| P7-6 | Run 详情页适配 | 让正式 Run 详情能复用现有 task/debug 展示能力 | 同时保证用户只看到有权限的 Run |

### 5.9 P8 联调、验证、发布收口

| 编号 | 小功能 | 在做什么 | 设计与实现 |
|---|---|---|---|
| P8-1 | 单元测试 | 覆盖角色、审核流、资产版本、生效规则、Run 鉴权 | 重点验证权限边界 |
| P8-2 | 集成测试 | 跑通聊天探索 → FlowDraft → 试跑 → 模板草稿 → 审核发布 → 模板执行 | 这是平台主闭环 |
| P8-3 | 前端关键路径验证 | 聊天探索、资产管理、模板执行、Run 详情、审计页 | 确保双入口体验闭环 |
| P8-4 | 发布与回滚说明 | 明确迁移顺序、配置项、回滚方案 | 避免平台语义与旧 xagent 能力冲突 |

## 7. 风险与限制

| 风险点 | 说明 | 应对方式 |
|---|---|---|
| 现有 xagent `Task` 语义过重 | 现在的 Task 同时承担太多执行语义，若不拆清楚会和平台 Run 混在一起 | 明确 Task=ConversationSession，Run=正式执行单 |
| 现有 `text2sql` 模型过轻 | 当前只是用户数据库配置，不足以承载高风险 SQL 资产审核后生效 | 引入 `SqlAsset + SqlAssetVersion`，不要硬套旧模型 |
| 现有模板 API 与平台模板冲突 | 当前模板更像内置 agent 模板库 | 新增 `datamakepool_*` 路由与模型，不直接混用 |
| 权限模型残留旧语义 | 旧 xagent 只有 `is_admin` | 用 `admin_system_scopes` 引入普通管理员，并统一文档与代码口径 |
| 待审核对象可见但不可用的状态表达不清 | 用户容易误以为能直接执行 | 列表、详情、按钮、执行入口都按状态显式区分 |
| SQL 审核后生效实现不完整 | 若版本指针设计不清，会导致待审核版本误进正式执行链 | 强制引入逻辑对象 + 版本对象 |

## 8. 发布与回滚建议

### 发布顺序

1. 先发数据库迁移
2. 再发角色、`systemShort`、审核基础能力
3. 再发 `FlowDraft` 与资产中心
4. 再发模板中心、Run 中心与前端管理台
5. 最后开放模板页正式执行入口

### 回滚原则

- 若前台适配出问题，可先回退 DataMakePool 管理台，不影响现有 xagent 基础能力
- 若模板执行链出问题，可暂时关闭模板页正式执行入口，只保留聊天探索
- 若 SQL 资产版本审核链出问题，不允许继续开放 SQL 正式执行能力
- 若审计链出问题，不允许继续开放危险 SQL 执行能力

## 9. 一句话总结

基于 Xagent 落地统一造数平台的关键，不是继续在现有 task/chat/template 上打补丁，
而是把前面已经澄清好的平台语义，系统性地沉淀成：

- `ConversationSession(Task)`
- `FlowDraft`
- `Asset`
- `Template / TemplateVersion`
- `Run`
- `Audit`

再围绕 `systemShort`、普通管理员审核、SQL 资产审核后生效，
把整个平台真正做成型。
