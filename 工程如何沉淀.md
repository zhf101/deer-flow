# 新项目工程沉淀指引

本指引用于指导你在刚拿到一个项目后，快速生成一组高质量工程文档，让 AI 后续开发时不再只依赖临时上下文，而是能按项目事实、历史经验和验证规则工作。

目标不是写一堆 Prompt，而是给项目建立一个：

```text
项目操作系统
```

它要让 AI 进入项目时能回答：

```text
项目是什么
怎么运行
代码在哪里
模块边界是什么
业务规则是什么
API 怎么约定
踩过什么坑
改代码前怎么看影响
改代码后怎么验证
```

适用场景：

- 刚接手一个新项目。
- 想长期用 Codex / Claude Code / Cursor Agent 辅助开发。
- 项目会维护数月或数年。
- 希望 AI 能做影响分析、测试建议、代码审查和持续经验积累。
- 希望减少“每次都重新解释项目背景”的成本。

---

## 一、最终要生成哪些文档

建议在项目根目录生成以下 9 个文件：

| 文件 | 作用 | AI 什么时候读 |
|---|---|---|
| `AGENTS.md` | 项目级工作规则和强约束 | 每次任务开始 |
| `CODE_MAP.md` | 模块地图、入口、核心文件、测试位置 | 定位代码前 |
| `ARCHITECTURE.md` | 架构边界、依赖方向、核心数据流 | 改架构或跨模块逻辑前 |
| `PRODUCT.md` | 业务对象、用户流程、产品约束 | 涉及业务逻辑前 |
| `API.md` | API 前缀、方法规则、字段契约、兼容策略 | 改接口前 |
| `RETRO.md` | 历史踩坑、错误模式、已验证失败方案 | 每次任务开始 |
| `DECISION_DRIFT.md` | 关键架构决策和防漂移规则 | 重构或改边界前 |
| `TEST_STRATEGY.md` | 变更到测试范围的映射 | 改完代码前 |
| `IMPACT_MAP.md` | 调用关系和影响范围地图 | 做影响分析和回归建议前 |

这几份文档的分工要清晰：

- `AGENTS.md` 写规则，不写长篇架构分析。
- `CODE_MAP.md` 写“去哪找代码”，不解释所有实现细节。
- `ARCHITECTURE.md` 写系统边界和依赖方向，不复述文件列表。
- `PRODUCT.md` 写业务含义，不写技术实现。
- `API.md` 写接口契约，不写业务长流程。
- `RETRO.md` 写错误模式，不写流水账。
- `DECISION_DRIFT.md` 写为什么这样设计，不写当前代码清单。
- `TEST_STRATEGY.md` 写怎么验证，不写测试实现细节。
- `IMPACT_MAP.md` 写改哪里会影响哪里，不写完整架构教程。

## 二、先盘点项目，不要急着写文档

让 AI 先读项目，而不是凭目录名猜。

### 1. 基础盘点

让 AI 执行或等价完成：

```powershell
git status --short
rg --files
Get-ChildItem -Force
```

需要识别：

- 是否已有未提交改动。
- 是否已有 `AGENTS.md`、`CLAUDE.md`、`README.md`、`docs/`。
- 是否有后端、前端、移动端、脚本、部署、测试目录。
- 是否有多个子项目，例如 `backend/`、`frontend/`、`admin/`、`packages/`。

### 2. 技术栈盘点

重点找这些文件：

| 生态 | 典型文件 |
|---|---|
| Python | `pyproject.toml`、`requirements.txt`、`uv.lock`、`pytest.ini` |
| Node | `package.json`、`pnpm-lock.yaml`、`vite.config.ts`、`next.config.js` |
| Java | `pom.xml`、`build.gradle` |
| Go | `go.mod` |
| Rust | `Cargo.toml` |
| Docker | `Dockerfile`、`docker-compose.yaml` |
| CI | `.github/workflows/*`、`.gitlab-ci.yml` |

需要记录：

- 启动命令。
- 构建命令。
- 测试命令。
- lint / format 命令。
- 包管理器。
- 运行时依赖，例如数据库、消息队列、缓存、第三方 API。

### 3. 入口盘点

搜索入口时不要只看 README。让 AI 搜索：

```powershell
rg -n "FastAPI|APIRouter|express|router|Controller|@RestController|createApp|Next|Vite|main\\(|if __name__" .
rg -n "pytest|vitest|jest|playwright|unittest|describe\\(|it\\(" .
```

需要识别：

- 后端应用入口。
- 路由注册入口。
- 前端路由入口。
- 任务调度入口。
- CLI 入口。
- 测试入口。
- 配置加载入口。

### 4. 业务域盘点

让 AI 从目录名、路由、模型、数据库表、页面名称中归纳业务域：

```text
用户 / 权限 / 订单 / 支付 / 库存 / 报表 / 工作流 / Agent / 数据源 / 场景
```

要求 AI 输出时区分：

- 代码层模块。
- 产品层业务对象。
- API 层资源。
- 数据层实体。

这一步的目的，是避免后续文档把技术目录误当成业务概念。

## 三、推荐生成顺序

不要一口气让 AI 自由生成所有文档。按下面顺序更稳。

### 第 1 步：生成 `CODE_MAP.md`

先解决“代码在哪里”的问题。

高质量标准：

- 文件路径必须真实存在。
- 只列核心入口和高频修改位置，不展开所有文件。
- 每个模块写清职责。
- 测试目录和生产代码能对应上。
- 给出推荐搜索命令。

建议结构：

```markdown
# CODE_MAP.md

## 项目主结构
## 后端入口
## 前端入口
## 核心业务模块
## 数据层模块
## 测试地图
## 推荐搜索命令
```

给 AI 的提示词：

```text
请先阅读项目目录、README、package/pyproject/构建配置、主要路由和测试目录。
然后生成 CODE_MAP.md。
要求：
1. 只写真实存在的路径。
2. 每个路径说明职责。
3. 区分入口、业务模块、公共模块、测试模块。
4. 不要写长篇架构解释。
5. 给出后续定位代码常用的 rg 搜索命令。
```

### 第 2 步：生成 `ARCHITECTURE.md`

再解决“系统怎么组织”的问题。

高质量标准：

- 说清分层边界。
- 说清依赖方向。
- 说清核心数据流。
- 说清哪些模块不能互相依赖。
- 说清新增能力应该放在哪一层。

建议结构：

```markdown
# ARCHITECTURE.md

## 一句话架构
## 分层边界
## 核心运行链路
## 数据和状态
## 依赖方向
## 架构变更原则
```

给 AI 的提示词：

```text
基于 CODE_MAP.md 和源码入口，生成 ARCHITECTURE.md。
重点不是介绍每个文件，而是说明：
1. 系统分几层。
2. 每层负责什么。
3. 请求/任务/数据如何流转。
4. 哪些依赖方向禁止反转。
5. 修改某类能力应该放在哪个模块。
如果无法确认，不要编造，写“需要进一步确认”并列出确认方式。
```

### 第 3 步：生成 `PRODUCT.md`

如果项目有业务逻辑，就必须写产品文档。纯工具库可以简化。

高质量标准：

- 业务对象和技术对象分开。
- 用户流程写成可执行步骤。
- 业务规则明确到“什么输入导致什么行为”。
- 写出开发期策略，例如是否需要兼容旧数据。

建议结构：

```markdown
# PRODUCT.md

## 产品定位
## 核心用户流程
## 业务对象
## 业务规则
## 权限和角色
## 开发期产品策略
```

给 AI 的提示词：

```text
请从页面、路由、模型、数据库表、测试用例和已有文档中归纳 PRODUCT.md。
要求：
1. 不要把目录结构当成业务规则。
2. 每个业务对象写清含义。
3. 用户流程按步骤写。
4. 业务约束必须可验证。
5. 不确定的业务规则列为“待确认”，不要猜。
```

### 第 4 步：生成 `API.md`

涉及前后端或外部调用的项目，必须生成。

高质量标准：

- 写清 API 前缀。
- 写清 HTTP 方法规则。
- 写清认证、权限、错误格式。
- 写清字段契约在哪里维护。
- 写清接口变更检查清单。

建议结构：

```markdown
# API.md

## 全局入口
## 认证和权限
## API 方法规则
## 主要路由组
## 请求响应模型规则
## 错误响应规则
## API 变更检查清单
```

给 AI 的提示词：

```text
请搜索所有路由定义和 API 客户端调用，生成 API.md。
要求：
1. 按路由组汇总，不要复制完整 OpenAPI。
2. 写清方法规则和参数位置。
3. 写清请求/响应模型所在位置。
4. 写清前端调用文件。
5. 列出修改 API 时必须同步的文件和测试。
```

### 第 5 步：生成 `IMPACT_MAP.md`

这个文件直接服务于 AI 后续开发。

高质量标准：

- 写出上游入口、当前模块、下游依赖。
- 能从“修改点”推导出“回归范围”。
- 不追求完整调用图，先覆盖高频改动区域。
- 可以用文本树，不需要复杂图工具。

建议结构：

```markdown
# IMPACT_MAP.md

## 总体依赖图
## 核心业务影响图
## 前端影响图
## 后端影响图
## 修改点到影响范围
## 影响分析输出模板
```

给 AI 的提示词：

```text
请基于 CODE_MAP.md、ARCHITECTURE.md、路由、服务、仓储、前端 API 调用生成 IMPACT_MAP.md。
要求：
1. 用文本树表达主要依赖链。
2. 对常见修改点列出上游入口、下游消费者和回归重点。
3. 输出一个影响分析模板，供后续每次任务复用。
4. 不要假装有完整静态调用分析，无法确认的依赖要标注。
```

### 第 6 步：生成 `TEST_STRATEGY.md`

这个文件让 AI 改完代码后知道该跑什么。

高质量标准：

- 命令必须来自真实脚本或配置。
- 按变更类型映射测试范围。
- 区分后端、前端、E2E、lint、构建。
- 包含功能、边界、异常、回归测试建议模板。

建议结构：

```markdown
# TEST_STRATEGY.md

## 通用验证原则
## 常用验证命令
## 变更到测试的映射
## 测试建议模板
## 高风险专项检查
```

给 AI 的提示词：

```text
请读取 package.json、pyproject.toml、CI 配置、测试目录，生成 TEST_STRATEGY.md。
要求：
1. 验证命令必须真实存在。
2. 如果命令需要进入子目录，明确写 cd 到哪里。
3. 按变更类型映射到测试文件或测试命令。
4. 加入功能、边界、异常、回归四类测试建议模板。
5. 文档变更、配置变更、API 变更、数据库变更分别写验证策略。
```

### 第 7 步：生成 `RETRO.md`

新项目刚拿到时，可能没有真实复盘。可以先写“从现有约束推导出的风险模式”，后续逐步补充。

高质量标准：

- 记录错误模式，不记录单次事故流水账。
- 每条都有现象、根因、解决方案、防护措施。
- 已升级到规则的条目标记清楚。
- 未解决问题单独列出。

建议结构：

```markdown
# RETRO.md

## 使用规则
## API 类
## 架构类
## 数据库类
## 前端类
## 测试类
## 未解决问题
```

条目模板：

```markdown
### RETRO-001 错误模式标题 [状态]

- 现象：
- 根因：
- 解决方案：
- 防护措施：
```

给 AI 的提示词：

```text
请基于已有 issue、测试命名、注释、待办标记、用户规则和最近修改，生成 RETRO.md 初版。
要求：
1. 只记录可复用的错误模式。
2. 不要编造生产事故。
3. 每条包含现象、根因、解决方案、防护措施。
4. 已经进入 AGENTS.md 的规则标记为“已升级”。
5. 没有证据的风险写成“待观察”，不要写成事实。
```

### 第 8 步：生成 `DECISION_DRIFT.md`

这个文件用于防止 AI 后续“看起来合理”地破坏原架构。

高质量标准：

- 每个决策写清“为什么”。
- 写清漂移信号。
- 写清验证方式。
- 不写成会议纪要。

建议结构：

```markdown
# DECISION_DRIFT.md

## ADR-001 决策标题

- 决策：
- 原因：
- 约束：
- 漂移信号：
- 验证：
```

给 AI 的提示词：

```text
请从 ARCHITECTURE.md、历史文档、目录结构、测试边界和用户规则中提炼 DECISION_DRIFT.md。
要求：
1. 每个 ADR 写清决策、原因、约束、漂移信号、验证方式。
2. 只写会影响后续开发判断的决策。
3. 不要写泛泛的工程原则。
4. 如果某个决策来源不明确，标注“从当前代码结构推断”。
```

### 第 9 步：最后生成 `AGENTS.md`

`AGENTS.md` 不应该一开始就写。它应该吸收前面文档中已经确认的规则。

高质量标准：

- 短、硬、可执行。
- 写阅读顺序。
- 写强制规则。
- 写修改前影响分析。
- 写测试要求。
- 写 RETRO 升级规则。
- 写交付输出格式。
- 不塞长篇业务细节。

建议结构：

```markdown
# AGENTS.md

## 阅读顺序
## 通用修改原则
## 项目强制规则
## 修改前影响分析
## 测试要求
## 发现新经验
## 交付输出
```

给 AI 的提示词：

```text
请基于 CODE_MAP.md、ARCHITECTURE.md、PRODUCT.md、API.md、RETRO.md、DECISION_DRIFT.md、TEST_STRATEGY.md、IMPACT_MAP.md 生成 AGENTS.md。
要求：
1. AGENTS.md 是规则入口，不是长文档。
2. 保留项目最重要的强制约束。
3. 写清每次任务前的阅读顺序。
4. 写清修改前影响分析和修改后交付格式。
5. 不要重复大段架构、API、产品内容，只链接到对应文件。
```

## 四、一次性生成所有文档的总提示词

如果你希望让 AI 一次完成初版，可以直接使用下面这段。

```text
我刚拿到这个项目，请帮我建立一套用于长期 AI 辅助开发的工程沉淀文档。

请先盘点项目，不要急着写文档：
1. 检查 git 状态，避免覆盖未提交改动。
2. 阅读 README、现有 docs、配置文件、路由入口、测试目录和主要业务目录。
3. 识别技术栈、启动命令、测试命令、模块边界、业务对象、API 规则和高风险区域。

然后在项目根目录生成或更新：
- AGENTS.md
- CODE_MAP.md
- ARCHITECTURE.md
- PRODUCT.md
- API.md
- RETRO.md
- DECISION_DRIFT.md
- TEST_STRATEGY.md
- IMPACT_MAP.md

写作要求：
1. 所有路径和命令必须来自真实项目，不能编造。
2. 根目录文档要短而可执行，长篇分析放到 docs/ 并在根文档中链接。
3. 每份文档职责要清晰，不要互相复制。
4. 对不确定的内容标注“待确认”，并写明如何确认。
5. AGENTS.md 只放强规则、阅读顺序、影响分析、测试要求和交付格式。
6. RETRO.md 记录错误模式，不记录流水账。
7. TEST_STRATEGY.md 必须把变更类型映射到测试命令或测试文件。
8. IMPACT_MAP.md 必须能辅助 AI 输出影响范围和回归测试建议。

完成后请自检：
1. 所有新增文档是否存在。
2. 文档中引用的路径是否存在。
3. 文档中写的命令是否在项目脚本或配置中存在。
4. 是否误把猜测写成事实。
5. 是否覆盖“修改内容、影响范围、已执行验证、测试建议、是否新增 RETRO、是否需要升级规则”的交付格式。
```

## 五、质量自检清单

生成后，让 AI 自检以下问题。

### 事实准确性

- 文档中的路径是否真实存在。
- 文档中的命令是否真实存在。
- API 前缀和方法是否来自源码。
- 测试文件名是否真实存在。
- 技术栈是否来自配置文件，而不是猜测。

### 可执行性

- AI 看 `CODE_MAP.md` 能不能快速定位代码。
- AI 看 `ARCHITECTURE.md` 能不能知道模块边界。
- AI 看 `API.md` 能不能知道改接口要同步哪里。
- AI 看 `TEST_STRATEGY.md` 能不能知道该跑哪些验证。
- AI 看 `IMPACT_MAP.md` 能不能输出影响范围。
- AI 看 `RETRO.md` 能不能避免重复踩坑。

### 边界清晰度

- 每份文档是否只承担自己的职责。
- 是否有大段重复内容。
- 是否把临时分析写成强规则。
- 是否把业务规则写进了架构文档。
- 是否把架构原因写进了代码地图。

### AI 友好度

- 是否有明确阅读顺序。
- 是否有修改前检查清单。
- 是否有修改后交付格式。
- 是否有测试建议模板。
- 是否有规则升级机制。

## 六、如何用这些文档辅助下一步开发

之后每次让 AI 开发前，可以这样要求：

```text
开始前请先阅读 AGENTS.md、RETRO.md，并根据任务类型阅读 CODE_MAP.md / ARCHITECTURE.md / PRODUCT.md / API.md / TEST_STRATEGY.md / IMPACT_MAP.md。
修改前先输出影响范围。
修改后输出修改内容、影响范围、已执行验证、测试建议、是否新增 RETRO、是否需要升级规则。
```

如果是接口变更：

```text
请先阅读 API.md、CODE_MAP.md、IMPACT_MAP.md、TEST_STRATEGY.md。
```

## 七、维护规则

这套文档不是一次性产物，需要随项目维护。

### 什么时候更新文档

- 新增核心模块：更新 `CODE_MAP.md`、`ARCHITECTURE.md`、`IMPACT_MAP.md`。
- 新增业务对象：更新 `PRODUCT.md`。
- 新增或修改 API 风格：更新 `API.md`、`TEST_STRATEGY.md`。
- 新增测试类型或命令：更新 `TEST_STRATEGY.md`。
- 发现重复问题：更新 `RETRO.md`。
- 架构决策变化：更新 `DECISION_DRIFT.md`。
- 某条经验升级为强规则：更新 `AGENTS.md`。

### RETRO 升级规则

如果同类问题累计出现 3 次及以上：

1. 在 `RETRO.md` 保留错误模式和案例。
2. 在 `AGENTS.md` 增加强制规则。
3. 在 `TEST_STRATEGY.md` 增加验证要求。
4. 在 `IMPACT_MAP.md` 增加影响链路。

这样经验会从“记忆”变成“制度”，后续 AI 不需要重新踩坑。

## 八、常见反模式

### 反模式 1：AGENTS.md 写成大杂烩

问题：

- AI 每次都要读大量细节。
- 规则、架构、业务、API 混在一起。
- 后续维护成本高。

正确做法：

- `AGENTS.md` 只写入口规则。
- 细节分流到其它文档。

### 反模式 2：文档里写不存在的路径和命令

问题：

- AI 后续会按错误路径搜索。
- 验证阶段浪费时间。

正确做法：

- 写文档前用 `Test-Path`、`rg --files`、`package.json`、`pyproject.toml` 核对。

### 反模式 3：RETRO 写事故流水账

问题：

- “某接口报错”无法指导下次开发。

正确做法：

- 写“DTO 变更遗漏 Mapper 同步”这类错误模式。

### 反模式 4：TEST_STRATEGY 只写测试命令

问题：

- AI 不知道改什么该跑什么。

正确做法：

- 建立“变更类型 -> 测试范围”的映射。

### 反模式 5：IMPACT_MAP 追求完整调用图

问题：

- 初期维护成本过高，很快过期。

正确做法：

- 先覆盖高频变更链路和关键模块边界。

## 九、最小落地版本

如果时间有限，至少先生成 4 个文件：

1. `AGENTS.md`
2. `CODE_MAP.md`
3. `RETRO.md`
4. `TEST_STRATEGY.md`

这 4 个文件能先解决：

- AI 该遵守什么规则。
- AI 去哪里找代码。
- AI 避免哪些坑。
- AI 改完怎么验证。

等项目进入持续开发，再补齐：

- `ARCHITECTURE.md`
- `PRODUCT.md`
- `API.md`
- `DECISION_DRIFT.md`
- `IMPACT_MAP.md`

## 十、判断文档是否真的有用

可以用下面 6 个问题测试：

1. 一个新 AI 只读这些文档，能不能找到要改的代码？
2. 它能不能说清修改会影响哪些模块？
3. 它能不能知道哪些架构边界不能破坏？
4. 它能不能知道改完该跑哪些测试？
5. 它能不能避免项目已经踩过的坑？
6. 它能不能按固定格式交付修改内容、影响范围、验证和测试建议？

如果答案是否定的，说明文档还停留在“说明书”，没有成为“项目操作系统”。
